# Claude Code가 대규모 코드베이스에서 동작하는 방식 — 베스트 프랙티스와 출발점

> 원문: [How Claude Code Works in Large Codebases: Best Practices and Where to Start](https://claude.com/blog/how-claude-code-works-in-large-codebases-best-practices-and-where-to-start)
> 게시일: 2026-05-14
> 카테고리: Enterprise AI (시리즈: "Claude Code at Scale")

---

## 글의 맥락

이 글은 엔지니어링 조직이 **엔터프라이즈 스케일에서 Claude Code를 도입**할 때 참고할 만한 베스트 프랙티스를 다루는 "Claude Code at Scale" 시리즈의 일부다. 글이 가정하고 있는 환경은 꽤 가혹하다:

- 수백만 라인 규모의 모노레포
- 수십 년에 걸쳐 쌓인 레거시 시스템
- 수십 개의 레포로 분산된 아키텍처
- 수천 명 규모의 개발자가 함께 일하는 조직

Anthropic은 이런 환경에서 Claude Code가 실제로 잘 동작하는 것을 보아왔고, 그 경험에서 얻은 패턴을 이 글에 정리했다.

## 1. Claude Code는 대규모 코드베이스를 어떻게 탐색하는가

Claude Code는 **사람 엔지니어가 코드베이스를 읽는 방식**을 그대로 흉내 낸다:

- 파일 시스템을 따라 이동한다
- 파일을 직접 읽는다
- `grep`으로 필요한 부분을 정확히 찾는다
- 코드베이스 전반의 참조(reference)를 따라간다

핵심은 이 모든 작업이 **개발자 로컬 머신에서 직접 일어난다**는 점이다. 별도의 중앙 집중식 코드베이스 인덱싱이 필요하지 않다.

### Agentic Search vs. RAG

많은 AI 코딩 도구가 RAG(Retrieval-Augmented Generation)에 의존한다. 코드베이스 전체를 임베딩해 두고, 질의 시점에 관련된 청크를 가져오는 방식이다. 문제는:

- 활발하게 작업하는 팀의 변경 속도를 임베딩 파이프라인이 따라가지 못한다
- 결과적으로 이미 이름이 바뀐 함수, 삭제된 모듈에 대한 **오래된 참조**가 반환된다

Claude Code는 RAG 대신 **agentic search** 방식을 쓴다. 매번 실제 파일 시스템을 살펴보기 때문에 인덱스가 stale 해질 일이 없다.

**대신 트레이드오프가 있다.** Agentic search는 "어디를 봐야 하는지" 알 수 있는 **충분한 시작 컨텍스트**가 주어졌을 때 가장 잘 작동한다. 따라서 코드베이스를 잘 탐색하려면 CLAUDE.md와 skill을 통한 적절한 셋업이 필요하다.

## 2. 모델만큼이나 중요한 것은 Harness다

Anthropic은 Claude Code의 성공이 모델 자체만큼이나 그것을 둘러싼 **harness(주변 도구 구성)** 의 품질에 달려 있다고 본다. 다음 7가지 확장 포인트를 잘 활용해야 한다.

### 2.1 CLAUDE.md 파일 — 가장 먼저 로드되는 컨텍스트

- 세션 시작 시 Claude가 자동으로 읽는 컨텍스트 파일
- **계층 구조**로 두는 것이 핵심: 루트에는 큰 그림, 하위 디렉터리에는 그 영역의 로컬 컨벤션
- 핵심 원칙: **가볍게(lean), 광범위하게 적용되는 정보만 담는다**
- 흔한 실수: "재사용 가능한 전문성(reusable expertise)"을 여기에 다 쑤셔 넣는 것 — 그건 skill에 들어가야 한다

### 2.2 Hooks — 자기 개선(self-improving) 셋업

- 세션 중 특정 순간에 트리거되는 스크립트
- 가장 큰 가치는 **지속적 개선과 자기 반성(self-reflection)**
- 활용 예:
  - **Stop hook**: 컨텍스트가 아직 신선할 때 CLAUDE.md 업데이트를 제안하게 한다
  - **Start hook**: 개발자별로 팀 컨텍스트를 동적으로 로드한다
  - 린트/포맷팅을 **결정적(deterministic)** 으로 강제한다
- 흔한 실수: 자동화로 풀어야 할 일을 매번 프롬프트로 시키는 것

### 2.3 Skills — 필요할 때만 호출되는 전문성

- 특정 작업 유형에 맞춰 포장된 instruction 묶음
- **관련 있을 때만 로드**되므로 컨텍스트 부담이 적다
- Progressive disclosure를 활용해 특수 워크플로를 따로 빼낼 수 있다
- 특정 경로에만 스코프 지정 가능 (예: 결제 서비스 배포 전용 skill)
- 모든 전문성이 컨텍스트 공간을 두고 경쟁하는 상황을 막아준다
- 흔한 실수: 모든 걸 CLAUDE.md에 몰아넣고 skill로 분리하지 않는 것

### 2.4 Plugins — 동작하는 셋업을 배포하기

- skill, hook, MCP 설정을 묶어 **설치 가능한 패키지**로 만든다
- 신규 입사자가 처음부터 검증된 컨텍스트와 능력을 그대로 받을 수 있다
- 조직 내부에서 관리형 마켓플레이스로 배포한다
- 대규모 조직의 고질적인 **"부족 지식(tribal knowledge)"** 문제를 해결한다
- 흔한 실수: 좋은 셋업이 한 팀/한 사람에게만 머무는 것

### 2.5 LSP (Language Server Protocol) 통합 — 심볼 단위 정밀도

- 텍스트 매칭에서 얻을 수 없는 **심볼 수준의 정확성**을 제공한다
- "go to definition", "find all references"가 가능해진다
- 서로 다른 언어에 같은 이름의 함수가 있어도 구분된다
- 텍스트 기반 검색이 일으키는 패턴 매칭 오류를 막아준다
- 사례: 한 엔터프라이즈 고객은 C/C++ 작업의 신뢰성을 위해 Claude Code 전사 도입 **전에** LSP를 먼저 깔았다
- 흔한 실수: LSP가 자동으로 켜져 있다고 가정하는 것

### 2.6 MCP Servers — 능력 확장

- 내부 도구, 데이터 소스, API에 Claude를 연결한다
- 더 성숙한 팀은 **구조화된 검색(structured search)** 을 호출 가능한 도구로 노출하는 MCP 서버를 직접 만든다
- 내부 문서, 티켓 시스템, 분석 플랫폼 등과의 연결
- 한 번 설정해두면 **상시 사용 가능**

### 2.7 Subagents — 격리된 태스크 실행

- **독립된 컨텍스트 윈도우**를 가진 별도의 Claude 인스턴스
- 작업을 위임받아 처리하고 최종 결과만 부모에게 돌려준다
- 활용 패턴: **탐색(exploration)과 편집(editing)을 분리**한다
- 성숙한 팀은 read-only subagent를 띄워 서브시스템을 매핑한다
- 흔한 실수: 탐색과 편집을 한 세션에서 같이 돌리는 것

## 3. 성공적인 배포에서 반복적으로 보이는 3가지 구성 패턴

### 3.1 코드베이스를 스케일에서도 탐색 가능하게 만들기

#### CLAUDE.md를 가볍게, 그리고 계층적으로 두라

- 루트 파일은 **핵심 포인터와 결정적인 함정(gotcha)** 만 담는다
- 하위 디렉터리 파일은 거기에 **추가적으로(additively)** 로드되어 로컬 컨벤션을 보강한다
- 안 그러면 파일이 노이즈로 가득 차 오히려 성능을 떨어뜨린다

#### 레포 루트가 아니라 서브디렉터리에서 init 하라

- 작업 영역에 맞게 Claude의 스코프를 좁혀라
- Claude는 **자동으로 디렉터리 트리를 거슬러 올라가며** CLAUDE.md를 모두 로드한다
- 그러므로 서브디렉터리에서 init 해도 루트 컨텍스트는 잃지 않는다

#### 테스트/린트 명령어를 서브디렉터리별로 정의하라

- 각 영역에 맞는 명령어를 명시하라
- 단일 서비스 변경에 대해 전체 테스트 스위트가 도는 일을 막아준다 (타임아웃 방지)
- 무관한 출력이 컨텍스트를 잡아먹는 것도 줄여준다
- 빌드 명령어가 서비스마다 다른 **서비스 지향 코드베이스에 필수**

#### `.claudeignore`를 활용하라

- 생성 파일, 빌드 산출물, 서드파티 코드를 제외한다
- `permissions.deny` 룰은 **버전 관리되도록 `.claude/settings.json`에 커밋**한다
- 팀 전체에 일관된 노이즈 감소가 적용된다
- 코드 제너레이터를 만드는 개발자처럼 예외가 필요한 경우, 로컬 오버라이드로 처리하면 팀에는 영향을 주지 않는다

#### 비관습적 구조에는 "코드베이스 맵"을 만들어라

- 레포 루트에 가벼운 마크다운 파일을 두고 최상위 폴더 목록을 적는다
- 각 디렉터리 내용에 한 줄 설명을 단다
- Claude가 파일을 열기 전에 **목차(table of contents)** 부터 훑어볼 수 있다
- 최상위 폴더가 수백 개라면 루트 + 서브디렉터리 디테일의 **계층 구조**로 가라
- 대안: Claude가 봐야 할 파일이나 디렉터리를 **@-mention**으로 직접 지정하기

#### LSP 서버로 심볼 기반 검색을 돌려라

- 대규모 코드베이스에서 grep은 수천 개의 매칭을 토해낸다
- LSP는 **같은 심볼에 대한 참조만** 돌려준다
- 즉, Claude가 무언가를 읽기도 전에 **필터링이 끝난 상태**다
- 필요한 것: 언어용 code intelligence 플러그인 + 언어 서버 바이너리

### 3.2 모델이 진화하면 CLAUDE.md를 능동적으로 손봐라

모델이 똑똑해지면, 과거에 모델 한계를 보정하려고 넣었던 지시들이 오히려 **제약**이 된다. 예:

- "리팩토링은 한 번에 한 파일씩 처리하라"는 룰 — 새 모델은 여러 파일을 조율하며 잘 처리할 수 있는데도 이를 막아버린다
- 특정 모델의 약점을 보정하던 skill/hook — 그 약점이 사라지면 그저 오버헤드만 남는다
- 실제 사례: Perforce edit을 가로채던 hook이, Claude Code가 네이티브 Perforce 모드를 추가하면서 무용지물이 되었다

**권장 주기**: **3~6개월마다** 의미 있는 설정 리뷰를 한다. 특히 **메이저 모델 릴리스 직후**, 또는 성능이 정체된다고 느낄 때 한 번 보는 것이 좋다.

### 3.3 Claude Code 운영과 도입을 담당할 오너십을 지정하라

기술적 설정만으로는 도입이 이루어지지 않는다.

#### 사전 인프라 투자

- 폭넓게 액세스를 풀기 **전에** 작은 팀(때로는 1인)이 도구를 정비해 둔다
- 개발자가 처음 Claude를 만났을 때 자기 워크플로에 잘 맞아야 한다
- 첫 경험이 **답답함이 아니라 생산성**이어야 한다
- 이 기반 위에서 도입이 자연스럽게 퍼져나간다

#### 조직 내 배치

- 보통 **Developer Experience / Developer Productivity** 조직 산하에 둔다
- 이 팀은 보통 온보딩과 개발자 툴링도 책임진다
- 새로 떠오르는 역할: **"Agent Manager"** — Claude Code 생태계 관리를 전담하는 PM/엔지니어 하이브리드

#### 최소한의 거버넌스 구조 (MVP)

지정된 **DRI(Directly Responsible Individual)** 1명. 이 사람이 다음을 책임진다:

- Claude Code 설정 전반
- 권한과 설정 정책 결정 권한
- 플러그인 마켓플레이스 관리
- CLAUDE.md 컨벤션
- 시간이 지나도 설정을 최신 상태로 유지

#### 파편화를 막아라

- 바텀업으로 시작된 도입은 열정은 만들지만 **파편화** 된다
- 누군가는 컨벤션을 모아 정리하고 전도해야 한다
- 조직 전반의 표준화된 CLAUDE.md 계층 구조
- 큐레이션된 skill과 plugin 세트
- 이게 없으면 지식은 **부족 지식**으로 남고 도입은 정체된다

#### 규제 산업에서의 거버넌스

- **일찍부터 cross-functional 워킹그룹**으로 풀어라
- 엔지니어링, 보안, 거버넌스가 한자리에서
- 요구사항을 함께 정의하고 롤아웃 로드맵을 같이 짠다
- 가장 매끄러운 배포는 이런 그룹을 초기에 세팅해둔다

#### 미리 답해두면 좋은 거버넌스 질문들

- 어떤 skill과 plugin을 사용 가능하게 할지 **누가 통제**하는가?
- 수천 명의 엔지니어가 **같은 능력을 독립적으로 재구축하는 일**을 어떻게 막을 것인가?
- AI가 만든 코드도 **사람이 만든 코드와 동일한 리뷰 프로세스**를 거치도록 어떻게 보장할 것인가?

#### 시작 전략

- 승인된 skill **한정된 세트로 시작**
- 모든 AI 생성 코드에 코드 리뷰 요구
- 초기 접근 권한 제한
- 신뢰가 쌓이면 단계적으로 확장

## 4. 어떤 환경에 잘 맞는가

Claude Code는 **관습적인 소프트웨어 엔지니어링 환경**을 가정하고 설계됐다:

- 코드베이스 기여자는 주로 엔지니어
- Git을 사용
- 표준적인 디렉터리 구조

대부분의 대규모 코드베이스가 여기 해당된다. 다만 다음과 같은 비전통적 환경은 **추가 설정 작업**이 필요하다:

- 대용량 바이너리 자산이 많은 게임 엔진
- 비관습적인 버전 관리 시스템
- 엔지니어가 아닌 사람도 코드베이스에 기여하는 환경

Anthropic의 **Applied AI 팀**이 이런 조직 특수 요구사항에 맞춰 패턴을 번역해주는 역할을 한다.

## 5. 핵심 요약 (의사결정에 바로 쓰는 체크리스트)

이 글에서 실제로 가져갈 행동 아이템:

- [ ] CLAUDE.md는 **루트 + 서브디렉터리 계층**으로 잘게 쪼개고, 각 파일을 lean하게 유지한다
- [ ] 재사용 가능한 전문성은 CLAUDE.md가 아니라 **skill**로 분리한다
- [ ] 서비스/서브디렉터리별로 **테스트·린트 명령어**를 명시한다
- [ ] `.claudeignore` + `permissions.deny`를 **버전 관리**한다
- [ ] 비관습적 구조라면 레포 루트에 **코드베이스 맵(목차)** 을 둔다
- [ ] 가능한 언어는 **LSP 플러그인**을 설치해 심볼 단위 검색을 켠다
- [ ] **탐색용 subagent와 편집용 세션을 분리**한다
- [ ] 좋은 셋업은 **plugin으로 묶어** 사내에 배포한다
- [ ] **Stop/Start hook**을 활용해 CLAUDE.md 자가 개선과 팀별 컨텍스트 주입을 자동화한다
- [ ] 메이저 모델 릴리스 직후를 포함해 **3~6개월마다** 설정 리뷰를 한다
- [ ] Claude Code 운영을 책임지는 **DRI 1명**을 지정하고, DevEx/DevProd 산하에 둔다
- [ ] 규제 산업이라면 엔지니어링/보안/거버넌스가 모인 **워킹그룹을 초기부터** 세팅한다
