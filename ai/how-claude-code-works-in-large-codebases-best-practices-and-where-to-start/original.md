# Claude Code가 대규모 코드베이스에서 작동하는 방식: 모범 사례 및 시작점

> 가장 성공적인 Claude Code 배포 사례는 구성, 도구 및 조직 구조 전반에 걸쳐 공통적인 패턴을 공유합니다. 이 글은 엔터프라이즈 규모에서 Claude Code를 사용하는 엔지니어링 조직을 위한 모범 사례를 다루는 새로운 시리즈인 "대규모 Claude Code"의 일부입니다.
> 

---

Claude Code는 수백만 줄에 달하는 모노레포, 수십 년 된 레거시 시스템, 수십 개의 리포지토리에 걸쳐 있는 분산 아키텍처, 그리고 수천 명의 개발자가 있는 조직 등 다양한 환경에서 성공적으로 운영되고 있습니다. 이러한 환경은 규모가 작고 단순한 코드베이스에서는 발생하지 않는 여러 가지 문제점을 안고 있습니다. 예를 들어, 하위 디렉터리마다 빌드 명령이 다르거나, 루트 디렉터리가 공유되지 않은 채 여러 폴더에 흩어져 있는 레거시 코드 등이 있습니다.

이 글에서는 Claude Code를 대규모로 성공적으로 도입하는 데 도움이 된 패턴들을 살펴봅니다. 여기서 "대규모 코드베이스"는 수백만 줄의 모노레포, 수십 년에 걸쳐 구축된 레거시 시스템, 여러 리포지토리에 분산된 수십 개의 마이크로서비스 등 다양한 배포 환경을 포괄합니다. 또한 C, C++, C#, Java, PHP와 같이 AI 코딩 도구와는 다소 거리가 먼 언어로 작성된 코드베이스도 포함됩니다. (특히 최근 모델 릴리스를 기준으로 볼 때, Claude Code는 대부분의 팀이 예상하는 것보다 훨씬 뛰어난 성능을 보여줍니다.) 모든 대규모 코드베이스 배포는 특정 버전 관리, 팀 구조 및 축적된 규칙에 따라 형태가 달라지지만, 여기에 제시된 패턴은 이러한 모든 경우에 일반화될 수 있으며 Claude Code 도입을 고려하는 팀에게 좋은 출발점이 될 수 있습니다.

## 클로드 코드(Claude Code)는 어떻게 대규모 코드베이스를 탐색할까요?

클로드 코드는 소프트웨어 엔지니어가 코드베이스를 탐색하는 방식과 동일하게 작동합니다. 파일 시스템을 탐색하고, 파일을 읽고, grep 명령어를 사용하여 필요한 부분을 정확하게 찾고, 코드베이스 전체에 걸쳐 참조를 따라갑니다. 개발자 컴퓨터에서 로컬로 실행되므로 코드베이스 인덱스를 구축, 유지 관리하거나 서버에 업로드할 필요가 없습니다.

RAG 기반 AI 코딩 도구는 전체 코드베이스를 임베드하고 쿼리 시점에 관련 코드 덩어리를 검색하는 방식으로 작동합니다. 하지만 대규모 환경에서는 임베드 파이프라인이 활발한 엔지니어링 팀의 작업 속도를 따라가지 못해 이러한 시스템이 제대로 작동하지 않을 수 있습니다. 개발자가 인덱스를 쿼리할 때쯤이면 인덱스는 몇 주, 며칠, 심지어 몇 시간 전의 코드베이스 상태를 반영하게 됩니다. 따라서 검색 결과는 팀이 2주 전에 이름을 바꾼 함수나 지난 스프린트에서 삭제된 모듈을 참조하는 경우가 발생할 수 있으며, 이러한 정보가 최신 상태가 아니라는 표시도 없습니다.

에이전틱 검색(Agentic search)은 이러한 문제점을 해결합니다. 수천 명의 엔지니어가 새 코드를 커밋할 때 유지 관리해야 할 임베딩 파이프라인이나 중앙 집중식 인덱스가 없습니다. 각 개발자 인스턴스는 실제 코드베이스에서 작동합니다.

하지만 이 접근 방식에는 단점이 있습니다. Claude는 어디를 찾아야 할지 알 수 있을 만큼 충분한 초기 컨텍스트를 가지고 있을 때 가장 잘 작동합니다. 즉, Claude의 탐색 품질은 코드베이스가 얼마나 잘 구성되어 있는지, 즉 CLAUDE.md 파일과 스킬을 통해 컨텍스트가 계층적으로 구축되어 있는지에 따라 결정됩니다. 수십억 줄에 달하는 코드베이스에서 모호한 패턴의 모든 인스턴스를 찾도록 요청하면 작업이 시작되기도 전에 컨텍스트 창의 한계에 부딪히게 됩니다. 코드베이스 설정에 투자하는 팀은 더 나은 결과를 얻습니다.

## 하네스는 모델만큼이나 중요합니다.

클로드 코드에 대한 가장 흔한 오해 중 하나는 그 기능이 오로지 사용되는 모델에 의해서만 결정된다는 것입니다. 팀들은 모델의 벤치마크와 테스트 작업에서의 성능에만 집중합니다. 하지만 실제로는 모델을 둘러싼 생태계, 즉 하네스가 클로드 코드의 성능을 결정하는 데 있어 모델 자체보다 훨씬 더 큰 영향을 미칩니다.

하네스는 CLAUDE.md 파일, 훅, 스킬, 플러그인, MCP 서버라는 다섯 가지 확장 지점으로 구성되며, 각 지점은 서로 다른 기능을 수행합니다. 각 계층은 이전 계층을 기반으로 구축되므로 팀에서 이러한 요소들을 구축하는 순서가 중요합니다. 여기에 LSP 통합과 서브에이전트라는 두 가지 추가 기능이 더해져 완벽한 구성을 이룹니다. 아래에서는 이러한 각 구성 요소와 기능의 역할을 설명합니다.

CLAUDE.md 파일이 가장 중요합니다. 이 파일은 Claude가 매 세션 시작 시 자동으로 읽는 컨텍스트 파일입니다. 루트 파일은 전체적인 맥락을, 하위 디렉터리 파일은 로컬 규칙을 담고 있습니다. 이 파일들은 Claude가 모든 작업을 효율적으로 수행하는 데 필요한 코드베이스 정보를 제공합니다. 작업 종류와 관계없이 모든 세션에서 로드되므로, 전반적으로 적용되는 내용에 집중하면 성능 저하를 방지할 수 있습니다.

훅(Hook)은 설정 과정을 자체적으로 개선합니다. 대부분의 팀은 훅을 Claude의 오류를 방지하는 스크립트로 생각하지만, 훅의 진정한 가치는 지속적인 개선에 있습니다. 정지 훅은 세션 중에 발생한 상황을 분석하여 컨텍스트가 최신 상태일 때 CLAUDE.md 업데이트를 제안할 수 있습니다. 시작 훅은 팀별 컨텍스트를 동적으로 로드하여 모든 개발자가 수동 설정 없이 자신의 모듈에 맞는 설정을 사용할 수 있도록 합니다. 린팅 및 포맷팅과 같은 자동화된 검사의 경우, 훅을 사용하면 규칙을 확정적으로 적용하여 Claude가 명령어를 기억하도록 하는 것보다 더 일관된 결과를 얻을 수 있습니다.

스킬을 사용하면 모든 세션에 불필요한 리소스를 추가하지 않고도 필요한 전문 지식을 필요에 따라 즉시 활용할 수 있습니다. 수십 가지 작업 유형이 있는 대규모 코드베이스에서는 모든 전문 지식이 모든 세션에 필요한 것은 아닙니다. 스킬은 점진적 공개 방식을 통해 이러한 문제를 해결합니다. 즉, 컨텍스트 공간을 두고 경쟁할 수 있는 전문 워크플로와 도메인 지식을 분리하여 작업에 필요할 때만 로드합니다. 예를 들어, 보안 검토 스킬은 Claude가 코드의 취약점을 검토할 때 로드되고, 문서 처리 스킬은 코드 변경 후 문서를 업데이트해야 할 때 로드됩니다.

또한 스킬은 특정 경로로 범위를 지정하여 코드베이스의 관련 부분에서만 활성화되도록 할 수 있습니다. 결제 서비스를 담당하는 팀은 배포 스킬을 해당 디렉터리에 바인딩하여 모노레포의 다른 위치에서 작업할 때는 자동으로 로드되지 않도록 할 수 있습니다.

플러그인은 효과적인 기능을 배포합니다. 대규모 코드베이스의 문제점 중 하나는 잘 구성된 환경이 특정 사용자층에만 국한될 수 있다는 것입니다. 플러그인은 스킬, 훅, MCP 구성을 하나의 설치 가능한 패키지로 묶어 제공하므로, 신입 엔지니어가 입사 첫날 해당 플러그인을 설치하면 기존 Claude 사용자와 동일한 컨텍스트와 기능을 즉시 활용할 수 있습니다. 플러그인 업데이트는 관리형 마켓플레이스를 통해 조직 전체에 배포할 수 있습니다.

예를 들어, 저희와 협력하는 대형 소매 기업은 비즈니스 분석가가 워크플로를 벗어나지 않고도 성과 데이터를 가져올 수 있도록 Claude를 내부 분석 플랫폼에 연결하는 스킬을 개발했습니다. 이 스킬은 기업 전체에 배포하기 전에 플러그인 형태로 제공되었습니다.

언어 서버 프로토콜(LSP) 통합을 통해 Claude는 개발자가 IDE에서 사용하는 것과 동일한 탐색 기능을 제공합니다. 대부분의 대규모 코드베이스 IDE는 이미 LSP를 실행하여 "정의로 이동" 및 "모든 참조 찾기" 기능을 지원합니다. Claude에 LSP를 제공하면 심볼 수준의 정밀도를 확보할 수 있습니다. 즉, 함수 호출을 따라 정의까지 이동하고, 파일 간 참조를 추적하며, 서로 다른 언어로 작성된 동일한 이름의 함수를 구분할 수 있습니다. LSP가 없으면 Claude는 텍스트 패턴 매칭을 사용하므로 잘못된 심볼을 찾을 수 있습니다. 저희와 협력했던 한 엔터프라이즈 소프트웨어 회사는 Claude Code를 배포하기 전에 조직 전체에 LSP 통합을 구축하여 특히 대규모 환경에서 C 및 C++ 탐색의 안정성을 확보했습니다. 다국어 코드베이스의 경우 이는 가장 가치 있는 투자 중 하나입니다.

MCP 서버는 모든 것을 확장합니다. Claude는 MCP 서버를 통해 자체적으로 접근할 수 없는 내부 도구, 데이터 소스 및 API에 연결합니다. 가장 정교한 팀은 Claude가 직접 호출할 수 있는 도구로 구조화된 검색 기능을 제공하는 MCP 서버를 구축했습니다. 다른 팀은 Claude를 내부 문서, 티켓팅 시스템 또는 분석 플랫폼에 연결합니다.

서브에이전트는 탐색과 편집을 분리합니다. 서브에이전트는 자체 컨텍스트 창을 가진 격리된 Claude 인스턴스로, 작업을 받아 수행한 후 최종 결과만 상위 에이전트에 반환합니다. 시스템이 구축되면 일부 팀은 읽기 전용 서브에이전트를 실행하여 하위 시스템을 매핑하고 결과를 파일에 기록한 다음, 메인 에이전트에서 전체 정보를 바탕으로 편집합니다.

아래 표는 각 구성 요소의 기능, 로드 시점, 그리고 각 구성 요소에서 가장 흔히 발생하는 오류를 요약한 것입니다:

## 성공적인 배포 사례에서 나타나는 세 가지 구성 패턴

대규모 코드베이스에 맞게 Claude Code를 구성하는 방법은 해당 코드베이스의 구조에 따라 크게 달라집니다. 하지만 저희가 관찰한 배포 사례들에서 공통적으로 세 가지 패턴이 나타났습니다.

### 대규모 환경에서도 코드베이스를 쉽게 탐색할 수 있도록 만들기

클로드(Claude)가 대규모 코드베이스에서 도움을 줄 수 있는 능력은 적절한 컨텍스트를 찾는 능력에 달려 있습니다.
매 세션마다 너무 많은 컨텍스트가 로드되면 성능이 저하되고, 컨텍스트가 너무 적으면 클로드는 제대로 탐색할 수 없습니다.
가장 효과적인 배포는 클로드가 코드베이스를 쉽게 이해할 수 있도록 사전에 투자하는 것입니다. 몇 가지 공통적인 패턴이 있습니다.

- **CLAUDE.md 파일을 간결하고 계층적으로 유지** 클로드는 코드베이스를 탐색하면서 파일을 점진적으로 로드합니다.
  루트 파일은 전체적인 그림을, 하위 디렉터리 파일은 로컬 규칙을 나타냅니다.
  루트 파일에는 포인터와 중요한 주의 사항만 포함되어야 하며, 그 외의 모든 내용은 불필요한 정보입니다.

- **리포지토리 루트가 아닌 하위 디렉토리에서 초기화합니다.** Claude는 코드베이스에서 실제로 작업과 관련된 부분에 범위가 지정될 때 가장 잘 작동합니다.
  모노리포 환경에서는 툴이 종종 루트 액세스를 가정하기 때문에 직관적이지 않게 느껴질 수 있지만,
  Claude는 디렉토리 트리를 자동으로 탐색하고 그 과정에서 발견되는 모든 CLAUDE.md 파일을 로드하므로 루트 수준의 컨텍스트가 손실되지 않습니다.

- **하위 디렉토리별로 테스트 및 린트 명령의 범위를 지정합니다.** Claude가 하나의 서비스를 변경했을 때 전체 스위트를 실행하면 시간 초과가 발생하고
  관련 없는 출력에 컨텍스트가 낭비됩니다.
  하위 디렉토리 수준의 CLAUDE.md 파일에는 해당 코드베이스 부분에 적용되는 명령이 지정되어야 합니다.
  이는 각 디렉토리에 자체 테스트 및 빌드 명령이 있는 서비스 지향 코드베이스에 적합합니다. 컴파일 언어로 작성된 모노레포에서 디렉터리 간 종속성이 깊은 경우, 하위 디렉터리별 범위 지정이 더 어려워지며 프로젝트별 빌드 구성이 필요할 수 있습니다.

- **`.ignore` 파일을 사용하여 생성된 파일, 빌드 아티팩트 및 타사 코드를 제외합니다.** `.claude/settings.json`에 `permissions.deny` 규칙을 커밋하면 제외 사항이 버전 관리되므로 팀의 모든 개발자가 직접 구성하지 않고도 동일한 노이즈 감소 효과를 얻을 수 있습니다.
  일부 코드베이스에서는 생성된 파일 자체가 개발 작업의 대상입니다.
  코드 생성기를 사용하는 개발자는 팀의 나머지 구성원에게 영향을 주지 않고 로컬 설정에서 프로젝트 수준의 제외 사항을 재정의할 수 있습니다.

- **디렉토리 구조가 충분하지 않을 때 코드베이스 맵을 구축합니다.** 코드가 일반적인 디렉토리 구조로 통합되지 않은 조직의 경우, 저장소 루트에 각 최상위 폴더와 해당 폴더에 있는 내용에 대한 한 줄 설명을 나열하는 간단한 마크다운 파일을 만들면 Claude가 파일을 열기 전에 스캔할 수 있는 목차를 제공합니다. 최상위 폴더가 수백 개에 달하는 코드베이스의 경우,
  계층형 접근 방식이 가장 효과적입니다. 루트 파일은 최상위 구조만 설명하고, 하위 디렉터리의 CLAUDE.md 파일은 다음 단계의 세부 정보를 제공하며, Claude가 트리를 탐색할 때 필요에 따라 로드됩니다.
  더 간단한 경우에는 Claude가 참조해야 하는 특정 파일이나 디렉터리를 @멘션하는 것으로도 동일한 효과를 얻을 수 있습니다.

- **Claude가 문자열이 아닌 심볼로 검색하도록 LSP 서버를 실행합니다.** 대규모 코드베이스에서 일반적인 함수 이름을 grep으로 검색하면 수천 개의 일치 항목이 반환되고, Claude는 어떤 것이 중요한지 파악하기 위해 파일을 여는 데 컨텍스트를 소모합니다.
  LSP는 동일한 심볼을 가리키는 참조만 반환하므로,
  Claude가 코드를 읽기 전에 필터링이 수행됩니다. ****이 설정을 위해서는 해당 언어에 맞는 [코드 인텔리전스 플러그인](https://code.claude.com/docs/en/discover-plugins?utm_content=inline_link&utm_source=it&utm_medium=email&utm_campaign=2026_Q2_SS_MKTG_Dev_Newsletter_May&utm_term=claude_code&utm_campaignId=18227556#code-intelligence)과 해당 언어 서버 바이너리를 설치해야 합니다.
  Claude Code 문서에서 사용 가능한 플러그인 및 문제 해결 방법을 확인할 수 있습니다.


**한 가지 주의 사항:** 계층적 CLAUDE.md 접근 방식조차도 제대로 작동하지 않는 예외적인 경우가 있습니다. 예를 들어 수십만 개의 폴더와 수백만 개의 파일로 구성된 코드베이스 또는 Git이 아닌 다른 버전 관리 시스템을 사용하는 레거시 시스템과 같은 경우입니다. 이러한 문제에 대한 해결책은 이 시리즈의 향후 편에서 다루겠습니다.

### 모델 지능이 발전함에 따라 CLAUDE.md 파일을 적극적으로 관리합니다.

모델이 발전함에 따라 현재 모델을 위해 작성된 지침이 향후 모델에 오히려 방해가 될 수 있습니다. 예를 들어, 이전 모델에서 어려움을 겪었던 패턴을 따라 Claude를 안내했던 CLAUDE.md 파일은 차기 모델이 출시될 때 불필요해지거나 오히려 제약 조건이 될 수 있습니다. CLAUDE.md 파일에 모든 리팩토링을 단일 파일 변경으로 분리하도록 지시하는 규칙이 있다면 이전 모델에서는 도움이 되었을 수 있지만, 최신 모델에서는 여러 파일에 걸쳐 조정된 편집을 원활하게 수행하지 못하게 할 수 있습니다.

모델의 추론 방식이나 Claude Code 자체 도구의 한계를 보완하기 위해 구축된 스킬과 훅은 해당 한계가 사라지면 오히려 불필요한 오버헤드가 됩니다. 예를 들어, Perforce 코드베이스에서 p4 편집을 강제하기 위해 파일 쓰기를 가로채는 훅은 Claude Code에 Perforce 네이티브 모드가 추가되면서 더 이상 필요 없어졌습니다.

팀은 3~6개월마다 의미 있는 구성 검토를 수행해야 하지만, 주요 모델 릴리스 이후 성능이 정체되었다고 느껴질 때도 검토를 진행하는 것이 좋습니다.

### Claude Code 관리 및 도입에 대한 책임 소재 규명

기술적 구성만으로는 도입을 촉진할 수 없습니다. 성공적인 조직들은 조직 차원에도 투자를 아끼지 않았습니다.

가장 빠르게 확산된 사례들은 광범위한 접근을 허용하기 전에 전용 인프라에 투자한 것이 특징입니다. 소규모 팀, 심지어 단 한 명의 인원이 툴링을 구축하여 개발자들이 처음 접할 때부터 클로드(Claude)를 활용할 수 있도록 워크플로우를 최적화했습니다. 한 회사에서는 두 명의 엔지니어가 플러그인 및 MCP(Master Code Provider) 제품군을 개발하여 출시 당일부터 사용할 수 있도록 했습니다. 또 다른 회사에서는 AI 코딩 툴 관리에 집중하는 팀 전체가 출시 시작 전에 인프라를 구축해 놓았습니다. 두 경우 모두 개발자들의 첫 경험이 좌절감이 아닌 생산적이었고, 이를 계기로 도입이 확산되었습니다.

오늘날 이러한 업무를 담당하는 팀은 대개 개발자 경험 또는 개발자 생산성 부서에 속하며, 이 부서는 일반적으로 신규 엔지니어 온보딩 및 개발자 도구 구축을 담당합니다. 여러 조직에서 새롭게 부상하는 역할은 에이전트 관리자입니다. 이는 클로드 코드(Claude Code) 생태계를 관리하는 PM/엔지니어 겸직 형태입니다. 전담 팀이 없는 조직의 경우, 최소한의 필수 구성 요소는 DRI(Dual Resource Initiative)입니다. DRI는 클로드 코드 구성에 대한 소유권, 설정 변경 권한, 권한 정책, 플러그인 마켓플레이스 및 CLAUDE.md 규칙에 대한 책임, 그리고 이러한 사항들을 최신 상태로 유지할 책임을 가진 담당자입니다.

상향식 도입은 열정을 불러일으키지만, 효과적인 방법을 중앙 집중화할 사람이 없으면 파편화될 수 있습니다. 적절한 클로드 코드 규칙(예: 표준화된 CLAUDE.md 계층 구조 또는 엄선된 스킬 및 플러그인 세트)을 수립하고 전파할 담당자 또는 팀이 필요합니다. 이러한 노력이 없다면 지식은 파벌에 머물고 도입은 정체될 것입니다.

대규모 조직, 특히 규제 산업에 속한 조직에서는 거버넌스 관련 질문이 초기에 제기됩니다. 예를 들어, 어떤 기술과 플러그인을 사용할 수 있는지 누가 통제하는지, 수천 명의 엔지니어가 독립적으로 동일한 것을 재구축하는 것을 어떻게 방지하는지, AI가 생성한 코드가 사람이 작성한 코드와 동일한 검토 과정을 거치도록 어떻게 보장하는지 등이 있습니다. 이러한 문제를 조기에 해결하기 위해, 승인된 기술 세트, 필수 코드 검토 프로세스, 그리고 제한적인 초기 접근 권한으로 시작하여, 자신감이 쌓이면 점차 확대해 나가는 방식을 제안합니다.

엔지니어링, 정보 보안 및 거버넌스 담당자를 모아 요구 사항을 함께 정의하고 배포 로드맵을 구축하는 교차 기능 워킹 그룹을 조기에 구성하는 조직에서 가장 원활한 배포가 이루어지는 것을 확인했습니다.

### 이러한 패턴을 조직에 적용하기

Claude Code는 엔지니어가 코드베이스에 주로 기여하고, 리포지토리가 Git을 사용하며, 코드가 표준 디렉터리 구조를 따르는 기존 소프트웨어 엔지니어링 환경을 기반으로 설계되었습니다. 대부분의 대규모 코드베이스는 이러한 틀에 부합하지만, 대용량 바이너리 애셋을 사용하는 게임 엔진, 일반적이지 않은 버전 관리 시스템을 사용하는 환경, 또는 엔지니어가 아닌 구성원이 코드베이스에 기여하는 경우와 같은 비전통적인 환경에서는 추가적인 구성 작업이 필요합니다. 본 가이드는 기존 환경을 전제로 하며, 설명된 패턴은 이미 많은 고객에게 적용되어 효과를 입증했습니다. 남아 있는 복잡한 문제는 코드베이스, 도구, 조직 환경에 따른 구체적인 판단이 필요합니다. 바로 이 부분에서 Anthropic의 응용 AI 팀이 엔지니어링 팀과 직접 협력하여 이러한 패턴을 조직의 특정 요구 사항에 맞게 적용해 드립니다.
