# Claude Code가 대규모 코드베이스에서 작동하는 방식 — 베스트 프랙티스와 출발점

> 가장 성공적인 Claude Code 도입 사례들을 보면 설정, 도구, 조직 구조 전반에 공통된 패턴이 보입니다. 이 글은 엔지니어링 조직이 엔터프라이즈 규모에서 Claude Code를 운영할 때 참고할 만한 베스트 프랙티스를 다루는 시리즈 "Claude Code at Scale"의 일부입니다.

---

Claude Code는 수백만 줄 규모의 모노레포, 수십 년 묵은 레거시 시스템, 수십 개 레포로 흩어진 분산 아키텍처, 수천 명 규모의 개발 조직에서 실제로 잘 굴러가고 있습니다. 이런 환경에는 작고 단순한 코드베이스에서는 만나기 어려운 문제들이 따라옵니다. 하위 디렉터리마다 빌드 명령이 다르거나, 루트가 공유되지 않은 채 여러 폴더로 흩어져 있는 레거시 코드 같은 것들이죠.

이 글은 Claude Code를 대규모로 도입하면서 효과를 본 패턴들을 정리합니다. 여기서 말하는 "대규모 코드베이스"는 폭이 넓습니다. 수백만 줄짜리 모노레포, 수십 년에 걸쳐 쌓인 레거시, 여러 레포로 쪼개진 수십 개의 마이크로서비스 모두를 포함합니다. C, C++, C#, Java, PHP처럼 AI 코딩 도구와는 다소 거리가 멀다고 여겨지던 언어의 코드베이스도 마찬가지입니다. (특히 최근 모델 릴리스 기준으로는, 대부분의 팀이 예상하는 것보다 Claude Code가 훨씬 잘 해냅니다.) 모든 대규모 배포는 결국 그 조직의 버전 관리 방식, 팀 구조, 누적된 컨벤션에 맞춰 모양이 달라지지만, 여기서 소개하는 패턴들은 그런 차이를 가로질러 일반화될 수 있고, 도입을 검토 중인 팀에게 좋은 출발점이 됩니다.

## Claude Code는 어떻게 대규모 코드베이스를 탐색하는가

Claude Code는 사람 엔지니어가 코드베이스를 읽는 방식 그대로 움직입니다. 파일 시스템을 따라 이동하고, 파일을 읽고, grep으로 필요한 부분을 정확히 짚어내고, 코드베이스 곳곳의 참조를 따라갑니다. 이 작업이 모두 개발자 로컬 머신에서 일어나기 때문에 코드베이스 인덱스를 따로 만들고 유지하거나 서버로 올릴 필요가 없습니다.

RAG 기반의 AI 코딩 도구는 코드베이스 전체를 임베딩해두고 질의 시점에 관련 청크를 가져오는 방식으로 작동합니다. 문제는 대규모 환경에서 이 방식이 잘 안 굴러간다는 점입니다. 임베딩 파이프라인이 활발하게 작업하는 팀의 속도를 따라가지 못해, 개발자가 인덱스를 조회할 때쯤이면 인덱스는 이미 몇 주, 며칠, 길어야 몇 시간 전의 코드베이스를 비추고 있습니다. 그 결과 2주 전에 이름이 바뀐 함수, 지난 스프린트에 삭제된 모듈이 검색 결과에 그대로 끌려나오면서도, 이 정보가 낡았다는 표시는 어디에도 없습니다.

**에이전틱 검색(agentic search)** 은 이 문제를 정면으로 풉니다. 수천 명의 엔지니어가 새 코드를 커밋해도 유지해야 할 임베딩 파이프라인이나 중앙 인덱스가 없습니다. 각 개발자 인스턴스는 매번 실제 코드베이스 위에서 직접 동작합니다.

다만 이 방식에는 분명한 트레이드오프가 있습니다. Claude는 "어디를 봐야 할지" 알 수 있을 만큼 초기 컨텍스트가 충분히 주어졌을 때 가장 잘 작동합니다. 즉, 탐색 품질은 코드베이스가 얼마나 잘 정돈되어 있는지, 다시 말해 CLAUDE.md 파일과 스킬을 통해 컨텍스트가 계층적으로 잘 깔려 있는지에 달려 있습니다. 수십억 줄 규모의 코드베이스에서 모호한 패턴을 빠짐없이 찾아 달라고 시키면, 일을 시작하기도 전에 컨텍스트 윈도우 한계에 부딪힙니다. 코드베이스 셋업에 투자하는 팀일수록 더 좋은 결과를 얻는 이유가 여기에 있습니다.

## 모델만큼 중요한 것이 하네스(harness)다

Claude Code에 대한 가장 흔한 오해는 "결국 모델 성능이 전부다"라는 것입니다. 팀들은 벤치마크 점수와 특정 태스크의 성능만 들여다봅니다. 하지만 실제 결과를 가르는 것은 모델 그 자체가 아니라 모델을 둘러싼 생태계 — 즉 **하네스** — 인 경우가 더 많습니다.

하네스는 다섯 개의 확장 포인트로 구성됩니다. CLAUDE.md 파일, 훅(hook), 스킬(skill), 플러그인(plugin), MCP 서버. 각 계층은 앞 계층 위에 얹히는 구조라서, 팀이 어떤 순서로 이걸 쌓아 올리는지가 중요합니다. 여기에 LSP 통합과 서브에이전트라는 두 가지가 더 붙으면 그림이 완성됩니다. 아래에서 각 구성 요소를 하나씩 짚어보겠습니다.

**CLAUDE.md 파일이 제일 먼저입니다.** Claude가 매 세션 시작 시 자동으로 읽어들이는 컨텍스트 파일이죠. 루트 파일은 큰 그림을, 하위 디렉터리 파일은 그 영역의 로컬 컨벤션을 담습니다. 이 파일들이 Claude가 효율적으로 일하는 데 필요한 코드베이스 정보를 제공합니다. 작업 종류와 무관하게 모든 세션에 로드되기 때문에, 폭넓게 적용되는 내용만 담는 게 성능을 지키는 핵심입니다.

**훅은 셋업 자체를 자가 개선합니다.** 대부분의 팀은 훅을 "Claude의 실수를 막아주는 스크립트" 정도로 생각하지만, 훅의 진짜 가치는 지속적인 개선에 있습니다. Stop 훅은 한 세션에서 일어난 일을 분석해서, 컨텍스트가 아직 신선할 때 CLAUDE.md 업데이트를 제안할 수 있습니다. SessionStart 훅은 팀별 컨텍스트를 동적으로 끌어와서, 개발자가 수동 설정 없이도 자기 모듈에 맞는 환경을 바로 갖추게 해줍니다. 린트나 포맷팅처럼 자동화로 풀어야 하는 검사는 훅이 결정적(deterministic)으로 강제하는 편이, "이 명령을 기억해 두라"고 Claude에게 시키는 것보다 훨씬 일관됩니다.

**스킬은 필요한 전문성을 필요할 때만 끌어옵니다.** 대규모 코드베이스에는 수십 가지 작업 유형이 있고, 그 모든 전문성이 매 세션에 필요한 건 아닙니다. 스킬은 progressive disclosure를 통해, 컨텍스트 공간을 두고 경쟁할 만한 전문 워크플로와 도메인 지식을 따로 빼두었다가 그 작업에 들어갈 때만 로드합니다. 예를 들어 보안 리뷰 스킬은 Claude가 취약점을 점검할 때만 로드되고, 문서화 스킬은 코드 변경 후 문서를 업데이트해야 할 때만 끼어듭니다.

스킬은 특정 경로로 스코프를 좁힐 수도 있어서, 코드베이스 중에서도 관련된 영역에서만 활성화되도록 만들 수 있습니다. 결제 서비스 팀은 배포 스킬을 자기 디렉터리에 묶어두고, 모노레포의 다른 위치에서 작업할 때는 자동 로딩이 일어나지 않게 할 수 있습니다.

**플러그인은 잘 다듬어진 셋업을 배포합니다.** 대규모 환경의 흔한 문제 중 하나가, 잘 정돈된 환경이 특정 그룹 안에만 머무른다는 것입니다. 플러그인은 스킬·훅·MCP 설정을 하나의 설치 가능한 패키지로 묶기 때문에, 신입 엔지니어가 입사 첫날 플러그인 하나만 설치하면 기존 사용자와 동일한 컨텍스트와 능력을 즉시 갖출 수 있습니다. 업데이트는 관리형 마켓플레이스를 통해 조직 전체로 한 번에 배포됩니다.

저희와 협업한 한 대형 소매 기업은, 비즈니스 분석가가 워크플로를 벗어나지 않고도 성과 데이터를 가져올 수 있도록 Claude를 내부 분석 플랫폼에 연결하는 스킬을 만들었습니다. 이 스킬은 전사에 배포되기 전, 먼저 플러그인 형태로 묶여 제공되었습니다.

**LSP(Language Server Protocol) 통합**을 켜면 Claude도 개발자가 IDE에서 누리는 것과 같은 탐색 능력을 갖게 됩니다. 대부분의 대규모 코드베이스용 IDE는 이미 LSP를 띄워두고 "정의로 이동", "모든 참조 찾기"를 지원합니다. Claude에게 LSP를 쥐여주면 심볼 단위 정밀도가 생깁니다. 함수 호출을 따라 정의까지 따라가고, 파일을 가로지르는 참조를 추적하며, 서로 다른 언어에 같은 이름의 함수가 있어도 헷갈리지 않습니다. LSP가 없으면 Claude는 텍스트 패턴 매칭에 의존하기 때문에 엉뚱한 심볼을 잡을 수도 있죠. 저희와 함께 일한 한 엔터프라이즈 소프트웨어 회사는 Claude Code 도입 **이전에** 전사에 LSP 통합을 깔아두고서야 C/C++ 탐색의 신뢰성을 확보했습니다. 다국어 코드베이스라면 LSP는 가장 가성비 좋은 투자 중 하나입니다.

**MCP 서버는 그 모든 것을 바깥으로 확장합니다.** Claude는 MCP 서버를 통해 자체적으로는 접근할 수 없는 내부 도구, 데이터 소스, API에 연결됩니다. 가장 성숙한 팀들은 한 발 더 나아가, Claude가 직접 호출할 수 있는 도구 형태로 **구조화된 검색**을 노출하는 MCP 서버를 직접 만들기도 합니다. 다른 팀들은 내부 문서, 티켓 시스템, 분석 플랫폼 같은 곳에 Claude를 붙입니다.

**서브에이전트는 탐색과 편집을 분리해줍니다.** 서브에이전트는 자기만의 컨텍스트 윈도우를 가진 격리된 Claude 인스턴스입니다. 작업을 받아 처리한 뒤 최종 결과만 부모 에이전트에 돌려줍니다. 어느 정도 시스템이 자리 잡은 팀들은 read-only 서브에이전트를 띄워 서브시스템을 매핑하고 결과를 파일에 적어두게 한 다음, 메인 에이전트는 그 정보를 받아서 실제 편집을 수행합니다.

아래 표는 각 구성 요소의 역할, 로드 시점, 가장 흔한 오해를 한눈에 정리한 것입니다.

| 구성 요소 | 정의 | 로드 시점 | 가장 잘 맞는 용도 | 자주 보이는 오해·실수 |
|---|---|---|---|---|
| **CLAUDE.md** | Claude가 자동으로 읽는 컨텍스트 파일 | 모든 세션 | 프로젝트 전반의 규칙, 코드베이스 지식 | 스킬에 들어가야 할 재사용성 높은 전문 지식까지 욱여넣음 |
| **훅(hook)** | 주요 시점에 트리거되는 스크립트 | 이벤트 발생 시 | 일관된 동작 자동화, 세션에서 얻은 학습 기록 | 자동화로 풀 일을 매번 프롬프트로 시킴 |
| **스킬(skill)** | 특정 작업 유형용으로 묶인 지침 패키지 | 필요할 때 on-demand | 세션·프로젝트를 가로지르는 재사용성 높은 전문 지식 | 전부 CLAUDE.md에 몰아넣어버림 |
| **플러그인(plugin)** | 스킬·훅·MCP 설정을 한 묶음 | 한 번 구성하면 상시 사용 | 검증된 셋업을 조직 전체로 배포 | 잘 만든 셋업을 한 팀에만 묶어둠 |
| **LSP\*** | 언어 서버를 통한 실시간 코드 인텔리전스 | 한 번 구성하면 상시 사용 | 타입 있는 언어에서 심볼 단위 탐색, 자동 에러 감지 | 자동으로 켜져 있다고 가정 |
| **MCP 서버** | 외부 도구·데이터와의 연결 | 한 번 구성하면 상시 사용 | Claude가 다른 방법으로는 못 닿는 내부 도구 접근 | 기본기를 갖추기 전부터 MCP부터 깔려고 함 |
| **서브에이전트\*** | 특정 작업용 별도 Claude 인스턴스 | 호출 시 | 탐색과 편집의 분리, 병렬 작업 | 같은 세션에서 탐색과 편집을 같이 굴림 |

_\*LSP는 플러그인 계층을 통해 접근합니다. 서브에이전트는 별도 확장 포인트라기보다 "위임(delegation) 능력"에 가깝습니다._

## 성공적인 배포에서 반복적으로 보이는 세 가지 구성 패턴

대규모 코드베이스에 맞춰 Claude Code를 어떻게 구성할지는 그 코드베이스의 구조에 크게 좌우됩니다. 그럼에도 저희가 본 배포 사례 전반에서 공통으로 등장하는 세 가지 패턴이 있습니다.

### 1. 대규모 환경에서도 코드베이스를 "탐색 가능하게" 만들기

Claude가 대규모 코드베이스에서 도움이 될 수 있는지는 결국 **적절한 컨텍스트를 찾아내는 능력**에 달려 있습니다. 매 세션 너무 많은 컨텍스트를 끌고 가면 성능이 떨어지고, 너무 적게 가져가면 Claude가 길을 잃습니다. 가장 효과적인 배포 사례들은 Claude가 코드베이스를 쉽게 이해할 수 있도록 사전 작업에 시간을 들였습니다. 다음과 같은 공통 패턴들이 보입니다.

- **CLAUDE.md를 가볍게, 그리고 계층적으로 둔다.** Claude는 코드베이스를 따라 이동하며 파일을 점진적으로 로드합니다. 루트 파일은 큰 그림, 하위 디렉터리 파일은 로컬 컨벤션을 담습니다. 루트에는 핵심 포인터와 결정적인 함정(gotcha)만 들어가야 하고, 그 외의 모든 것은 노이즈입니다.

- **레포 루트가 아니라 하위 디렉터리에서 init 한다.** Claude는 작업과 실제로 관련된 영역으로 스코프가 좁혀졌을 때 가장 잘 작동합니다. 모노레포 환경에서는 "도구는 루트에서 돌리는 것"이라는 기존 관념 때문에 이게 직관에 어긋나 보일 수 있지만, Claude는 디렉터리 트리를 자동으로 거슬러 올라가며 만나는 CLAUDE.md를 모두 로드하기 때문에 루트 컨텍스트는 잃지 않습니다.

- **테스트·린트 명령을 하위 디렉터리별로 지정한다.** 서비스 하나만 바꿨는데 전체 테스트 스위트를 돌리면 타임아웃이 나고, 무관한 출력이 컨텍스트를 잡아먹습니다. 하위 디렉터리 수준의 CLAUDE.md에는 그 영역에 맞는 명령이 명시돼야 합니다. 서비스마다 자체 테스트·빌드 명령을 갖는 서비스 지향 코드베이스에 특히 잘 맞습니다. 컴파일 언어로 작성됐고 디렉터리 간 의존성이 깊은 모노레포에서는 하위 디렉터리 단위 스코핑이 까다로워질 수 있고, 프로젝트 단위의 별도 빌드 설정이 필요할 수도 있습니다.

- **`.ignore` 파일을 활용해 생성 파일, 빌드 산출물, 서드파티 코드를 걸러낸다.** `permissions.deny` 규칙을 `.claude/settings.json`에 커밋해서 버전 관리하면, 팀 모두가 각자 설정하지 않아도 동일한 노이즈 감소 효과를 누립니다. 어떤 코드베이스에서는 생성된 파일 자체가 작업 대상이기도 한데, 그런 경우 코드 제너레이터를 다루는 개발자만 로컬 설정에서 프로젝트 단위 제외 규칙을 오버라이드하면 됩니다. 팀 전체에는 영향을 주지 않습니다.

- **디렉터리 구조만으로 부족하다면 코드베이스 맵을 만든다.** 일반적인 디렉터리 구조로 정리되지 않은 조직이라면, 레포 루트에 가벼운 마크다운 파일 하나를 두고 최상위 폴더와 각 폴더의 내용을 한 줄씩 적어두는 것만으로도, Claude가 파일을 열기 전에 훑어볼 수 있는 "목차"가 됩니다. 최상위 폴더가 수백 개에 달한다면 계층적으로 가는 편이 낫습니다. 루트 파일은 최상위 구조만 설명하고, 하위 디렉터리의 CLAUDE.md가 다음 레벨의 세부 정보를 담아 Claude가 트리를 내려갈 때 필요한 만큼만 로드되도록 합니다. 더 단순한 경우엔, Claude가 봐줬으면 하는 파일·디렉터리를 그냥 `@`-mention 하는 것만으로도 같은 효과를 낼 수 있습니다.

- **문자열이 아니라 심볼로 검색하도록 LSP 서버를 띄운다.** 대규모 코드베이스에서 흔한 함수 이름을 grep 하면 수천 건의 일치 결과가 쏟아지고, Claude는 그중 뭐가 중요한지 가리려고 파일을 여느라 컨텍스트를 소모합니다. LSP는 같은 심볼을 가리키는 참조만 돌려주기 때문에, Claude가 코드를 읽기도 전에 이미 필터링이 끝난 상태가 됩니다. 셋업을 위해서는 해당 언어용 [code intelligence 플러그인](https://code.claude.com/docs/en/discover-plugins#code-intelligence)과 언어 서버 바이너리 설치가 필요합니다. 사용 가능한 플러그인과 트러블슈팅은 Claude Code 문서에서 확인할 수 있습니다.

**한 가지 짚어둘 점.** 위와 같은 계층적 CLAUDE.md 접근법으로도 잘 풀리지 않는 예외 케이스가 있습니다. 수십만 개의 폴더와 수백만 개의 파일로 된 코드베이스, Git이 아닌 버전 관리 시스템을 쓰는 레거시 시스템 같은 경우입니다. 이런 문제에 대한 해법은 이 시리즈의 후속 글에서 다룰 예정입니다.

### 2. 모델이 똑똑해질수록 CLAUDE.md를 적극적으로 손본다

모델이 진화하면, **지금 모델을 위해 쓴 지시문이 차기 모델에는 도리어 발목을 잡을 수 있습니다.** 예를 들어 이전 모델이 어려워하던 패턴을 우회하도록 Claude를 안내하던 CLAUDE.md는, 다음 모델이 나오면 불필요해지거나 심하면 제약이 됩니다. "모든 리팩토링은 한 파일씩 분리해서 진행하라"는 규칙이 옛 모델에는 도움이 됐을 수 있지만, 최신 모델이 여러 파일에 걸친 조율된 편집을 잘 해내는 상황에서는 그 능력을 막아버립니다.

특정 모델의 추론 한계나 Claude Code 도구의 한계를 보정하려고 만들어둔 스킬·훅도, 그 한계가 사라지면 그저 오버헤드로 남습니다. 예전에 Perforce 코드베이스에서 `p4 edit`을 강제하려고 파일 쓰기를 가로채던 훅이 있었는데, Claude Code에 Perforce 네이티브 모드가 추가되면서 더는 필요 없어진 사례가 있습니다.

권장은 **3~6개월마다** 의미 있는 설정 리뷰를 한 번씩 돌리는 것입니다. 다만 주요 모델 릴리스 직후나 "왜인지 성능이 정체된 것 같다"는 느낌이 들 때도 한 번씩 들여다보는 게 좋습니다.

### 3. Claude Code의 운영·도입에 대한 책임 소재를 분명히 한다

기술적 셋업만으로는 도입이 굴러가지 않습니다. 성공한 조직들은 한결같이 **조직 차원에서도** 투자를 했습니다.

가장 빠르게 확산된 사례들은 광범위한 접근을 풀기 **전에** 전담 인프라에 먼저 투자했습니다. 작은 팀, 때로는 단 한 명이 도구를 정비해 두어, 개발자가 처음 Claude를 만나는 순간 그 워크플로에 잘 맞도록 다듬어 놨습니다. 어느 회사는 출시 당일부터 쓸 수 있도록 두 명의 엔지니어가 플러그인과 MCP 서버 묶음을 미리 만들어 두었습니다. 또 다른 회사는 출시 시점 전에 AI 코딩 도구 운영만 전담하는 팀이 인프라를 다 세팅해 두었습니다. 두 경우 모두 개발자들의 첫 경험이 **답답함이 아니라 생산성**이었고, 그 위에서 도입이 자연스럽게 번졌습니다.

오늘날 이 일을 맡는 팀은 보통 Developer Experience 또는 Developer Productivity 조직에 속합니다. 신규 엔지니어 온보딩과 개발자 도구 구축도 일반적으로 같은 팀의 책임입니다. 여러 조직에서 새로 떠오르고 있는 역할이 있는데, 바로 **"에이전트 매니저(Agent Manager)"** 입니다. Claude Code 생태계 관리를 전담하는 PM/엔지니어 하이브리드 포지션이죠. 전담 팀까지는 둘 여건이 안 된다면, 최소한 갖춰야 할 것은 **DRI(Directly Responsible Individual, 직접 책임자)** 한 명입니다. DRI는 Claude Code 설정 전반의 소유권, 설정·권한 정책에 대한 결정 권한, 플러그인 마켓플레이스 관리, CLAUDE.md 컨벤션, 그리고 이 모든 것을 **최신 상태로 유지**할 책임을 함께 갖습니다.

바텀업 도입은 열기를 만들어내지만, 효과적인 방식을 중앙에 모아줄 사람이 없으면 파편화됩니다. 적절한 Claude Code 컨벤션 — 예를 들어 표준화된 CLAUDE.md 계층 구조나, 큐레이션된 스킬·플러그인 세트 — 을 정리하고 전도할 사람 혹은 팀이 필요합니다. 그게 없으면 지식은 부족 단위에 머무르고, 도입은 정체됩니다.

큰 조직, 특히 규제 산업이라면 거버넌스 질문이 일찍부터 따라옵니다. **어떤 스킬과 플러그인을 사용할 수 있도록 허용할지** 누가 통제할 것인가? 수천 명의 엔지니어가 같은 능력을 제각각 재구축하는 일을 어떻게 막을 것인가? AI가 생성한 코드도 사람이 작성한 코드와 동일한 리뷰 프로세스를 거치도록 어떻게 보장할 것인가? 저희가 권하는 출발점은 **승인된 스킬 세트**로 시작하고, 모든 AI 생성 코드에 코드 리뷰 프로세스를 요구하며, 초기 접근 권한은 제한적으로 풀어두는 것입니다. 그리고 신뢰가 쌓이는 만큼 점진적으로 확대합니다.

배포가 가장 매끄럽게 진행된 조직들은, 엔지니어링·정보 보안·거버넌스 담당자를 모은 **크로스펑셔널 워킹그룹**을 초기에 꾸려 요구사항을 함께 정의하고 롤아웃 로드맵까지 같이 그렸습니다.

### 이 패턴들을 우리 조직에 적용하기

Claude Code는 **엔지니어가 코드베이스의 주된 기여자**이고, 레포가 **Git**을 쓰며, 코드가 **표준적인 디렉터리 구조**를 따른다는 전제 위에 설계되어 있습니다. 대부분의 대규모 코드베이스가 이 틀 안에 들어오지만, 다음과 같은 비전통적인 환경은 추가 설정 작업이 필요합니다. 대용량 바이너리 자산을 다루는 게임 엔진, 일반적이지 않은 버전 관리 시스템을 쓰는 환경, 엔지니어가 아닌 사람도 코드베이스에 기여하는 환경 등이죠. 이 글은 전통적인 환경을 전제로 하며, 여기서 설명한 패턴들은 이미 여러 고객사에서 효과를 입증했습니다. 그 너머에 남는 복잡한 문제들은 결국 그 코드베이스, 도구, 조직 환경에 맞춘 구체적인 판단이 필요한 영역인데, 바로 그 지점에서 Anthropic의 **Applied AI 팀**이 엔지니어링 팀과 직접 협업해 이 패턴들을 조직별 요구사항에 맞게 번역해드립니다.
