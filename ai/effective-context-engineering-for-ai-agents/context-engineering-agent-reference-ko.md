# AI 에이전트를 위한 컨텍스트 엔지니어링 — 운영 지식 기반

> 이 문서는 그 자체로 완결된 참고 자료입니다. 추가적인 자료는 필요하지 않습니다.

> 출처: 인류 응용 AI 팀의 컨텍스트 엔지니어링 프레임워크(2025).

---

## 1. 핵심 정의

**컨텍스트**: 단일 LLM 추론 호출 동안 존재하는 모든 토큰의 전체 집합입니다. 여기에는 시스템 프롬프트, 도구 정의, few-shot 예제, 메시지 기록, 검색된 문서, MCP 데이터 및 모델이 볼 수 있는 기타 모든 정보가 포함됩니다.

**컨텍스트 엔지니어링**: 에이전트 작동의 각 단계에서 모델의 컨텍스트 창에 어떤 정보가 들어갈지 선별하고 관리하는 분야입니다. 프롬프트 엔지니어링의 후속 개념으로, 프롬프트의 표현 방식뿐만 아니라 컨텍스트 내의 모든 것을 다루기 때문에 범위가 더 넓습니다.

**어텐션 예산**: LLM은 정보에 어텐션할 수 있는 용량이 제한되어 있습니다. 컨텍스트에 추가되는 모든 토큰은 이 예산의 일부를 소모합니다. 토큰을 추가한다고 해서 성능이 선형적으로 향상되는 것은 아니며, 오히려 성능이 저하될 수 있습니다.

**컨텍스트 로트**: 컨텍스트 길이가 길어질수록 정보 검색 및 추론에 대한 모델 정확도가 감소하는 현상으로 잘 알려져 있습니다. 이는 아키텍처적인 문제로, 트랜스포머의 n² 쌍방향 어텐션 메커니즘이 긴 시퀀스에서 한계에 부딪히기 때문입니다. 이는 수정해야 할 버그가 아니라, 관리해야 할 근본적인 제약 조건입니다.

## 2. 핵심 원칙

**원하는 결과가 나올 가능성을 극대화하는 최소한의 핵심 신호 토큰 집합을 찾으세요.**

단순히 간결함을 추구하는 것이 아닙니다. "최소"라는 것은 모든 토큰이 제 역할을 다한다는 것을 의미합니다. 5,000개의 토큰으로 이루어진 시스템 프롬프트라도 모든 줄이 올바른 동작을 유도한다면 최소한이라고 할 수 있습니다. 반대로 500개의 토큰으로 이루어진 프롬프트라도 절반이 중복된다면 지나치게 비대하다고 볼 수 있습니다.

## 3. 컨텍스트 구성 요소 및 각 구성 요소 최적화 방법

### 3.1 시스템 프롬프트

**"적절한 고도" 규칙**: 시스템 프롬프트는 두 가지 방식으로 실패합니다.

| 실패 유형 | 특징 | 실패 원인 |
|---|---|---|
| 너무 경직됨 (낮은 고도) | 하드코딩된 if-else 문, 모든 예외 상황에 대한 포괄적인 규칙 목록 | 취약하고 유지 관리 비용이 높으며 새로운 입력에 대해 오류 발생 |
| 너무 모호함 (높은 고도) | "유용하고 정확하게", 구체적인 신호 없이 일반적인 목표 설정 | 모델에 실행 가능한 지침 부족, 잘못된 기본값 가정 |

**최적의 지점**: 중요한 경로에서 동작을 제한할 만큼 충분히 구체적이면서도 모델이 새로운 상황에 일반화될 수 있도록 충분히 유연해야 합니다. 하드코딩된 논리가 아닌 휴리스틱을 사용하십시오.

**구조적 모범 사례**:
- 레이블이 지정된 섹션으로 구성(XML 태그 `<instructions>`, `<tool_guidance>` 또는 Markdown 헤더 사용)
- 사용 가능한 최적의 모델에 대한 최소한의 안내로 시작
- 관찰된 오류 모드에 대해서만 지침을 추가하고, 가상의 오류 모드는 추가하지 않음

### 3.2 도구

도구는 에이전트와 환경 간의 계약을 정의합니다. 에이전트가 작동하고 새로운 정보를 검색하는 방식을 결정합니다.

**설계 원칙**:
- 각 도구는 명확한 단일 목적(잘 설계된 함수처럼)을 가진 독립적인 도구여야 합니다.
- 도구 설명은 모호하지 않아야 합니다. 사람이 시나리오에서 어떤 도구를 사용해야 하는지 알 수 없다면 에이전트도 알 수 없습니다.
- 토큰 효율적인 응답을 반환해야 합니다. 불필요한 메타데이터, 장황한 형식 또는 중복 필드를 제거해야 합니다.
- 도구의 총 개수를 최소화해야 합니다. 과도하게 많은 도구는 의사 결정의 모호성을 초래하고 사용되지 않는 도구 정의에 컨텍스트를 낭비합니다.

**일반적인 안티 패턴**: 여러 도구가 동일한 요청을 처리할 수 있는 경우 도구가 중복됩니다. 이로 인해 모델은 실행 대신 도구 중에서 선택하는 데 시간을 낭비하게 됩니다.

### 3.3 예시 (소규모)

- 예상되는 동작을 보여주는 다양하고 대표적인 예시들을 엄선합니다.
- 모든 예외 상황을 규칙으로 나열하지 마십시오. 예시는 규칙보다 패턴을 더 효과적으로 가르쳐줍니다.
- 각 예시는 원하는 동작의 다양한 측면을 보여주어야 합니다.

### 3.4 메시지 기록

- 오래된 대화에는 불필요한 도구 결과, 대체된 결정, 탐색상의 막다른 길과 같은 노이즈가 누적됩니다.
- 적극적인 관리가 없으면 메시지 기록은 주의력 예산의 주요 소비 요소가 됩니다.
- 전략: 오래된 대화를 적극적으로 정리하거나 요약합니다(5절의 압축 참조).

## 4. 컨텍스트 검색 — 에이전트가 정보를 얻는 방법

외부 정보를 컨텍스트에 통합하는 데에는 세 가지 방식이 있습니다.

### 4.1 사전 계산 검색 (기존 RAG)

인덱싱 시점에 문서를 임베드하고, 추론 전에 관련성이 높은 상위 k개의 청크를 검색합니다.

- **장점**: 빠르고 예측 가능한 지연 시간
- **단점**: 오래된 인덱스, 작업 도중에 관련성이 생기는 컨텍스트를 놓칠 수 있음, 검색 품질이 임베드 유사도에 따라 달라지는데, 이는 작업 관련성과 일치하지 않을 수 있음

### 4.2 적시 검색 (에이전트 검색)

에이전트는 경량 참조(파일 경로, 저장된 쿼리, URL)를 보유하고 실행 중에 도구를 사용하여 데이터를 동적으로 컨텍스트에 통합합니다.

- **장점**: 항상 최신 정보를 유지하고, 에이전트가 관련성 체인을 따라갈 수 있으며, 메타데이터(파일 이름, 폴더 구조, 타임스탬프)를 통해 탐색 신호를 무료로 제공받을 수 있습니다.

- **단점**: 속도가 느리고, 잘 설계된 탐색 도구가 필요하며, 적절한 안내 없이는 막다른 길에 컨텍스트를 낭비할 위험이 있습니다.

**예시**: Claude Code는 특정 SQL 쿼리를 작성하고, `head`/`tail`을 사용하여 대용량 파일을 검사하고, `grep`/`glob`으로 코드베이스를 탐색합니다. 전체 파일을 컨텍스트에 로드하지 않습니다.

**점진적 정보 공개**: 각 검색 단계는 다음 단계에 대한 정보를 제공합니다. 파일 크기는 복잡성을 암시하고, 명명 규칙은 목적을 나타내며, 타임스탬프는 최신성을 나타냅니다. 에이전트는 점진적으로 정보를 구축합니다.

### 4.3 하이브리드(권장 기본값)

속도 향상을 위해 필수적인 컨텍스트(예: 프로젝트 구성, 주요 참조 문서)를 미리 로드한 다음, 에이전트가 작업별 세부 정보를 자율적으로 탐색하도록 합니다.

- **예시**: Claude Code는 `CLAUDE.md`를 컨텍스트에 미리 배치한 다음, `grep`/`glob`을 사용하여 필요한 파일을 즉시 찾습니다.
- 최고의 조언: **"효과가 있는 가장 간단한 방법을 사용하세요."** 간단하게 시작하고, 필요할 때만 복잡성을 추가하세요.

## 5. 장기 작업 전략

작업이 컨텍스트 윈도우(수십 분에서 수 시간에 걸쳐)에 담을 수 있는 것보다 더 많은 토큰을 생성할 경우, 다음 기법 중 하나 이상을 사용하십시오.

### 5.1 압축

**내용**: 현재 대화를 요약하고 요약 내용을 사용하여 컨텍스트 윈도우를 다시 시작합니다.

**구현 방법**:
1. 전체 메시지 기록을 요약 프롬프트와 함께 모델에 전달합니다.
2. 유지: 아키텍처 결정, 미해결 문제, 주요 구현 세부 정보, 현재 목표
3. 폐기: 중복되는 도구 출력, 더 이상 사용되지 않는 메시지, 탐색상의 막다른 길
4. 요약 내용과 가장 최근에 액세스한 파일(예: 5개 파일)로 컨텍스트를 다시 초기화합니다.

**튜닝 접근 방식**: 먼저 재현율(모든 것을 캡처)을 최대화한 다음, 정밀도(노이즈 제거)를 개선하기 위해 반복합니다. 복잡하고 실제적인 에이전트 추적 데이터에서 테스트합니다.

**가장 가벼운 방식**: 도구 결과 삭제 - 상담원의 결론은 유지하면서 이전 메시지에서 도구 호출 결과 원본을 제거합니다. 도구 결과는 거의 다시 읽히지 않으므로 이 방식이 가장 안전한 압축 방법입니다.

**가장 적합한 용도**: 양방향 대화 흐름이 많은 작업.

### 5.2 구조화된 메모 작성(에이전트 메모리)

**내용**: 상담원이 작업 실행 중에 외부 저장소(파일, 데이터베이스 또는 메모리 도구)에 영구 메모를 기록하고, 컨텍스트가 재설정된 후 해당 메모를 다시 읽어옵니다.

**작동 방식**:
- 에이전트는 스크래치패드 파일(예: `NOTES.md`, 할 일 목록 또는 구조화된 상태 파일)을 유지 관리합니다.
- 진행 상황 업데이트, 결정 사항, 발견된 종속성 및 다음 단계를 기록합니다.
- 압축 또는 컨텍스트 재설정 후, 에이전트는 자신의 메모를 읽어 작업 상태를 복원합니다.

**실제 사례**: 포켓몬 게임을 플레이하는 AI 에이전트는 목표("1,234걸음 동안 1번 도로에서 피카츄를 목표 레벨 10개 중 8개까지 훈련시켰습니다")를 추적하고, 탐험한 지역의 지도를 유지 관리하며, 전투 전략 메모를 기록했습니다. 이 모든 작업은 메모리 구조에 대한 프롬프트 없이 수행되었습니다. 컨텍스트 재설정 후, 에이전트는 메모를 읽고 수 시간 동안 진행되는 전략을 원활하게 계속 수행했습니다.

**최적의 용도**: 명확한 마일스톤과 측정 가능한 진행 상황이 있는 반복적인 작업.

### 5.3 서브 에이전트 아키텍처

**내용**: 각각 깨끗한 컨텍스트 창에서 작동하는 특화된 자식 에이전트에게 집중된 하위 작업을 위임합니다. 리드 에이전트는 높은 수준에서 조정하고 결과를 종합합니다.

**작동 방식**:
1. 리드 에이전트가 높은 수준의 계획을 수립합니다.
2. 서브 에이전트는 구체적인 지침을 받고, 각 서브 작업에 필요한 모든 컨텍스트 예산을 활용하여 작업을 실행합니다.
3. 각 서브 에이전트는 탐색 과정에서 수만 개의 토큰을 소비할 수 있지만, 최종적으로는 압축된 요약 정보(일반적으로 1,000~2,000개 토큰)만 반환합니다.
4. 리드 에이전트는 서브 에이전트의 상세 탐색 컨텍스트에 영향을 받지 않고 서브 에이전트의 출력을 종합합니다.

**주요 이점**: 관심사 분리. 상세 탐색은 서브 에이전트에서 독립적으로 수행됩니다. 리드 에이전트는 종합 및 의사 결정을 위해 깨끗한 컨텍스트를 유지합니다.

**최적의 용도**: 복잡한 연구, 분석 또는 여러 경로를 병렬적으로 탐색하는 것이 가치를 더하는 모든 작업.

### 5.4 선택 가이드

| 작업 특성                | 권장 기법 |
|----------------------|---|
| 여러 차례의 턴이 있는 긴 대화 흐름 | 압축 |
| 단계별 개발 및 명확한 점검      | 체계적인 노트 필기 |
| 다각적인 조사 또는 분석        | 서브 에이전트 아키텍처 |
| 매우 긴 작업(수 시간)        | 세 가지 모두 조합 |

## 6. 의사결정 프레임워크 — 빠른 참조

에이전트를 구축하거나 구성할 때 다음 체크리스트를 활용하세요.

1. **시스템 프롬프트의 모든 토큰이 제 역할을 다하고 있습니까?** 올바른 동작을 직접적으로 유도하지 않는 모든 것을 제거하세요.
2. **사람이 특정 작업에 어떤 도구를 사용해야 하는지 즉시 알 수 있습니까?** 그렇지 않다면 도구의 경계를 통합하거나 명확히 하세요.
3. **도구 응답이 토큰 효율적입니까?** 장황한 메타데이터, 중복된 서식 및 불필요한 필드를 제거하세요.
4. **이전 메시지 기록이 적극적으로 관리되고 있습니까?** 최소한 긴 대화의 경우 도구 결과 삭제 기능을 구현하세요.
5. **검색 방식이 사전 계산, 즉시 또는 하이브리드 방식입니까?** 기본적으로 하이브리드 방식을 사용하세요. 필수적인 데이터는 미리 로드하고 나머지는 탐색하세요.
6. **긴 작업의 경우 어떤 지속성 전략이 적합합니까?** 흐름에는 압축, 마일스톤에는 메모, 병렬 작업에는 하위 에이전트를 사용하세요.
7. **과도하게 설계하고 있습니까?** 간단하게 시작하세요. 관찰된 실패를 해결하기 위해서만 복잡성을 추가하십시오.

## 7. 기억해야 할 핵심 원칙

- **컨텍스트는 유한하며 효율성이 떨어집니다.** 토큰 수가 많다고 성능이 향상되는 것은 아닙니다.
- **냉정하게 큐레이션하십시오.** 최적의 컨텍스트는 작업을 완료하는 데 필요한 최소한의 컨텍스트입니다.
- **도구는 컨텍스트의 일부입니다.** 과도하게 정의된 도구는 사용하지 않을 때조차도 주의 예산을 낭비합니다.
- **능력 있는 모델은 능력을 발휘하도록 하십시오.** 모델이 개선됨에 따라, 정의를 줄이고 자율성을 높이십시오.
- **컨텍스트 창을 더 크게 만든다고 해서 이러한 문제가 해결되는 것은 아닙니다.** 컨텍스트 노후화와 주의력 저하는 아키텍처적인 문제이며, 단순히 용량의 문제가 아닙니다.
- **"효과가 있는 가장 간단한 것을 하십시오."** 빠르게 변화하는 분야에서는 이것이 최선의 기본 전략입니다.
