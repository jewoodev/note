# 다중 에이전트 연구 시스템을 구축한 방법

> 2025년 6월 13일 발행

## 저희 연구 기능은 여러 개의 Claude 에이전트를 사용하여 복잡한 주제를 더욱 효과적으로 탐색합니다. 이 시스템을 구축하면서 겪었던 엔지니어링 과제와 얻은 교훈을 공유합니다..

클로드(Claude)는 이제 웹, Google Workspace 및 다양한 통합 기능을 활용하여 복잡한 작업을 수행할 수 있는 [연구 기능](https://www.anthropic.com/news/research)을 갖추게 되었습니다.

이 다중 에이전트 시스템을 프로토타입에서 실제 제품으로 개발하는 과정에서 시스템 아키텍처, 도구 설계 및 신속한 엔지니어링에 대한 중요한 교훈을 얻었습니다. 다중 에이전트 시스템은 여러 에이전트(LLM이 도구를 사용하여 루프 내에서 자율적으로 작동)가 함께 작동하는 시스템입니다. 저희의 연구 기능은 사용자 쿼리를 기반으로 연구 프로세스를 계획하는 에이전트와, 도구를 사용하여 정보를 동시에 검색하는 병렬 에이전트를 생성하는 에이전트로 구성됩니다. 다중 에이전트 시스템은 에이전트 조정, 평가 및 신뢰성 측면에서 새로운 과제를 제시합니다.

이 글에서는 저희가 성공적으로 구현한 원칙들을 분석합니다. 여러분이 자신만의 다중 에이전트 시스템을 구축할 때 유용하게 활용하시길 바랍니다.

### 다중 에이전트 시스템의 장점

연구 작업은 사전에 필요한 단계를 예측하기 매우 어려운 미해결 문제를 다룹니다. 복잡한 주제를 탐구하는 데 있어 고정된 경로를 미리 정해둘 수는 없습니다. 연구 과정은 본질적으로 역동적이고 경로 의존적이기 때문입니다. 사람들은 연구를 수행할 때 발견한 내용을 바탕으로 접근 방식을 지속적으로 업데이트하고, 조사 과정에서 나타나는 단서를 따라갑니다.

이러한 예측 불가능성 때문에 AI 에이전트는 연구 작업에 특히 적합합니다. 연구는 조사가 진행됨에 따라 방향을 전환하거나 관련성을 탐색할 수 있는 유연성을 요구합니다. 모델은 중간 결과를 바탕으로 어떤 방향으로 나아갈지 결정하면서 여러 단계를 자율적으로 수행해야 합니다. 선형적이고 일회성으로 진행되는 파이프라인으로는 이러한 작업을 처리할 수 없습니다.

검색의 핵심은 압축, 즉 방대한 데이터에서 통찰력을 추출하는 것입니다. 하위 에이전트는 각자 고유한 컨텍스트 창을 가지고 병렬로 작동함으로써 압축을 용이하게 합니다. 질문의 다양한 측면을 동시에 탐색한 후, 가장 중요한 정보를 리더 연구 에이전트에게 전달합니다. 또한 각 하위 에이전트는 서로 다른 도구, 프롬프트, 탐색 경로를 제공하여 관심사를 분리함으로써 경로 의존성을 줄이고 철저하고 독립적인 조사를 가능하게 합니다.

지능이 일정 임계점에 도달하면 다중 에이전트 시스템은 성능 확장에 필수적인 요소가 됩니다. 예를 들어, 지난 10만 년 동안 개개인의 지능은 꾸준히 향상되었지만, 정보화 시대에 인간 사회가 집단 지능과 협력 능력을 통해 기하급수적으로 발전해 온 것은 바로 이러한 발전 덕분입니다. 일반적인 지능을 가진 에이전트조차도 개별적으로 작동할 때는 한계에 부딪히지만, 에이전트 그룹은 훨씬 더 많은 것을 해낼 수 있습니다.

저희 내부 평가 결과에 따르면, 다중 에이전트 연구 시스템은 특히 여러 독립적인 방향을 동시에 탐색하는 너비 우선 탐색(BFS) 쿼리에서 탁월한 성능을 보입니다. 리드 에이전트로 Claude Opus 4를, 서브 에이전트로 Claude Sonnet 4를 사용한 다중 에이전트 시스템이 단일 에이전트 Claude Opus 4 시스템보다 내부 연구 평가에서 90.2% 더 높은 성능을 나타냈습니다. 예를 들어, 정보 기술 분야 S&P 500 기업의 모든 이사회 구성원을 식별하는 쿼리에서, 다중 에이전트 시스템은 작업을 여러 서브 에이전트로 나누어 수행함으로써 정답을 찾아냈지만, 단일 에이전트 시스템은 느리고 순차적인 탐색 방식으로는 정답을 찾지 못했습니다.

다중 에이전트 시스템은 문제를 해결하는 데 필요한 토큰을 적절히 사용하도록 설계되어 있어 효과적입니다. 분석 결과, [BrowseComp](https://openai.com/index/browsecomp/) 평가(찾기 어려운 정보를 검색하는 에이전트의 능력을 테스트하는 도구)에서 성능 변동의 95%를 설명하는 세 가지 요인을 발견했습니다. 토큰 사용량 자체가 변동의 80%를 설명하며, 도구 호출 횟수와 모델 선택이 나머지 두 가지 설명 요인으로 작용합니다. 이러한 결과는 각 에이전트에 서로 다른 컨텍스트 창을 할당하여 병렬 추론 용량을 늘리는 아키텍처의 타당성을 입증합니다. 최신 Claude 모델은 토큰 사용량을 크게 향상시켜 줍니다. Claude Sonnet 4로 업그레이드하는 것이 Claude Sonnet 3.7에서 토큰 예산을 두 배로 늘리는 것보다 성능 향상 효과가 더 큽니다. 다중 에이전트 아키텍처는 단일 에이전트의 한계를 넘어서는 작업에 대해 토큰 사용량을 효과적으로 확장할 수 있습니다.

단점은 이러한 아키텍처가 실제로는 토큰을 빠르게 소모한다는 것입니다. 저희 데이터에 따르면, 에이전트는 일반적으로 채팅 상호작용보다 약 4배 더 많은 토큰을 사용하고, 멀티 에이전트 시스템은 채팅보다 약 15배 더 많은 토큰을 사용합니다. 경제적 타당성을 확보하려면 멀티 에이전트 시스템은 성능 향상에 따른 비용을 상쇄할 만큼 작업 가치가 높은 작업에만 적용 가능합니다. 또한, 모든 에이전트가 동일한 컨텍스트를 공유해야 하거나 에이전트 간에 많은 의존성이 존재하는 일부 영역은 현재 멀티 에이전트 시스템에 적합하지 않습니다. 예를 들어, 대부분의 코딩 작업은 연구 작업보다 병렬 처리가 가능한 작업이 적고, LLM 에이전트는 아직 실시간으로 다른 에이전트와 협업하고 작업을 위임하는 데 능숙하지 않습니다. 저희는 멀티 에이전트 시스템이 고도의 병렬 처리가 필요하고, 단일 컨텍스트 창을 초과하는 정보가 처리되어야 하며, 여러 복잡한 도구와 연동해야 하는 가치 있는 작업에서 탁월한 성능을 발휘한다는 것을 발견했습니다.

### 연구를 위한 아키텍처 개요

저희 연구 시스템은 오케스트레이터-워커 패턴의 멀티 에이전트 아키텍처를 사용합니다. 이 패턴에서는 리드 에이전트가 프로세스를 총괄하고, 각 에이전트는 병렬로 작동하는 전문화된 서브 에이전트들에게 작업을 위임합니다.

멀티 에이전트 아키텍처의 작동 방식: 사용자 질의는 리드 에이전트를 통해 전달되고, 리드 에이전트는 다양한 정보를 병렬로 검색할 전문화된 서브 에이전트들을 생성합니다.

[](https://www.anthropic.com/_next/image?url=https%3A%2F%2Fwww-cdn.anthropic.com%2Fimages%2F4zrzovbb%2Fwebsite%2F1198befc0b33726c45692ac40f764022f4de1bf2-4584x2579.png&w=3840&q=75)

사용자가 쿼리를 제출하면, 리드 에이전트가 이를 분석하고 전략을 수립한 후, 서브 에이전트를 생성하여 다양한 측면을 동시에 탐색합니다. 위 다이어그램에서 볼 수 있듯이, 서브 에이전트는 검색 도구를 반복적으로 사용하여 정보를 수집하는 지능형 필터 역할을 합니다. 이 경우, 수집된 정보는 2025년 AI 에이전트 관련 기업에 대한 것이며, 최종 답변을 도출하기 위해 기업 목록을 리드 에이전트에 반환합니다.

기존의 검색 증강 생성(RAG) 방식은 정적 검색을 사용합니다. 즉, 입력 쿼리와 가장 유사한 데이터 조각들을 가져와 이를 기반으로 응답을 생성합니다. 반면, 본 아키텍처는 관련 정보를 동적으로 찾고, 새로운 발견에 적응하며, 결과를 분석하여 고품질 답변을 도출하는 다단계 검색 방식을 사용합니다.

위 다이어그램은 다중 에이전트 연구 시스템의 전체 워크플로를 보여줍니다. 사용자가 쿼리를 제출하면, 시스템은 반복적인 연구 프로세스에 진입하는 리드 연구원(LeadResearcher) 에이전트를 생성합니다. 리드 연구원은 먼저 접근 방식을 구상하고 계획을 메모리에 저장하여 맥락을 유지합니다. 맥락 창이 20만 토큰을 초과하면 잘리기 때문에 계획을 보존하는 것이 중요합니다. 그런 다음 특정 연구 작업을 수행하는 전문화된 하위 에이전트(여기서는 두 개가 예시이지만 개수는 상관없음)를 생성합니다. 각 하위 에이전트는 독립적으로 웹 검색을 수행하고, 확장된 사고 방식(interleaved thinking)을 사용하여 도구 결과를 평가한 후 리드 연구원에게 결과를 반환합니다. 리드 연구원은 이러한 결과를 종합하여 추가 연구가 필요한지 판단합니다. 필요한 경우 추가 하위 에이전트를 생성하거나 전략을 개선할 수 있습니다. 충분한 정보가 수집되면 시스템은 연구 루프를 종료하고 모든 결과를 인용 에이전트에 전달합니다. 인용 에이전트는 문서와 연구 보고서를 처리하여 인용할 특정 위치를 식별합니다. 이를 통해 모든 주장이 출처에 정확하게 귀속되도록 합니다. 최종 연구 결과는 참고 문헌과 함께 사용자에게 반환됩니다.

### Prompt engineering and evaluations for research agents

다중 에이전트 시스템은 단일 에이전트 시스템과 여러 가지 주요 차이점을 가지고 있는데, 그중 하나는 조정 복잡성이 급격히 증가한다는 점입니다. 초기 에이전트들은 간단한 쿼리에 50개의 하위 에이전트를 생성하거나, 존재하지 않는 소스를 찾기 위해 웹을 끝없이 검색하거나, 과도한 업데이트로 서로를 방해하는 등의 오류를 범했습니다. 각 에이전트는 프롬프트에 따라 작동하기 때문에, 이러한 문제들을 개선하는 데 있어 프롬프트 엔지니어링이 핵심적인 역할을 했습니다. 아래는 에이전트 프롬프트에 대해 우리가 얻은 몇 가지 원칙입니다.:

1. **에이전트 입장에서 생각하세요.** 프롬프트를 개선하려면 프롬프트의 효과를 이해해야 합니다. 이를 위해 저희는 [콘솔](https://console.anthropic.com/)을 사용하여 시스템에서 사용하는 것과 동일한 프롬프트와 도구로 시뮬레이션을 구축하고 에이전트의 작업 과정을 단계별로 관찰했습니다. 그 결과, 에이전트가 이미 충분한 결과를 얻었음에도 불구하고 작업을 계속하거나, 지나치게 장황한 검색 쿼리를 사용하거나, 잘못된 도구를 선택하는 등의 실패 유형을 즉시 파악할 수 있었습니다. 효과적인 프롬프트는 에이전트의 행동에 대한 정확한 정신적 모델을 구축하는 데 달려 있으며, 이를 통해 가장 효과적인 변경 사항을 명확하게 파악할 수 있습니다.
2. **지휘자에게 위임 방법을 가르치세요.** 저희 시스템에서 리드 에이전트는 쿼리를 하위 작업으로 분해하고 하위 에이전트에게 설명합니다. 각 하위 에이전트는 목표, 출력 형식, 사용할 도구 및 소스에 대한 지침, 그리고 명확한 작업 경계를 필요로 합니다. 상세한 작업 설명이 없으면 에이전트들은 작업을 중복하거나, 누락된 부분을 남기거나, 필요한 정보를 찾지 못합니다. 처음에는 리드 에이전트가 '반도체 부족 사태를 조사하세요'와 같이 간단하고 짧은 지시를 내릴 수 있도록 했지만, 이러한 지시가 너무 모호해서 하위 에이전트들이 작업을 잘못 이해하거나 다른 에이전트와 똑같은 검색을 하는 경우가 많다는 것을 발견했습니다. 예를 들어, 한 하위 에이전트는 2021년 자동차 칩 위기를 조사했고, 다른 두 에이전트는 효과적인 분업 없이 2025년 현재 공급망을 조사하는 작업을 중복했습니다.
3. **질문 복잡성에 따른 투입량 조정.** 에이전트들은 다양한 작업에 필요한 적절한 투입량을 판단하는 데 어려움을 겪기 때문에, 프롬프트에 투입량 조정 규칙을 포함시켰습니다. 간단한 사실 확인에는 에이전트 1명이 3~10회 도구 호출만으로 충분하지만, 직접 비교에는 2~4명의 하위 에이전트가 각각 10~15회 호출을 수행해야 할 수 있으며, 복잡한 조사에는 10명 이상의 하위 에이전트가 명확하게 역할을 분담하여 투입해야 할 수 있습니다. 이러한 명확한 지침은 리드 에이전트의 리소스를 효율적으로 할당하고, 초기 버전에서 흔히 발생했던 단순 질의에 대한 과도한 투입을 방지하는 데 도움이 됩니다.
4. **도구 설계 및 선택은 매우 중요합니다.** 에이전트-도구 인터페이스는 인간-컴퓨터 인터페이스만큼이나 중요합니다. 올바른 도구를 사용하는 것은 효율적일 뿐만 아니라, 종종 필수적입니다. 예를 들어, Slack에만 존재하는 컨텍스트를 웹에서 검색하는 에이전트는 처음부터 실패할 수밖에 없습니다. 모델이 외부 도구에 접근할 수 있도록 해주는 [MCP 서버](https://modelcontextprotocol.io/introduction)를 사용하면, 에이전트가 설명의 질이 매우 다양한 미지의 도구를 접하게 되므로 이 문제는 더욱 심각해집니다. 우리는 에이전트에게 명확한 휴리스틱을 부여했습니다. 예를 들어, 사용 가능한 모든 도구를 먼저 검토하거나, 도구 사용을 사용자 의도에 맞추거나, 광범위한 외부 탐색을 위해 웹을 검색하거나, 일반적인 도구보다 특화된 도구를 선호하는 등의 규칙이 있습니다. 도구 설명이 부실하면 에이전트가 완전히 잘못된 방향으로 나아갈 수 있으므로, 각 도구는 명확한 목적과 설명이 필요합니다.
5. **에이전트가 스스로 개선하도록 하세요.** 저희는 Claude 4 모델이 탁월한 프롬프트 엔지니어가 될 수 있음을 발견했습니다. 프롬프트와 오류 모드를 제공하면 에이전트가 실패하는 이유를 진단하고 개선 사항을 제안할 수 있습니다. 심지어 도구 테스트 에이전트도 개발했습니다. 결함이 있는 MCP 도구가 주어지면 해당 도구를 사용해 보고 오류를 방지하도록 도구 설명을 다시 작성합니다. 이 에이전트는 도구를 수십 번 테스트하여 핵심적인 미묘한 차이와 버그를 찾아냈습니다. 이러한 도구 사용성 개선 프로세스를 통해 새로운 설명을 사용하는 향후 에이전트의 작업 완료 시간이 40% 단축되었습니다. 대부분의 오류를 방지할 수 있었기 때문입니다.
6. **넓은 범위에서 시작하여 점차 좁혀나가세요.** 검색 전략은 전문가의 인적 조사 방식을 따라야 합니다. 구체적인 사항에 집중하기 전에 전체적인 상황을 먼저 살펴보세요. 상담원들은 종종 결과가 거의 나오지 않는 지나치게 길고 구체적인 검색어를 사용하는 경향이 있습니다. 우리는 상담원들이 짧고 광범위한 검색어로 시작하여 이용 가능한 정보를 평가한 다음, 점차 범위를 좁혀나가도록 유도함으로써 이러한 경향을 해결했습니다.
7. **사고 과정을 안내하세요.** [확장된 사고 모드](https://docs.anthropic.com/en/docs/build-with-claude/extended-thinking)는 클로드(Claude)가 가시적인 사고 과정에서 추가 토큰을 출력하도록 유도하며, 제어 가능한 스크래치패드 역할을 합니다. 리드 에이전트는 사고를 통해 접근 방식을 계획하고, 작업에 적합한 도구를 평가하고, 쿼리 복잡성과 하위 에이전트 수를 결정하고, 각 하위 에이전트의 역할을 정의합니다. 테스트 결과, 확장된 사고 모드는 지시 준수, 추론 및 효율성을 향상시키는 것으로 나타났습니다. 하위 에이전트 또한 계획을 세운 후, 도구 결과가 나온 뒤 [중첩된 사고](https://docs.anthropic.com/en/docs/build-with-claude/extended-thinking#interleaved-thinking)를 사용하여 품질을 평가하고, 부족한 부분을 파악하고, 다음 쿼리를 개선합니다. 이를 통해 하위 에이전트는 어떤 작업에도 더욱 효과적으로 적응할 수 있습니다.
8. **병렬 도구 호출로 속도와 성능 혁신** 복잡한 연구 작업은 필연적으로 여러 소스를 탐색해야 합니다. 초기 에이전트는 순차 검색을 수행했는데, 이는 매우 느렸습니다. 속도 향상을 위해 두 가지 유형의 병렬화를 도입했습니다. (1) 리드 에이전트가 3~5개의 서브 에이전트를 직렬이 아닌 병렬로 실행하고, (2) 서브 에이전트가 3개 이상의 도구를 병렬로 사용합니다. 이러한 변경으로 복잡한 쿼리에 대한 연구 시간이 최대 90%까지 단축되어, 기존 시스템보다 더 많은 정보를 다루면서도 몇 시간씩 걸리던 작업을 단 몇 분 만에 완료할 수 있게 되었습니다.

우리의 프롬프트 전략은 엄격한 규칙보다는 좋은 휴리스틱을 심어주는 데 중점을 두었습니다. 숙련된 사람들이 연구 과제에 접근하는 방식을 연구하고, 어려운 질문을 더 작은 과제로 분해하거나, 자료의 질을 신중하게 평가하거나, 새로운 정보에 따라 검색 방식을 조정하거나, 깊이 있는 조사(하나의 주제를 자세히 탐구)와 폭넓은 조사(여러 주제를 동시에 탐색) 중 어느 쪽에 집중해야 할지 판단하는 등의 전략을 프롬프트에 반영했습니다. 또한, 에이전트가 통제 불능 상태에 빠지는 것을 방지하기 위해 명시적인 안전장치를 설정하여 의도치 않은 부작용을 사전에 완화했습니다. 마지막으로, 관찰 가능성과 테스트 케이스를 갖춘 빠른 반복 루프에 집중했습니다.

### 효과적인 에이전트 평가

신뢰할 수 있는 AI 애플리케이션을 구축하려면 정확한 평가가 필수적이며, 에이전트 시스템도 예외는 아닙니다. 하지만 다중 에이전트 시스템을 평가하는 것은 고유한 어려움을 수반합니다. 기존의 평가 방식은 AI가 매번 동일한 단계를 따른다고 가정하는 경우가 많습니다. 즉, 입력 X가 주어지면 시스템은 경로 Y를 따라 출력 Z를 생성해야 한다는 것입니다. 그러나 다중 에이전트 시스템은 그렇지 않습니다. 동일한 시작점을 갖더라도 에이전트들은 목표에 도달하기 위해 완전히 다른 유효한 경로를 택할 수 있습니다. 한 에이전트는 세 개의 소스를 탐색하는 반면 다른 에이전트는 열 개의 소스를 탐색할 수도 있고, 동일한 답을 찾기 위해 서로 다른 도구를 사용할 수도 있습니다. 어떤 단계가 올바른지 항상 알 수 없기 때문에, 에이전트가 사전에 규정한 "올바른" 단계를 따랐는지 여부만 확인하는 방식으로는 충분하지 않습니다. 따라서 에이전트가 합리적인 프로세스를 따르면서도 올바른 결과를 달성했는지 판단할 수 있는 유연한 평가 방법이 필요합니다.

**소규모 샘플로 즉시 평가를 시작하세요.** 초기 에이전트 개발 단계에서는 손쉽게 개선할 수 있는 부분이 많기 때문에 변경 사항이 큰 영향을 미치는 경향이 있습니다. 신속한 수정으로 성공률을 30%에서 80%까지 끌어올릴 수 있습니다. 이처럼 효과 크기가 큰 경우, 몇 개의 테스트 케이스만으로도 변경 사항을 파악할 수 있습니다. 저희는 실제 사용 패턴을 나타내는 약 20개의 쿼리 세트로 시작했습니다. 이러한 쿼리를 테스트할 때마다 변경 사항의 영향을 명확하게 확인할 수 있었습니다. AI 개발팀이 수백 개의 테스트 케이스를 포함하는 대규모 평가만이 유용하다고 생각하여 평가를 미루는 경우가 종종 있습니다. 하지만 더 철저한 평가를 구축할 때까지 미루기보다는 몇 가지 예제를 사용하여 소규모 테스트를 즉시 시작하는 것이 좋습니다.

**LLM을 활용한 평가 척도는 제대로 구현되었을 때 효과적입니다.** 연구 결과물은 자유 형식의 텍스트이며 정답이 하나만 있는 경우가 드물기 때문에 프로그램적으로 평가하기 어렵습니다. LLM은 이러한 결과물을 평가하는 데 매우 적합합니다. 저희는 LLM 평가 도구를 사용하여 각 결과물을 평가 기준표에 따라 평가했습니다. 평가 기준에는 사실 정확성(주장이 출처와 일치하는가?), 인용 정확성(인용된 출처가 주장과 일치하는가?), 완성도(요청된 모든 측면이 다루어졌는가?), 출처 품질(품질이 낮은 2차 자료보다 1차 자료를 사용했는가?), 도구 효율성(적절한 도구를 적절한 횟수로 사용했는가?)이 포함됩니다. 각 구성 요소를 평가하기 위해 여러 평가 도구를 사용해 보았지만, 단일 프롬프트를 사용하여 0.0~1.0점 범위의 점수와 합격/불합격 등급을 출력하는 단일 LLM 호출 방식이 가장 일관성이 높고 사람의 판단과 일치하는 것으로 나타났습니다. 이 방법은 평가 테스트 케이스에 명확한 답이 있을 때 특히 효과적이었으며, LLM 평가 도구를 사용하여 답이 정확한지 여부만 확인할 수 있었습니다(예: R&D 예산이 가장 큰 상위 3개 제약 회사를 정확하게 나열했는지 여부). LLM을 평가 도구로 사용함으로써 수백 개의 결과물을 대규모로 평가할 수 있었습니다.

**인간 평가는 자동화 시스템이 놓치는 부분을 잡아냅니다.** 에이전트를 테스트하는 사람은 평가 과정에서 놓치는 예외적인 상황들을 발견합니다. 여기에는 특이한 쿼리에 대한 잘못된 답변, 시스템 오류, 미묘한 소스 선택 편향 등이 포함됩니다. 저희의 경우, 초기 에이전트들이 학술 PDF나 개인 블로그처럼 권위는 있지만 검색 순위가 낮은 소스보다 SEO에 최적화된 콘텐츠를 일관되게 선택하는 것을 인간 테스터들이 발견했습니다. 프롬프트에 소스 품질 휴리스틱을 추가함으로써 이 문제를 해결할 수 있었습니다. 자동화된 평가가 보편화된 시대에도 수동 테스트는 여전히 필수적입니다.

다중 에이전트 시스템은 특정 프로그래밍 없이도 발생하는 새로운 행동 양식을 보입니다. 예를 들어, 리드 에이전트에 대한 작은 변경 사항이 하위 에이전트의 행동에 예측할 수 없는 변화를 가져올 수 있습니다. 성공적인 시스템 운영을 위해서는 개별 에이전트의 행동뿐 아니라 상호작용 패턴을 이해하는 것이 중요합니다. 따라서 이러한 에이전트에 대한 최적의 프롬프트는 엄격한 지침이 아니라, 업무 분담, 문제 해결 접근 방식, 투입 노력 등을 정의하는 협업 프레임워크입니다. 이를 위해서는 신중한 프롬프트 및 도구 설계, 탄탄한 휴리스틱, 관찰 가능성, 그리고 긴밀한 피드백 루프가 필수적입니다. ****저희 시스템의 프롬프트 예시는 [저희 쿡북의 오픈 소스 프롬프트](https://platform.claude.com/cookbook/patterns-agents-basic-workflows)를 참조하세요.

### 생산 신뢰성 및 엔지니어링 과제

기존 소프트웨어 개발 방식에서는 버그가 기능 장애, 성능 저하 또는 서비스 중단을 초래할 수 있습니다. 에이전트 기반 시스템에서는 사소한 변경 사항이 큰 동작 변화로 이어지기 때문에, 장시간 실행되는 프로세스에서 상태를 유지해야 하는 복잡한 에이전트를 위한 코드를 작성하는 것이 매우 어렵습니다.

에이전트는 상태를 유지하며, 오류는 누적됩니다. 에이전트는 여러 도구 호출에 걸쳐 상태를 유지하면서 장시간 실행될 수 있습니다. 따라서 코드를 안정적으로 실행하고 오류를 처리해야 합니다. 효과적인 완화 조치가 없다면, 사소한 시스템 오류가 에이전트에 치명적인 결과를 초래할 수 있습니다. 오류 발생 시 처음부터 다시 시작하는 것은 비용이 많이 들고 사용자에게 불편함을 초래합니다. 따라서 오류 발생 시점부터 에이전트의 실행을 재개할 수 있는 시스템을 구축했습니다. 또한 모델의 지능을 활용하여 문제를 원활하게 처리합니다. 예를 들어, 도구 사용 중 오류가 발생했을 때 에이전트에게 이를 알리고 적응하도록 하는 방식이 놀라울 정도로 효과적입니다. Claude 기반 AI 에이전트의 적응성과 재시도 로직 및 정기적인 체크포인트와 같은 결정론적 안전장치를 결합했습니다.

**새로운 접근 방식을 통해 디버깅이 향상되었습니다.** 에이전트는 동적인 결정을 내리고, 동일한 프롬프트가 주어지더라도 실행 간에 비결정적인 동작을 보입니다. 이 때문에 디버깅이 더욱 어려워집니다. 예를 들어, 사용자들이 에이전트가 "명백한 정보를 찾지 못한다"고 보고했지만, 그 원인을 파악할 수 없었습니다. 에이전트가 잘못된 검색 쿼리를 사용했는지, 부적절한 소스를 선택했는지, 도구 오류가 발생했는지 알 수 없었습니다. 전체 프로덕션 추적 기능을 추가함으로써 에이전트 실패 원인을 진단하고 체계적으로 문제를 해결할 수 있었습니다. 표준적인 관찰 가능성을 넘어, 에이전트의 결정 패턴과 상호 작용 구조를 모니터링했습니다. 사용자 개인 정보 보호를 위해 개별 대화 내용은 모니터링하지 않았습니다. 이러한 고수준 관찰 가능성을 통해 근본 원인을 진단하고, 예상치 못한 동작을 발견하고, 일반적인 오류를 수정할 수 있었습니다.

**배포에는 신중한 조정이 필요합니다.** 에이전트 시스템은 프롬프트, 도구, 실행 로직이 복잡하게 얽혀 거의 지속적으로 실행되는 상태 저장 구조입니다. 즉, 업데이트를 배포할 때마다 에이전트는 프로세스의 어느 단계에 있든 관계없이 실행될 수 있습니다. 따라서 의도한 코드 변경으로 인해 기존 에이전트가 손상되는 것을 방지해야 합니다. 모든 에이전트를 동시에 새 버전으로 업데이트할 수는 없습니다. 대신, 실행 중인 에이전트의 작동을 중단하지 않고 기존 버전에서 새 버전으로 트래픽을 점진적으로 전환하면서 두 버전을 동시에 실행하는 [레인보우 배포](https://brandon.dimcheff.com/2018/02/rainbow-deploys-with-kubernetes/)를 사용합니다.

**동기 실행은 병목 현상을 초래합니다.** 현재, 저희의 리드 에이전트는 서브 에이전트를 동기적으로 실행하며, 각 서브 에이전트 세트가 완료될 때까지 기다린 후 다음 단계로 진행합니다. 이는 조정은 간소화하지만 에이전트 간 정보 흐름에 병목 현상을 일으킵니다. 예를 들어, 리드 에이전트는 서브 에이전트를 제어할 수 없고, 서브 에이전트들은 서로 협력할 수 없으며, 단일 서브 에이전트의 검색 완료를 기다리는 동안 전체 시스템이 차단될 수 있습니다. 비동기 실행은 에이전트들이 동시에 작업하고 필요에 따라 새로운 서브 에이전트를 생성하는 등 병렬 처리를 가능하게 합니다. 하지만 이러한 비동기성은 결과 조정, 상태 일관성 및 서브 에이전트 간 오류 전파에 어려움을 더합니다. 모델이 더 길고 복잡한 연구 작업을 처리할 수 있게 됨에 따라, 성능 향상이 이러한 복잡성을 정당화할 것으로 기대합니다.

### Conclusion

AI 에이전트를 구축할 때, 마지막 단계는 여정의 대부분을 차지하는 경우가 많습니다. 개발자 환경에서 작동하는 코드베이스도 안정적인 프로덕션 시스템으로 전환하려면 상당한 엔지니어링 작업이 필요합니다. 에이전트 시스템에서 오류는 복합적으로 발생하기 때문에, 기존 소프트웨어에서는 사소한 문제였던 것이 에이전트 시스템에서는 완전히 오작동을 일으킬 수 있습니다. 한 단계의 오류가 에이전트를 완전히 다른 경로로 이끌고 예측 불가능한 결과를 초래할 수도 있습니다. 이 글에서 설명한 모든 이유로 인해 프로토타입과 프로덕션 시스템 간의 격차는 예상보다 훨씬 큰 경우가 많습니다.

이러한 어려움에도 불구하고, 다중 에이전트 시스템은 개방형 연구 과제에 매우 유용하다는 것이 입증되었습니다. 사용자들은 Claude 덕분에 미처 생각하지 못했던 비즈니스 기회를 발견하고, 복잡한 의료 옵션을 탐색하고, 까다로운 기술적 버그를 해결하고, 혼자서는 찾을 수 없었을 연구 연결 고리를 발견하여 작업 시간을 며칠씩 절약할 수 있었다고 말합니다. 다중 에이전트 연구 시스템은 세심한 엔지니어링, 철저한 테스트, 세부적인 프롬프트 및 도구 설계, 견고한 운영 방식, 그리고 현재 에이전트 기능에 대한 깊은 이해를 가진 연구, 제품 및 엔지니어링 팀 간의 긴밀한 협업을 통해 대규모 환경에서도 안정적으로 운영될 수 있습니다. 우리는 이미 이러한 시스템이 사람들이 복잡한 문제를 해결하는 방식을 혁신하는 것을 목격하고 있습니다.

## 부록

다음은 다중 에이전트 시스템에 대한 몇 가지 추가적인 유용한 팁입니다.

**여러 턴에 걸쳐 상태를 변경하는 에이전트의 최종 상태 평가** 여러 턴에 걸친 대화에서 지속적인 상태를 수정하는 에이전트를 평가하는 것은 고유한 어려움을 수반합니다. 읽기 전용 연구 작업과 달리 각 행동은 후속 단계의 환경을 변경할 수 있으므로 기존 평가 방법으로는 처리하기 어려운 종속성이 발생합니다. 저희는 턴별 분석보다는 최종 상태 평가에 집중하는 것이 효과적이라는 것을 발견했습니다. 에이전트가 특정 프로세스를 따랐는지 여부를 판단하는 대신, 올바른 최종 상태에 도달했는지 여부를 평가합니다. 이 접근 방식은 에이전트가 의도한 결과를 제공하면서도 동일한 목표에 도달하는 여러 경로를 찾을 수 있음을 인정합니다. 복잡한 워크플로의 경우, 모든 중간 단계를 검증하려고 시도하기보다는 특정 상태 변경이 발생해야 하는 개별적인 체크포인트로 평가를 나누는 것이 좋습니다.

**장기적인 대화 관리** 실제 운영 환경에서 에이전트는 종종 수백 턴에 걸친 대화에 참여하므로 신중한 컨텍스트 관리 전략이 필요합니다. 대화가 길어짐에 따라 표준 컨텍스트 창으로는 충분하지 않으므로 지능적인 압축 및 메모리 메커니즘이 필요합니다. 에이전트가 완료된 작업 단계를 요약하고 새로운 작업으로 진행하기 전에 필수 정보를 외부 메모리에 저장하는 패턴을 구현했습니다. 컨텍스트 제한에 가까워지면 에이전트는 신중한 인수인계를 통해 연속성을 유지하면서 깨끗한 컨텍스트를 가진 새로운 하위 에이전트를 생성할 수 있습니다. 또한 컨텍스트 제한에 도달했을 때 이전 작업을 손실하는 대신 연구 계획과 같은 저장된 컨텍스트를 메모리에서 불러올 수 있습니다. 이러한 분산 접근 방식은 컨텍스트 오버플로를 방지하는 동시에 장시간 상호 작용에 걸쳐 대화의 일관성을 유지합니다.

**하위 에이전트 출력을 파일 시스템에 저장하여 '전화 게임'과 같은 정보 전달 문제를 최소화합니다.** 특정 유형의 결과에 대해서는 하위 에이전트가 메인 코디네이터를 거치지 않고 직접 출력할 수 있어 정확도와 성능을 모두 향상시킬 수 있습니다. 하위 에이전트가 모든 것을 리드 에이전트를 통해 전달하도록 요구하는 대신, 전문 에이전트가 독립적으로 저장되는 출력을 생성할 수 있는 아티팩트 시스템을 구현합니다. 하위 에이전트는 도구를 호출하여 작업을 외부 시스템에 저장한 다음, 가벼운 참조를 코디네이터에게 전달합니다. 이를 통해 다단계 처리 중 정보 손실을 방지하고 대화 기록을 통해 대용량 출력을 복사하는 데 필요한 토큰 오버헤드를 줄일 수 있습니다. 이 패턴은 코드, 보고서 또는 데이터 시각화와 같은 구조화된 출력물에 특히 효과적이며, 하위 에이전트의 특수 프롬프트가 일반 코디네이터를 통해 필터링하는 것보다 더 나은 결과를 생성합니다.
