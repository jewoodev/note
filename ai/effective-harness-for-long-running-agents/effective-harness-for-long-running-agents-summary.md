# 근본 문제: 컨텍스트 윈도우의 한계
Claude는 세션이 바뀌면 이전 기억이 없습니다. 교대 근무하는 개발자가 인수인계 없이 출근하는 것과 같은 상황이죠. 단순히 컨텍스트 압축(compaction)만으로는 해결이 안 되고, 구조적인 장치가 필요합니다.

# 가이드의 핵심: 2단계 에이전트 구조
## 1단계 — 초기화 에이전트 (첫 세션에서만 실행)
첫 세션에서는 코딩을 바로 시작하지 말고, 미래의 에이전트가 작업할 수 있는 환경을 먼저 세팅하라고 합니다. 구체적으로는 사용자의 고수준 프롬프트를 200개 이상의 세부 기능으로 분해한 JSON 기능 목록, 개발 서버를 띄우는 init.sh 스크립트, 진행 상황을 기록할 claude-progress.txt, 그리고 초기 git 커밋을 만드는 것입니다.

## 2단계 — 코딩 에이전트 (이후 모든 세션)
매 세션마다 정해진 루틴을 따릅니다. pwd로 작업 디렉토리 확인 → git 로그와 진행 파일로 이전 작업 파악 → 개발 서버 띄우고 기본 기능 동작 검증 → 기능 목록에서 미완료 항목 하나만 선택해서 구현 → 브라우저 자동화(Puppeteer)로 엔드투엔드 테스트 → git 커밋 + 진행 파일 업데이트 후 세션 종료.

## 글에서 강조하는 실천 원칙들
**한 번에 하나의 기능만 작업하라.** Claude에게 자유를 주면 한 번에 전체를 완성하려다 중간에 컨텍스트가 소진되고, 다음 세션에서 반쯤 구현된 코드 위에서 혼란이 생깁니다.

**세션 종료 시 항상 "머지 가능한 상태"를 유지하라.** 설명적인 커밋 메시지로 git에 커밋하고, 진행 파일에 요약을 남겨서 다음 에이전트가 즉시 상황을 파악할 수 있게 합니다.

**기능 목록은 JSON으로 관리하라.** Markdown보다 JSON이 모델이 임의로 내용을 수정하거나 덮어쓸 가능성이 낮습니다. 또한 passes 필드만 변경하도록 강하게 제한을 걸어야 합니다.

**반드시 엔드투엔드 테스트를 하라.** 단위 테스트나 curl만으로는 실제 동작 여부를 확인하기 어렵습니다. Puppeteer 같은 브라우저 자동화 도구로 실제 사용자처럼 테스트해야 합니다.

**새 기능 전에 기존 기능 검증부터 하라.** 앱이 깨진 상태에서 새 기능을 추가하면 문제가 더 악화됩니다.
