# 장기 실행 에이전트를 위한 효과적인 하네스

> 출처: [Effective harnesses for long-running agents](https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents)

> 에이전트는 여전히 여러 컨텍스트 윈도우에 걸쳐 작업할 때 어려움을 겪고 있습니다. 우리는 장기 실행 에이전트를 위한 더 효과적인 하네스를 만들기 위해 인간 엔지니어에게서 영감을 얻었습니다.

AI 에이전트의 능력이 향상됨에 따라, 개발자들은 에이전트에게 몇 시간 또는 며칠에 걸친 복잡한 작업을 점점 더 많이 맡기고 있습니다. 그러나 여러 컨텍스트 윈도우에 걸쳐 에이전트가 일관되게 진전을 이루도록 하는 것은 여전히 미해결 과제입니다.

장기 실행 에이전트의 핵심 문제는, 에이전트가 불연속적인 세션 단위로 작업해야 하며, 각 새 세션은 이전에 무슨 일이 있었는지에 대한 기억 없이 시작된다는 것입니다. 교대 근무를 하는 엔지니어로 구성된 소프트웨어 프로젝트를 상상해 보세요. 새로운 엔지니어가 도착할 때마다 이전 교대에서 무슨 일이 있었는지 전혀 기억하지 못하는 상황입니다. 컨텍스트 윈도우가 제한되어 있고 대부분의 복잡한 프로젝트가 단일 윈도우 내에서 완료될 수 없기 때문에, 에이전트에게는 코딩 세션 간의 간극을 메우는 방법이 필요합니다.

우리는 [Claude Agent SDK](https://platform.claude.com/docs/en/agent-sdk/overview)가 여러 컨텍스트 윈도우에 걸쳐 효과적으로 작동할 수 있도록 두 가지 해결책을 개발했습니다: 첫 실행 시 환경을 설정하는 **초기화 에이전트(initializer agent)**, 그리고 매 세션마다 점진적 진전을 이루면서 다음 세션을 위한 명확한 산출물을 남기는 **코딩 에이전트(coding agent)** 입니다. 함께 제공되는 [퀵스타트](https://github.com/anthropics/claude-quickstarts/tree/main/autonomous-coding)에서 코드 예제를 확인할 수 있습니다.

## 장기 실행 에이전트 문제

Claude Agent SDK는 코딩뿐 아니라 모델이 도구를 사용해 맥락을 수집하고, 계획을 세우고, 실행해야 하는 다양한 작업에 능숙한 범용 에이전트 하네스입니다. 이 SDK에는 압축(compaction)과 같은 컨텍스트 관리 기능이 있어, 에이전트가 컨텍스트 윈도우를 소진하지 않으면서 작업을 수행할 수 있습니다. 이론적으로 이 구성이 갖춰지면, 에이전트가 임의로 긴 시간 동안 유용한 작업을 계속하는 것이 가능해야 합니다.

그러나 압축만으로는 충분하지 않습니다. 기본 설정 그대로, Opus 4.5 같은 최첨단 코딩 모델이 Claude Agent SDK에서 여러 컨텍스트 윈도우를 반복 실행하더라도, "claude.ai의 클론을 만들어라" 같은 고수준 프롬프트만 주어진 경우 프로덕션 품질의 웹 앱을 구축하지 못합니다.

Claude의 실패는 두 가지 패턴으로 나타났습니다. 첫째, 에이전트가 한 번에 너무 많은 것을 하려는 경향이 있었습니다 — 본질적으로 앱을 한 번에 완성하려는 시도였습니다. 이로 인해 구현 도중 컨텍스트가 소진되어, 다음 세션이 반쯤 구현되고 문서화되지 않은 기능 위에서 시작되는 일이 많았습니다. 에이전트는 무슨 일이 있었는지 추측해야 했고, 기본 앱을 다시 동작시키는 데 상당한 시간을 소비해야 했습니다. 이 현상은 압축을 사용해도 발생하는데, 압축이 다음 에이전트에게 항상 완벽하게 명확한 지시를 전달하는 것은 아니기 때문입니다.

두 번째 실패 모드는 프로젝트 후반부에서 자주 발생했습니다. 일부 기능이 이미 구축된 후, 이후의 에이전트 인스턴스가 주변을 둘러보고 진전이 있었음을 확인한 뒤, 작업이 완료되었다고 선언해 버리는 것이었습니다.

이를 통해 문제가 두 부분으로 분해됩니다. 첫째, 주어진 프롬프트가 요구하는 *모든* 기능의 기반을 마련하는 초기 환경을 설정하여, 에이전트가 단계별로, 기능별로 작업하도록 유도해야 합니다. 둘째, 각 에이전트가 목표를 향해 점진적으로 진전을 이루면서도 세션 종료 시 환경을 깨끗한 상태로 남기도록 프롬프트해야 합니다. "깨끗한 상태"란 메인 브랜치에 머지하기 적합한 수준의 코드를 의미합니다: 주요 버그가 없고, 코드가 정돈되어 있으며 잘 문서화되어 있고, 일반적으로 개발자가 관련 없는 혼란을 먼저 정리하지 않고도 새 기능 작업을 쉽게 시작할 수 있는 상태입니다.

내부 실험에서 우리는 두 부분으로 구성된 해결책을 사용하여 이 문제들을 해결했습니다:

1. **초기화 에이전트**: 첫 번째 에이전트 세션은 전문화된 프롬프트를 사용하여 초기 환경을 설정합니다: `init.sh` 스크립트, 에이전트들이 수행한 작업 로그를 유지하는 claude-progress.txt 파일, 그리고 어떤 파일이 추가되었는지 보여주는 초기 git 커밋을 생성합니다.
2. **코딩 에이전트**: 이후 모든 세션에서는 모델에게 점진적 진전을 이루고 구조화된 업데이트를 남기도록 요청합니다.¹

여기서의 핵심 인사이트는, 새로운 컨텍스트 윈도우에서 시작할 때 에이전트가 작업 상태를 빠르게 파악할 수 있는 방법을 찾는 것이었습니다. 이는 claude-progress.txt 파일과 git 히스토리를 통해 달성됩니다. 이러한 관행들의 영감은 효과적인 소프트웨어 엔지니어들이 매일 하는 일에서 비롯되었습니다.

## 환경 관리

업데이트된 [Claude 4 프롬프팅 가이드](https://docs.claude.com/en/docs/build-with-claude/prompt-engineering/claude-4-best-practices#multi-context-window-workflows)에서 우리는 "첫 번째 컨텍스트 윈도우에 대해 다른 프롬프트"를 사용하는 하네스 구조를 포함하여 다중 컨텍스트 윈도우 워크플로우에 대한 모범 사례를 공유했습니다. 이 "다른 프롬프트"는 초기화 에이전트에게 미래의 코딩 에이전트가 효과적으로 작업하는 데 필요한 모든 컨텍스트를 갖춘 환경을 설정하도록 요청합니다. 여기서는 이러한 환경의 핵심 구성 요소에 대해 더 깊이 살펴봅니다.

### 기능 목록

에이전트가 앱을 한 번에 완성하려 하거나 프로젝트가 완료되었다고 조기에 판단하는 문제를 해결하기 위해, 우리는 초기화 에이전트에게 사용자의 초기 프롬프트를 확장하여 기능 요구사항의 포괄적인 파일을 작성하도록 프롬프트했습니다. claude.ai 클론 예제의 경우, "사용자가 새 채팅을 열고, 쿼리를 입력하고, 엔터를 누르고, AI 응답을 볼 수 있다"와 같은 200개 이상의 기능이 포함되었습니다. 이 기능들은 모두 초기에 "실패(failing)"로 표시되어, 이후의 코딩 에이전트가 전체 기능의 완성된 모습이 어떤 것인지 명확한 윤곽을 갖게 했습니다.

```json
{
    "category": "functional",
    "description": "New chat button creates a fresh conversation",
    "steps": [
      "Navigate to main interface",
      "Click the 'New Chat' button",
      "Verify a new conversation is created",
      "Check that chat area shows welcome state",
      "Verify conversation appears in sidebar"
    ],
    "passes": false
}
```

우리는 코딩 에이전트에게 이 파일을 수정할 때 passes 필드의 상태만 변경하도록 프롬프트하고, "테스트를 제거하거나 편집하는 것은 기능 누락이나 버그로 이어질 수 있으므로 용납할 수 없다"와 같은 강력한 표현의 지시를 사용합니다. 실험을 거쳐, 이 파일 형식으로 JSON을 선택했는데, 모델이 Markdown 파일에 비해 JSON 파일을 부적절하게 변경하거나 덮어쓸 가능성이 낮기 때문입니다.

### 점진적 진전

이러한 초기 환경 스캐폴딩이 갖춰진 상태에서, 코딩 에이전트의 다음 반복에서는 한 번에 하나의 기능만 작업하도록 요청했습니다. 이 점진적 접근 방식은 에이전트가 한 번에 너무 많은 것을 하려는 경향을 해결하는 데 결정적이었습니다.

점진적으로 작업하더라도, 코드 변경 후 모델이 환경을 깨끗한 상태로 남기는 것이 여전히 필수적입니다. 실험에서 이 행동을 이끌어내는 가장 좋은 방법은, 모델에게 설명적인 커밋 메시지와 함께 git에 진행 상황을 커밋하고, 진행 파일에 작업 요약을 작성하도록 요청하는 것이었습니다. 이를 통해 모델은 git을 사용하여 잘못된 코드 변경을 되돌리고 코드베이스의 동작하는 상태를 복구할 수 있었습니다.

이러한 접근 방식은 효율성도 높였는데, 에이전트가 이전에 무슨 일이 있었는지 추측하고 기본 앱을 다시 동작시키는 데 시간을 소비할 필요가 없어졌기 때문입니다.

### 테스트

우리가 관찰한 마지막 주요 실패 모드는 Claude가 적절한 테스트 없이 기능을 완료로 표시하는 경향이었습니다. 명시적인 프롬프트 없이는, Claude가 코드를 변경하고 단위 테스트나 개발 서버에 대한 `curl` 명령으로 테스트하기는 했지만, 기능이 엔드투엔드로 동작하지 않는다는 것을 인식하지 못하는 경우가 있었습니다.

웹 앱을 구축하는 경우, 브라우저 자동화 도구를 사용하고 인간 사용자가 하듯이 모든 테스트를 수행하도록 명시적으로 프롬프트하면, Claude는 기능을 엔드투엔드로 검증하는 데 대체로 잘 수행했습니다.

Claude에게 이러한 종류의 테스트 도구를 제공하면 성능이 극적으로 향상되었습니다. 에이전트가 코드만으로는 명확하지 않은 버그를 식별하고 수정할 수 있었기 때문입니다.

일부 문제는 여전히 남아 있습니다. Claude의 비전 한계와 브라우저 자동화 도구의 제약으로 인해 모든 종류의 버그를 식별하기 어려운 경우가 있습니다. 예를 들어, Claude는 Puppeteer MCP를 통해 브라우저 네이티브 알림 모달을 볼 수 없으며, 이러한 모달에 의존하는 기능은 결과적으로 더 버그가 많은 경향이 있었습니다.

## 작업 상태 파악하기

위의 모든 것이 갖춰진 상태에서, 모든 코딩 에이전트는 상황을 파악하기 위해 일련의 단계를 거치도록 프롬프트됩니다. 일부는 매우 기본적이지만 여전히 도움이 됩니다:

1. *`pwd`를 실행하여 작업 중인 디렉토리를 확인하세요. 이 디렉토리 내의 파일만 수정할 수 있습니다.*
2. *git 로그와 진행 파일을 읽어 최근 작업 내용을 파악하세요.*
3. *기능 목록 파일을 읽고 아직 완료되지 않은 가장 높은 우선순위의 기능을 선택하여 작업하세요.*

이 접근 방식은 모든 세션에서 Claude가 코드를 테스트하는 방법을 스스로 파악하지 않아도 되므로 토큰을 절약합니다. 또한 초기화 에이전트에게 개발 서버를 실행할 수 있는 init.sh 스크립트를 작성하도록 요청하고, 새 기능을 구현하기 전에 기본적인 엔드투엔드 테스트를 실행하는 것도 도움이 됩니다.

claude.ai 클론의 경우, 에이전트가 항상 로컬 개발 서버를 시작하고 Puppeteer MCP를 사용하여 새 채팅을 시작하고 메시지를 보내고 응답을 받는다는 것을 의미했습니다. 이를 통해 Claude는 앱이 제대로 작동하지 않는 상태로 남아 있는지 신속하게 파악하고 기존 버그를 즉시 수정할 수 있었습니다. 만약 에이전트가 새로운 기능을 구현하기 시작했다면 문제가 더 악화되었을 가능성이 높습니다.

이 모든 것을 감안하면, 일반적인 세션은 다음과 같은 어시스턴트 메시지로 시작됩니다:

```
[어시스턴트] 현재 프로젝트의 상태를 파악하는 것부터 시작하겠습니다.
[도구 사용] <bash - pwd>
[도구 사용] <read - claude-progress.txt>
[도구 사용] <read - feature_list.json>
[어시스턴트] git 로그를 확인하여 최근 작업을 살펴보겠습니다.
[도구 사용] <bash - git log --oneline -20>
[어시스턴트] 이제 서버를 재시작하는 init.sh 스크립트가 있는지 확인하겠습니다.
<개발 서버 시작>
[어시스턴트] 좋습니다! 이제 애플리케이션으로 이동하여 기본 기능이 여전히 동작하는지 확인하겠습니다.
<기본 기능 테스트>
[어시스턴트] 검증 테스트를 바탕으로, 기본 기능이 잘 동작하고 있음을 확인했습니다.
핵심 채팅 기능, 테마 전환, 대화 로딩, 오류 처리가 모두 올바르게 작동합니다.
이제 tests.json 파일을 더 종합적으로 검토하여 다음에 무엇을 구현해야 하는지 파악하겠습니다.
<새 기능 작업 시작>
```

## 향후 작업

이 연구는 장기 실행 에이전트 하네스에서 모델이 여러 컨텍스트 윈도우에 걸쳐 점진적 진전을 이룰 수 있게 하는 하나의 가능한 해결책 세트를 보여줍니다. 그러나 여전히 열린 질문들이 남아 있습니다.

가장 주목할 만한 것은, 단일 범용 코딩 에이전트가 여러 컨텍스트에 걸쳐 최상의 성능을 보이는지, 아니면 멀티 에이전트 아키텍처를 통해 더 나은 성능을 달성할 수 있는지가 아직 불분명하다는 것입니다. 테스트 에이전트, QA 에이전트, 코드 정리 에이전트 같은 전문화된 에이전트가 소프트웨어 개발 라이프사이클 전반의 하위 작업에서 더 나은 성과를 낼 수 있다는 것은 합리적으로 보입니다.

또한 이 데모는 풀스택 웹 앱 개발에 최적화되어 있습니다. 향후 방향은 이러한 발견을 다른 분야로 일반화하는 것입니다. 이러한 교훈의 일부 또는 전부가 과학 연구나 금융 모델링 같은 장기 실행 에이전트 작업에 적용될 수 있을 것입니다.
