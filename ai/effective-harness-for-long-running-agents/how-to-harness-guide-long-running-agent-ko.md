# 장기 실행 AI 에이전트 하네스: 구현 가이드

> 출처: [Effective harnesses for long-running agents](https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents)

## 이 문서의 목적

이 문서는 AI 코딩 에이전트가 하나의 프로젝트에서 여러 세션에 걸쳐 효과적으로 작업할 수 있도록 하는 하네스를 구축하기 위한 독립형 구현 가이드입니다. 이 가이드를 직접 따르면 됩니다 — 외부 참조 자료는 필요하지 않습니다.

---

## 해결하려는 핵심 제약

AI 에이전트는 고정 크기의 컨텍스트 윈도우 내에서 작동합니다. 해당 윈도우가 가득 차면 세션이 종료되고, 이전 세션에 대한 **기억이 전혀 없는** 상태로 새로운 세션이 시작됩니다. 구조화된 접근 방식이 없으면 다음과 같은 문제가 발생합니다:

- 에이전트가 한 세션에서 모든 것을 구축하려다 작업 도중 컨텍스트가 소진되어, 깨지고 문서화되지 않은 코드를 남깁니다.
- 에이전트가 이전 세션에서 부분적으로 완료된 작업을 보고 프로젝트가 완료된 것으로 오해하여 작업을 중단합니다.
- 에이전트가 컨텍스트 예산의 대부분을 이전에 무슨 일이 있었는지 파악하는 데 소비합니다.

해결책은 파일과 git 히스토리를 통해 에이전트에게 **외부 메모리**를 제공하고, **체계적인 세션별 워크플로우**를 강제하는 것입니다.

---

## 아키텍처: 두 가지 프롬프트 모드, 하나의 에이전트

두 모드 모두 동일한 기반 모델과 도구를 사용합니다. 유일한 차이점은 프롬프트입니다.

### 모드 1: 초기화 프롬프트 (첫 번째 세션에서만 사용)

**목표**: 향후 세션이 의존할 모든 파일과 구조를 생성합니다. 아직 애플리케이션 코드를 작성하지 마세요.

초기화 에이전트는 다음 산출물을 생성해야 합니다:

#### 1. 기능 목록 (`feature_list.json`)

사용자의 상위 수준 목표를 세분화된, 테스트 가능한 기능으로 분해합니다. Markdown이 아닌 JSON을 사용하세요 — 모델이 JSON에서 항목을 실수로 재구성하거나 삭제할 가능성이 낮습니다.

각 기능 항목에는 다음이 포함되어야 합니다:

```json
{
  "id": 1,
  "category": "functional",
  "description": "사용자가 새 채팅을 열고 AI 응답을 받을 수 있다",
  "verification_steps": [
    "메인 페이지로 이동",
    "'새 채팅' 클릭",
    "메시지를 입력하고 Enter를 누름",
    "AI 응답이 나타나는지 확인"
  ],
  "passes": false
}
```

이 파일에 대한 규칙:
- 모든 기능은 `"passes": false`로 시작합니다.
- 이후 에이전트는 `passes` 필드만 변경할 수 있습니다. 항목을 삭제하거나, 내용을 수정하거나, 순서를 변경해서는 안 됩니다.
- 코딩 에이전트 프롬프트에 다음 지시를 명시적으로 포함하세요: "기능 정의를 제거하거나 편집하는 것은 허용되지 않습니다. 통과 횟수 필드를 false에서 true로만 변경할 수 있습니다."

#### 2. 시작 스크립트 (`init.sh`)

다음을 수행하는 셸 스크립트:
- 필요 시 종속성 설치
- 개발 서버 시작
- 기본적인 스모크 테스트 실행 (예: 헬스 엔드포인트에 curl 요청, 또는 브라우저를 실행하여 메인 페이지 로드 확인)

이를 통해 이후 에이전트가 프로젝트 실행 방법을 파악하는 데 소비할 시간을 제거합니다.

#### 3. 진행 로그 (`claude-progress.txt`)

빈 텍스트 파일 또는 최소한의 초기 내용이 있는 텍스트 파일입니다. 이후 각 세션에서 여기에 요약을 추가합니다. 다음과 같은 형식으로 작성합시다.

```
## 세션 3 — 2025-11-26
- 기능 #12 구현: 사용자가 대화를 삭제할 수 있음
- 버그 수정: 삭제 후 사이드바가 업데이트되지 않는 문제
- 기존 모든 기능이 여전히 스모크 테스트를 통과함
- 다음 우선순위: 기능 #13 (대화 검색)
```

#### 4. Git 저장소

git 저장소를 초기화하고 모든 스캐폴딩 파일을 포함한 초기 커밋을 생성합니다. 이를 통해 이후 에이전트에게 깨끗한 시작점과 완전한 변경 이력을 제공합니다.

```bash
git init
git add .
git commit -m "프로젝트 스캐폴딩 초기화: 기능 목록, 초기화 스크립트, 진행 로그"
```

### 모드 2: 코딩 프롬프트 (첫 번째 이후 모든 세션)

**목표**: 정확히 하나의 기능에 대해 점진적 진전을 이루고, 코드베이스를 깨끗한 상태로 남기고, 무엇이 수행되었는지 문서화합니다.

모든 코딩 세션은 다음의 정확한 순서를 따라야 합니다:

#### 단계 1: 상황 파악

    1. `pwd`를 실행하여 작업 디렉토리를 확인합니다.
    2. `claude-progress.txt`를 읽어 이전 세션에서 수행된 작업을 파악합니다.
    3. `git log --oneline -20`을 실행하여 최근 커밋을 확인합니다.
    4. `feature_list.json`을 읽어 전체 진행 상황을 확인합니다.

#### 단계 2: 기존 기능 검증

    1. `init.sh`를 실행하여 개발 서버를 시작합니다.
    2. 핵심 기능에 대한 기본적인 엔드투엔드 테스트를 수행합니다.
       - 웹 앱의 경우: 브라우저를 열고 핵심 사용자 플로우를 수행하여 동작을 확인합니다.
       - 백엔드 API의 경우: 주요 엔드포인트를 호출하고 응답을 검증합니다.
    3. 문제가 있으면 새 작업을 시작하기 전에 먼저 수정합니다.

**왜 중요한가**: 이전 세션이 버그를 남겼다면, 깨진 코드 위에 새 기능을 구축하면 문제가 복잡해집니다. 항상 안정화를 먼저 하고 진전하세요.

#### 단계 3: 기능 하나 선택

    1. `feature_list.json`에서 `passes`가 `false`인 가장 높은 우선순위의 기능을 찾습니다.
    2. 해당 기능 하나에 대해서만 작업합니다. 같은 세션에서 두 번째 기능을 시작하지 마세요.

**왜 하나만인가**: 여러 기능을 시도하면 구현 도중 컨텍스트가 소진되어, 문서화 없이 반쯤 완성된 코드를 남길 위험이 있습니다 — 이 하네스가 방지하려는 바로 그 문제입니다.

#### 단계 4: 구현

선택한 기능에 대한 코드를 작성합니다. 표준 개발 관행을 적용합니다.

#### 단계 5: 엔드투엔드 테스트

    1. 단위 테스트나 수동 curl 명령에만 의존하지 마세요.
    2. 실제 사용자가 하듯이 기능을 테스트하세요:
       - 웹 앱의 경우: 브라우저 자동화(예: Puppeteer)를 사용하여 UI를 클릭합니다.
       - API의 경우: 현실적인 요청 시퀀스를 보내고 전체 응답 체인을 검증합니다.
    3. 이 테스트를 통과한 후에만 `feature_list.json`에서 해당 기능을 `"passes": true`로 표시합니다.

**일반적인 오류 유형**: 에이전트는 코드를 작성하고, 단위 테스트를 실행하고, 테스트가 통과하는 것을 확인한 후 기능을 완료로 표시합니다. 하지만 실제로는 기능이 엔드 투 엔드로 제대로 작동하지 않습니다. 실제 사용자로써 테스트하도록 명시적으로 안내하면 이 문제가 해결됩니다.

#### 단계 6: 커밋 및 문서화

    1. 설명적인 메시지와 함께 모든 변경 사항을 스테이징하고 커밋합니다:
       git add .
       git commit -m "기능 #12 구현: 사이드바 동기화와 함께 대화 삭제"

    2. `claude-progress.txt`에 세션 요약을 추가합니다:
       - 구현된 기능
       - 발견하고 수정한 버그
       - 애플리케이션의 현재 상태
       - 다음에 작업할 권장 기능

    3. `feature_list.json` 업데이트: 완료된 기능의 `passes`를 `true`로 설정합니다.

#### 단계 7: 코드베이스를 머지 가능한 상태로 남기기

세션을 종료하기 전에 다음을 확인합니다:
- [ ] 기존 기능에 주요 버그가 없음
- [ ] 코드가 읽기 쉽고 적절히 문서화되어 있음
- [ ] 반쯤 구현된 기능이 코드베이스에 남아 있지 않음
- [ ] 새로운 개발자(또는 에이전트)가 정리 작업 없이 즉시 다음 기능을 시작할 수 있음

## 프롬프트 템플릿

### 초기화 프롬프트 (세션 1)

    장기간 진행될 개발 프로젝트를 위한 작업 공간을 설정하고 있습니다.
    목표는 아직 애플리케이션 코드를 작성하는 것이 아닙니다. 대신, 향후 코딩
    세션이 효율적으로 진행될 수 있도록 기반을 마련하는 것입니다.

    사용자는 다음을 구축하고자 합니다: [사용자의 개략적인 설명]

    다음을 생성해야 합니다:
    1. feature_list.json — 위의 내용을 50~200개 이상의 세분화되고 테스트 가능한
       기능으로 분해합니다. 각 항목에는 id, category, description,
       verification_steps, passes(false로 설정)가 포함되어야 합니다.
       JSON 형식을 사용하세요. Markdown은 사용하지 마세요.

    2. init.sh — 종속성을 설치하고, 개발 서버를 시작하고,
       기본적인 스모크 테스트를 실행하는 스크립트입니다.

    3. claude-progress.txt — 목적을 설명하는 헤더가 있는 빈 진행 로그 파일입니다.

    4. Git 저장소를 초기화하고 모든 파일을 커밋합니다.

    애플리케이션 코드는 작성하지 마십시오.
    여러분의 유일한 임무는 이 환경을 설정하는 것입니다.

### 코딩 프롬프트 (세션 2 이후)

    당신은 기존 프로젝트에서 작업을 이어가는 코딩 에이전트입니다.

    필수 시작 시퀀스:
    1. `pwd` 실행
    2. `claude-progress.txt` 읽기
    3. `git log --oneline -20` 실행
    4. `feature_list.json` 읽기
    5. `init.sh`를 실행하여 개발 서버 시작
    6. 핵심 기능을 엔드투엔드로 테스트. 문제가 있으면 먼저 수정.

    그 다음:
    7. feature_list.json에서 passes가 false인 가장 높은 우선순위의 기능을 선택.
    8. 해당 기능 하나만 구현.
    9. 실제 사용자처럼 엔드투엔드로 테스트
       (웹 앱의 경우 브라우저 자동화 사용).
    10. 기능이 실제로 동작한 후에만 passes를 true로 설정.

    필수 종료 시퀀스:
    11. 설명적인 메시지와 함께 git add 및 commit.
    12. claude-progress.txt에 세션 요약 추가.
    13. 코드베이스가 깨끗하고 머지 가능한 상태인지 확인.

    규칙:
    - 세션당 정확히 하나의 기능만 작업할 것.
    - feature_list.json의 기능 정의를 절대 제거하거나 편집하지 말 것.
      passes 필드만 변경할 수 있음.
    - feature_list.json의 모든 기능이 passes: true가 아닌 한
      프로젝트가 완료되었다고 선언하지 말 것.
    - 기존 기능에서 버그를 발견하면 새 작업 전에 먼저 수정할 것.
    - 단위 테스트만이 아닌 항상 엔드투엔드로 테스트할 것.

## 흔한 함정과 완화 방법

| 함정 | 완화 방법 |
|---|---|
| 에이전트가 한 번에 모든 것을 구축하려 함 | 프롬프트에서 세션당 하나의 기능으로 제한 |
| 에이전트가 프로젝트를 조기에 완료로 선언 | 명시적인 `passes: false` 플래그가 있는 기능 목록이 에이전트에게 완료 상태를 확인하도록 강제 |
| 에이전트가 세션 종료 시 깨진 코드를 남김 | 필수 git 커밋 + 진행 로그 + 머지 가능 상태 체크리스트 |
| 에이전트가 실제 테스트 없이 기능을 완료로 표시 | 브라우저 자동화 또는 현실적인 API 호출을 통한 엔드투엔드 테스트 요구 |
| 에이전트가 앱 실행 방법을 파악하는 데 시간을 소비 | `init.sh` 스크립트가 이 오버헤드를 제거 |
| 에이전트가 기능 목록을 수정하거나 삭제 | JSON 사용 (실수로 재구성하기 어려움) + 강력한 프롬프트 가드레일 |
| 에이전트가 깨진 코드 위에 새 기능을 시작 | 새 작업 전 "기존 기능 검증" 단계 필수화 |

## 다양한 프로젝트 유형에 적용하기

이 가이드는 웹 앱 개발을 주요 예시로 사용하지만, 패턴은 일반화할 수 있습니다:

**백엔드 API 개발 (Spring Boot 등)**
- `init.sh`가 애플리케이션 서버를 시작하고 헬스 체크를 실행
- 엔드투엔드 테스트 = 실행 중인 엔드포인트에 대한 현실적인 HTTP 요청 시퀀스
- 기능 목록 항목은 API 동작을 기술: "POST /orders는 주문을 생성하고 201을 반환한다"

**데이터 파이프라인 / 배치 처리**
- `init.sh`가 테스트 데이터를 설정하고 최소한의 파이프라인을 실행
- 엔드투엔드 테스트 = 입력 → 처리 → 출력 체인 검증
- 기능 목록 항목은 변환을 기술: "파이프라인이 컬럼 X의 null 값을 올바르게 처리한다"

**인프라 / DevOps**
- `init.sh`가 도구 버전과 클라우드 자격 증명을 검증
- 엔드투엔드 테스트 = 스테이징 환경에 배포하고 검증
- 기능 목록 항목은 인프라 상태를 기술: "CPU 80%에서 오토스케일링이 트리거된다"
