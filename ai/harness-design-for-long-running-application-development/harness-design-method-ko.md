# 장기 실행 AI 에이전트 하네스: 구현 가이드

## 이 문서의 목적

이 문서는 AI 코딩 에이전트가 단일 프로젝트에서 여러 세션에 걸쳐 효율적으로 작동할 수 있도록 하는 하네스를 구축하기 위한 완벽한 구현 가이드입니다. 이 가이드를 그대로 따라 진행하십시오. 외부 참조는 필요하지 않습니다.

---

## 해결해야 할 핵심 제약 조건

AI 에이전트는 고정된 크기의 컨텍스트 윈도우 내에서 작동합니다. 이 윈도우가 가득 차면 세션이 종료되고 이전 세션에 대한 **기억** 없이 새 세션이 시작됩니다. 체계적인 접근 방식이 없으면 다음과 같은 문제가 발생합니다.

- 에이전트가 한 세션에서 모든 작업을 완료하려고 시도하다가 작업 도중 컨텍스트가 부족해져 오류가 발생하고 문서화되지 않은 코드를 남깁니다.
- 에이전트가 이전 세션에서 부분적으로 완료된 작업을 보고 프로젝트가 완료된 것으로 잘못 판단하여 작업을 중단합니다.
- 에이전트가 컨텍스트 예산의 대부분을 이전 세션에서 무슨 일이 있었는지 파악하는 데 소모합니다.

해결책은 파일과 Git 기록을 통해 에이전트에게 **외부 메모리**를 제공하고 **세션별로 정해진 워크플로**를 시행하는 것입니다.

---

## 배경 및 동기

저자는 이전에 두 가지 관련 프로젝트를 진행했습니다. 하나는 "프런트엔드 디자인 스킬"(Claude의 UI 출력 품질을 향상시키는 프롬프트/지시 세트)이고, 다른 하나는 장시간 실행되는 코딩 에이전트 하네스(Claude가 장시간 세션 동안 앱을 구축할 수 있도록 하는 오케스트레이션 시스템)입니다. 두 프로젝트 모두 프롬프트 엔지니어링과 하네스 설계를 통해 기준선 대비 성능 향상을 달성했지만, 결국 성능 한계에 도달했습니다.

이러한 한계를 극복하기 위해 저자는 GAN(생성적 적대 신경망)에서 영감을 받아, 개별 에이전트가 출력을 생성하고 평가하여 품질 향상을 위한 피드백 루프를 구축하는 시스템을 설계했습니다.

---

## 하네스가 해결하는 두 가지 실패 모드

### 1. 컨텍스트 저하

작업 시간이 길어지고 컨텍스트 창이 가득 차면 모델의 일관성이 떨어집니다. 일부 모델은 컨텍스트 한계에 가까워지고 있음을 감지하여 작업을 조기에 종료하는 "컨텍스트 불안" 현상을 보이기도 합니다.

**해결책: 컨텍스트 재설정.** 이전 대화를 요약(압축)하는 대신, 에이전트는 컨텍스트를 완전히 지우고 이전 에이전트의 상태와 다음 단계를 담은 구조화된 핸드오프 아티팩트를 사용하여 새로운 에이전트를 시작합니다. 이렇게 하면 에이전트가 깨끗한 상태에서 시작할 수 있습니다. 테스트 결과, Claude Sonnet 4.5는 컨텍스트 불안이 매우 심하여 압축만으로는 충분하지 않았으며, 컨텍스트 재설정이 필수적이었습니다.

> 참고: Opus 4.5 및 이후 Opus 4.6에서는 컨텍스트 불안이 대부분 사라져 컨텍스트 재설정의 필요성이 줄어들었습니다.

### 2. 자기 평가 편향

에이전트는 자신의 출력을 평가하도록 요청받았을 때, 인간의 눈으로 보기에 품질이 명백히 떨어지는 경우에도 일관적으로 자신의 작업을 과대평가하는 경향이 있습니다. 이는 주관적인 작업(설계)에서 특히 문제가 되지만, 객관적인 작업(기능적 정확성)에서도 나타납니다.

**해결책: 생성기와 평가기 분리.** 독립적인 평가기 에이전트가 생성기의 출력을 평가합니다. 독립적인 평가자가 회의적인 태도를 갖도록 조정하는 것은 생성기를 자기비판적으로 만드는 것보다 훨씬 더 다루기 쉽습니다. 평가자는 생성기가 개선해 나갈 수 있는 구체적인 피드백을 제공합니다.

## 1부: 프런트엔드 디자인 실험

### 문제

Claude는 아무런 개입 없이 안전하고 일반적인 레이아웃을 선호하는 경향이 있습니다. 기술적으로는 기능적이지만 시각적으로는 특별할 것 없는 레이아웃입니다.

### 접근 방식

저자는 생성기 및 평가기 에이전트 모두에게 다음과 같은 네 가지 평가 기준을 제시했습니다.

1. **디자인 품질** - 디자인이 일관된 분위기/정체성을 가진 하나의 완성된 작품처럼 느껴지는가? (색상, 타이포그래피, 레이아웃, 이미지가 조화롭게 어우러지는가?)
2. **독창성** - 템플릿 기본값이나 AI 생성 패턴이 아닌, 의도적인 창의적 선택이 반영된가? 흰색 카드 위에 보라색 그라데이션을 사용하는 것과 같은 "AI 기반 디자인의 허술함"은 감점 요인입니다.
3. **실력** - 기술적 완성도: 타이포그래피 계층 구조, 간격 일관성, 색상 조화, 명암비. 숙련도 검증. Claude는 기본적으로 이 부분에서 높은 점수를 받습니다.
4. **기능성** - 미적 요소와 무관한 사용성. 사용자가 인터페이스를 이해하고, 필요한 기능을 찾고, 작업을 완료할 수 있는가?

클로드 에이전트는 이미 완성도와 기능성 면에서 뛰어난 성능을 보였기 때문에 디자인 품질과 독창성에 더 큰 비중을 두었습니다.

### 피드백 루프 메커니즘

- 클로드 에이전트 SDK 기반으로 구축되었습니다.
- 생성기는 사용자의 입력에 따라 HTML/CSS/JS 프런트엔드를 생성합니다.
- 평가자는 Playwright MCP를 사용하여 실제 페이지(정적 스크린샷이 아닌)를 탐색하고 상호 작용한 후 각 기준에 점수를 매기고 상세한 비평을 작성합니다.
- 피드백은 다음 반복을 위해 생성기로 다시 전달됩니다.
- 생성당 5~15회의 반복이 진행되었으며, 전체 실행에는 최대 4시간이 소요되었습니다.
- 생성기는 각 평가 후 전략적 선택을 하도록 지시받았습니다. 점수 추세가 좋으면 현재 방향을 개선하고, 접근 방식이 효과적이지 않으면 완전히 다른 미적 방향으로 전환하도록 했습니다.

### 주요 결과

- 점수는 일반적으로 반복을 거듭할수록 향상되다가 정체되었지만, 항상 선형적으로 향상되지는 않았습니다. 작성자는 최종 버전보다 중간 버전을 선호하는 경우도 있었습니다.
- 생성기가 더욱 야심찬 솔루션을 추구함에 따라 구현 복잡성은 라운드를 거듭할수록 증가했습니다.
- 첫 번째 반복조차도 아무런 피드백이 없는 기준선보다 우수했는데, 이는 평가 기준 언어 자체가 평가자의 피드백이 있기 전에 모델을 일반적인 기본값에서 벗어나도록 유도했음을 시사합니다.
- 평가 기준의 표현 방식은 예상치 못한 방식으로 출력물의 특성을 형성했습니다. 예를 들어, "최고의 디자인은 박물관 수준이어야 한다"는 기준은 디자인을 특정 시각적 방향으로 이끌었습니다.
- 주목할 만한 예: 네덜란드 미술관 웹사이트 생성 과제. 9번의 반복을 거치는 동안 생성기는 세련된 다크 테마의 랜딩 페이지를 생성했습니다. 10번째 반복에서는 모든 것을 폐기하고 3D 공간 경험을 구축했습니다. 바둑판 무늬 바닥, 벽에 걸린 작품, 그리고 갤러리 공간 사이를 이동하는 출입구를 갖춘 CSS 원근 렌더링 방식의 방을 구현한 것입니다. 이러한 창의적인 도약은 단일 패스 생성 방식으로는 전례가 없는 일이었습니다.

## 파트 2: 풀스택 애플리케이션 개발

### 아키텍처 (V1 — 스프린트 방식의 3개 에이전트 시스템)

**플래너 에이전트:**
- 1~4문장으로 된 사용자 프롬프트를 입력받아 완전한 제품 사양으로 확장합니다.
- 범위에 대해 야심차게 설정하도록 지시받습니다.
- 세부적인 구현 사항보다는 제품의 맥락과 고수준 기술 설계에 집중합니다(사양 오류의 연쇄 발생 방지).
- AI 기능(Claude 기반)을 제품에 통합할 수 있는 기회를 찾도록 지시받습니다.

**제너레이터 에이전트:**
- 스프린트 방식으로 작업하며, 사양에서 한 번에 하나의 기능을 구현합니다.
- 기술 스택: React, Vite, FastAPI, SQLite (추후 PostgreSQL 사용 예정).
- 각 스프린트 종료 시 QA에 인계하기 전에 자체 평가를 수행합니다.
- 버전 관리를 위해 Git을 사용합니다.

**평가자(QA) 에이전트:**
- Playwright MCP를 사용하여 실제 사용자처럼 실행 중인 애플리케이션을 탐색합니다.
- UI 기능, API 엔드포인트 및 데이터베이스 상태를 테스트합니다.
- 제품의 완성도, 기능성, 시각적 디자인, 코드 품질 등의 기준에 따라 각 스프린트를 평가합니다.
- 각 기준에는 엄격한 합격/불합격 기준이 있습니다. 기준 중 하나라도 미달할 경우, 스프린트는 상세한 피드백과 함께 개발자에게 반환됩니다.

**스프린트 계약:**
- 각 스프린트 시작 전에 개발자와 평가자는 "완료"의 기준을 협의합니다.
- 개발자는 개발할 내용과 성공 검증 방법을 제안합니다.
- 평가자는 제안 내용을 검토하여 정확성을 확인합니다.
- 합의에 도달할 때까지 반복 작업을 진행합니다.
- 의사소통은 파일을 통해 이루어집니다(한 에이전트가 작성하고, 다른 에이전트가 읽고 응답합니다).

### V1 테스트: 레트로 게임 제작 도구

**요구사항:** "레벨 에디터, 스프라이트 에디터, 엔티티 동작, 플레이 가능한 테스트 모드 등의 기능을 갖춘 2D 레트로 게임 제작 도구를 만드세요."

| 접근 방식    | 소요 시간 | 비용 |
|----------|----------|------|
| 단일 에이전트 (하네스 없음) | 20분 | $9 |
| 3개 에이전트 하네스 전체 사용 | 6시간 | $200 |

**단일 에이전트 결과:**
- 앱은 처음 봤을 때는 괜찮아 보였습니다.
- 레이아웃이 공간 낭비가 심했습니다 (고정 높이 패널, 대부분 비어 있는 뷰포트).
- 필요한 순서에 대한 지침이 없는 경직된 워크플로였습니다 (스프라이트 생성 → 엔티티 생성 → 레벨 채우기).
- **게임에 근본적인 문제가 있었습니다:** 엔티티는 화면에 나타났지만 입력에 반응하지 않았습니다. 엔티티 정의와 게임 런타임 간의 연결이 끊어져 있었습니다.

**최종 결과:**
- 기획자는 10개의 스프린트에 걸쳐 16개의 기능을 포함하는 구체적인 사양으로 과제를 확장했습니다. 추가된 기능은 스프라이트 애니메이션, 동작 템플릿, 음향 효과/음악, AI 기반 스프라이트 생성기 및 레벨 디자이너, 공유 가능한 링크를 포함한 게임 내보내기 기능입니다.
- 앱은 전체 뷰포트를 사용했고, 일관된 시각적 정체성을 유지했으며, 패널 크기가 적절했습니다.
- **게임이 실제로 작동했습니다.** - 캐릭터가 움직이고 레벨을 플레이할 수 있었습니다.
- 물리 엔진은 미흡한 부분(캐릭터가 플랫폼과 겹치는 현상)과 AI 생성 레벨 디자인의 일부 문제점(넘을 수 없는 벽)이 있었지만, 핵심 기능은 온전했습니다.
- 내장된 Claude 통합 기능을 통해 프롬프트를 사용하여 게임 구성 요소를 생성할 수 있었습니다.

**평가자가 발견한 버그 예시:**

| 계약 기준 | 발견 사항 |
|---|---|
| 사각형 채우기 도구는 클릭 드래그로 영역을 채워야 합니다 | 실패 — 드래그 시작/끝 지점에만 타일이 배치되었습니다. `fillRectangle` 함수는 존재하지만 mouseUp 이벤트에서 실행되지 않습니다. |
| 사용자가 엔티티 생성 지점을 선택하고 삭제할 수 있습니다. | 실패 — 삭제 핸들러는 `selection`과 `selectedEntityId` 모두를 필요로 하지만, 클릭 시에는 `selectedEntityId`만 설정됩니다. |
| 사용자가 API를 통해 애니메이션 프레임 순서를 변경할 수 있습니다. | 실패 — `/{frame_id}` 경로 이후에 `PUT /frames/reorder`가 정의되어 있습니다. FastAPI는 "reorder"를 정수 frame_id로 인식하여 422 오류를 발생시킵니다. |

**평가기 튜닝 과제:**
- 기본 설정에서 Claude는 QA 에이전트로서의 기능이 부족했습니다.
- 실제 문제를 발견한 후에도 스스로 중요하지 않다고 판단하여 승인하는 경우가 있었습니다.
- 표면적인 테스트만 수행하여 예외적인 경우를 놓쳤습니다.
- 평가기 로그를 읽고 → 사람의 평가와 다른 부분을 찾아 → 프롬프트를 업데이트하고 → 이 과정을 여러 번 반복해야 했습니다.

## 3부: 하네스 간소화 (V2)

### 동기

V1 하네스는 효과적이었지만, 복잡하고 느리며 비용이 많이 들었습니다. 모든 하네스 구성 요소는 모델이 단독으로 수행할 수 없는 작업에 대한 가정을 내포하고 있습니다. 이러한 가정은 모델이 개선됨에 따라 잘못되었거나 시대에 뒤떨어질 수 있으므로 스트레스 테스트를 거쳐야 합니다.

핵심 원칙은 "가능한 한 가장 간단한 솔루션을 찾고, 필요한 경우에만 복잡성을 증가시킨다"였습니다.

### 간소화를 가능하게 한 Opus 4.6 기능

본 작업 중에 출시된 Opus 4.6은 다음과 같은 기능을 제공했습니다. 더욱 신중한 계획 수립, 더 오래 지속되는 에이전트 기반 작업, 대규모 코드베이스에서의 더욱 안정적인 작동, 향상된 코드 검토 및 디버깅, 그리고 크게 개선된 장기 컨텍스트 검색 기능. 이러한 기능들은 모두 하네스가 보완하기 위해 개발된 기능들이었습니다.

### 주요 변경 사항: 스프린트 구조 제거

- Opus 4.6은 스프린트 분할 없이도 2시간 이상 일관성 있게 작동할 수 있었습니다.
- 에이전트는 컨텍스트 증가를 처리하는 자동 압축 기능을 갖춘 하나의 연속적인 세션으로 실행되었습니다.
- 평가자는 스프린트별 실행에서 실행 종료 시 한 번만 실행되도록 변경되었습니다.
- 계획자와 평가자는 여전히 분명한 가치를 제공하므로 유지되었습니다.

### 평가자가 중요한 경우

평가자의 가치는 모델의 기능에 따라 달라집니다.

- Sonnet 4.5 버전: 빌드가 생성기가 단독으로 잘 처리할 수 있는 한계에 가까울 때 → 평가자가 중요한 문제를 지속적으로 발견했습니다.
- Opus 4.6 버전: 모델의 기본 기능이 향상되어 한계가 확장됨 → 이전에는 평가자가 필요했던 작업이 이제 단독으로도 잘 처리되었습니다. 하지만 생성기의 기능 한계에 여전히 있는 작업의 경우 평가자는 여전히 실질적인 성능 향상을 제공했습니다.

**핵심 요점:** 평가자 사용 여부는 고정된 결정이 아닙니다. 현재 모델이 단독으로 안정적으로 처리할 수 있는 범위를 넘어서는 작업이 있을 때 평가자를 사용하는 것이 비용 대비 가치가 있습니다.

### V2 테스트: 브라우저 기반 DAW(디지털 오디오 워크스테이션)

**주요 과제:** "웹 오디오 API를 사용하여 브라우저에서 모든 기능을 갖춘 DAW를 구축하세요."

- 전문적인 DAW는 아니고, 클로드는 청력이 없어서 음악적 취향에 대한 QA 효과가 제한적이지만, 곡 작곡에 필요한 핵심 기능들은 모두 갖춰져 있었고 에이전트가 자율적으로 사용할 수 있었습니다.

## 교훈 및 핵심 사항

1. **생성과 평가를 분리하세요.** 독립적인 평가자를 회의적으로 만드는 것은 생성기 자체를 비판적으로 만드는 것보다 훨씬 쉽습니다.

2. **주관적인 품질도 등급을 매길 수 있습니다.** "이것이 아름다운가?"라는 질문에 일관되게 답하기는 어렵지만, "이것이 구체적인 디자인 원칙을 따르는가?"라는 질문은 모델이 평가할 수 있는 구체적인 기준을 제공합니다.

3. **모든 구성 요소는 모델의 한계에 대한 가정입니다.** 이러한 가정은 모델이 개선됨에 따라 더 이상 유효하지 않으므로 정기적으로 스트레스 테스트를 수행해야 합니다.

4. **새로운 모델이 나오면 기존 구성을 재검토하세요.** 더 이상 중요하지 않은 부분을 제거하고, 이전에는 불가능했던 기능을 구현하기 위해 새로운 부분을 추가하세요.

5. **흥미로운 구성 요소 조합의 가능성은 모델이 개선됨에 따라 줄어들지 않고 오히려 확장됩니다.** AI 엔지니어의 과제는 끊임없이 새로운 조합을 찾아내는 것입니다.

6. **에이전트 추적 로그 읽기에 투자하세요.** 평가자 튜닝 루프(로그 읽기 → 판단 오류 찾기 → 프롬프트 업데이트)는 필수적이었으며, 단축할 수 없었습니다.

7. **플래너 에이전트는 모델 성능이 우수하더라도 추가적인 가치를 제공합니다.** 플래너가 없으면 생성기가 범위를 벗어나 기능이 부족한 애플리케이션을 생성합니다.
