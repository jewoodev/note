# 컨텍스트: Anthropic의 장시간 자율 코딩을 위한 멀티 에이전트 하네스

> 출처: Anthropic Engineering Blog, "Harness design for long-running application development", Prithvi Rajasekaran (Anthropic Labs) 저, 2026년 3월 24일 게시.
> 목적: 이 문서는 AI 에이전트가 원문에 접근하지 않고도 해당 글에 대해 논의, 분석하거나 질문에 답변할 수 있도록 완전한 컨텍스트를 제공한다.

---

## 이 글의 주제

이 글은 Anthropic Labs의 엔지니어가 Claude가 수 시간에 걸친 코딩 세션을 통해 완성도 높은 풀스택 애플리케이션을 자율적으로 구축할 수 있도록 하는 멀티 에이전트 하네스(오케스트레이션 시스템)를 설계하고 반복 개선한 과정을 다룬다. 작업은 프론트엔드 UI 디자인과 풀스택 애플리케이션 개발 두 영역을 포괄한다.

---

## 배경 및 동기

저자는 이전에 두 가지 관련 프로젝트를 수행한 바 있다: Claude의 UI 출력 품질을 향상시키는 프롬프트/지시 세트인 "프론트엔드 디자인 스킬"과, Claude가 장시간 세션에 걸쳐 앱을 구축할 수 있게 하는 장시간 코딩 에이전트 하네스다. 두 프로젝트 모두 프롬프트 엔지니어링과 하네스 설계를 통해 베이스라인 대비 성능 향상을 달성했지만, 결국 성능 한계에 도달했다.

이를 돌파하기 위해 저자는 GAN(Generative Adversarial Networks)에서 영감을 받아, 별도의 에이전트가 출력을 생성하고 평가하는 시스템을 설계하여 품질을 상향시키는 피드백 루프를 만들었다.

---

## 하네스가 해결하는 두 가지 실패 모드

### 1. 컨텍스트 저하

태스크가 길어지고 컨텍스트 윈도우가 채워질수록 모델은 일관성을 잃는다. 일부 모델은 "컨텍스트 불안(context anxiety)"도 보이는데, 컨텍스트 한계에 가까워지고 있다고 감지하면 작업을 조기에 마무리하려고 한다.

**해결책: 컨텍스트 리셋.** 이전 대화를 요약하여 그 자리에 남기는 방식(컴팩션) 대신, 컨텍스트를 완전히 비우고 이전 에이전트의 상태와 다음 단계를 담은 구조화된 핸드오프 아티팩트와 함께 새로운 에이전트를 시작한다. 이렇게 하면 에이전트에게 깨끗한 슬레이트를 제공할 수 있다. 테스트에서 Claude Sonnet 4.5는 컨텍스트 불안이 매우 강해 컴팩션만으로는 부족했기 때문에 컨텍스트 리셋이 필수적이었다.

참고: Opus 4.5와 이후 Opus 4.6에서는 컨텍스트 불안이 대부분 사라져 컨텍스트 리셋의 필요성이 줄었다.

### 2. 자기 평가 편향

에이전트에게 자신의 출력을 평가하라고 하면, 인간에게 품질이 명백히 평범해 보이는 경우에도 일관되게 자기 작업을 과대 칭찬한다. 이는 주관적 태스크(디자인)에서 특히 문제가 되지만, 객관적 태스크(기능 정확성)에서도 나타난다.

**해결책: 생성자와 평가자를 분리한다.** 독립된 평가자 에이전트가 생성자의 출력을 판단한다. 독립 평가자를 회의적으로 튜닝하는 것이 생성자를 자기 비판적으로 만드는 것보다 훨씬 다루기 쉽다. 평가자는 생성자가 반복 개선할 수 있는 구체적인 피드백을 제공한다.

## 파트 1: 프론트엔드 디자인 실험

### 문제

별도의 개입 없이 Claude는 안전하고 범용적인 레이아웃으로 수렴하는 경향이 있다 — 기술적으로 작동하지만 시각적으로 평범하다.

### 접근 방식

저자는 생성자와 평가자 에이전트 모두에게 제공할 네 가지 채점 기준을 만들었다:

1. **디자인 품질** — 디자인이 독특한 분위기/정체성을 가진 일관된 전체로 느껴지는가? (색상, 타이포그래피, 레이아웃, 이미지가 함께 작동하는가.)
2. **독창성** — 의도적인 창작 선택의 증거가 있는가, 아니면 템플릿 기본값과 AI 생성 패턴인가. 보라색 그라디언트 위의 흰색 카드 같은 "AI 슬롭"을 감점한다.
3. **완성도** — 기술적 실행: 타이포그래피 위계, 간격 일관성, 색상 조화, 대비 비율. 역량 검사. Claude는 기본적으로 여기서 높은 점수를 받는다.
4. **기능성** — 미학과 독립적인 사용성. 사용자가 인터페이스를 이해하고, 액션을 찾고, 태스크를 완료할 수 있는가?

Claude가 이미 완성도와 기능성에서 좋은 성적을 보였기 때문에 디자인 품질과 독창성에 더 높은 가중치를 부여했다.

### 피드백 루프 메커니즘

- Claude Agent SDK 위에 구축했다.
- 생성자가 사용자 프롬프트로부터 HTML/CSS/JS 프론트엔드를 생성한다.
- 평가자가 Playwright MCP를 사용해 라이브 페이지를 탐색하고 상호작용한 뒤(정적 스크린샷이 아님), 각 기준을 채점하고 상세한 비평을 작성한다.
- 피드백이 다음 반복을 위해 생성자에게 전달된다.
- 생성당 5~15회 반복. 전체 실행은 최대 4시간 소요.
- 생성자는 매 평가 후 전략적 선택을 하도록 지시받았다: 점수 추세가 좋으면 현재 방향을 개선하고, 접근법이 통하지 않으면 완전히 다른 미학으로 전환한다.

### 주요 발견

- 점수는 일반적으로 반복에 따라 개선되다가 고원에 도달했지만, 항상 선형적이지는 않았다. 저자는 때때로 중간 반복을 최종 반복보다 선호했다.
- 구현 복잡도가 라운드에 걸쳐 증가했으며, 생성자가 점점 더 야심찬 솔루션을 시도했다.
- 첫 번째 반복조차 프롬프팅 없는 베이스라인보다 나았으며, 이는 기준의 언어 자체가 평가자 피드백 이전에 모델을 범용적 기본값에서 벗어나게 했음을 시사한다.
- 기준의 표현이 예상치 못한 방식으로 출력 특성을 형성했다 — 예: "최고의 디자인은 박물관 수준"이라는 문구가 디자인을 특정 시각적 수렴 방향으로 밀어붙였다.
- 주목할 만한 사례: 네덜란드 미술관 웹사이트 프롬프트. 9번의 반복을 거쳐 생성자는 세련된 다크 테마 랜딩 페이지를 만들었다. 10번째 반복에서 모든 것을 폐기하고 3D 공간 경험을 구축했다 — CSS perspective로 렌더링된 체크무늬 바닥의 방, 벽에 걸린 작품, 갤러리 방 사이의 문 기반 네비게이션. 이런 종류의 창의적 도약은 단일 패스 생성에서는 전례가 없었다.

## 파트 2: 풀스택 애플리케이션 개발

### 아키텍처 (V1 — 스프린트 기반 3-에이전트 시스템)

**플래너 에이전트:**
- 1~4문장의 사용자 프롬프트를 받아 전체 제품 명세로 확장한다.
- 범위를 야심차게 잡도록 프롬프팅한다.
- 제품 컨텍스트와 고수준 기술 설계에 집중하며, 세부적인 구현 사항은 다루지 않는다(명세 오류의 연쇄를 방지하기 위해).
- AI 기능(Claude 기반)을 제품에 엮어 넣을 기회를 찾도록 지시한다.

**생성자 에이전트:**
- 스프린트 단위로 작업하며, 명세에서 한 번에 하나의 기능을 구현한다.
- 기술 스택: React, Vite, FastAPI, SQLite(이후 PostgreSQL).
- 각 스프린트 끝에 자체 평가를 수행한 후 QA에 넘긴다.
- Git으로 버전 관리한다.

**평가자 (QA) 에이전트:**
- Playwright MCP를 사용해 실행 중인 애플리케이션을 실제 사용자처럼 클릭하며 테스트한다.
- UI 기능, API 엔드포인트, 데이터베이스 상태를 테스트한다.
- 제품 깊이, 기능성, 비주얼 디자인, 코드 품질을 커버하는 기준에 따라 각 스프린트를 채점한다.
- 각 기준에 통과/실패 임계값이 있다. 하나라도 실패하면 스프린트가 상세한 피드백과 함께 생성자에게 반환된다.

**스프린트 계약:**
- 각 스프린트 전에 생성자와 평가자가 "완료"의 기준을 협상한다.
- 생성자가 무엇을 만들고 어떻게 성공을 검증할지 제안한다.
- 평가자가 제안을 검토하여 정확성을 확인한다.
- 합의할 때까지 반복한다.
- 커뮤니케이션은 파일을 통해 이루어진다(한 에이전트가 쓰면 다른 에이전트가 읽고 응답).

### V1 테스트: 레트로 게임 메이커

**프롬프트:** "레벨 에디터, 스프라이트 에디터, 엔티티 행동, 플레이 가능한 테스트 모드를 포함하는 2D 레트로 게임 메이커를 만들어라."

| 접근 방식 | 소요 시간 | 비용 |
|----------|----------|------|
| 단일 에이전트 (하네스 없음) | 20분 | $9 |
| 전체 3-에이전트 하네스 | 6시간 | $200 |

**단일 에이전트 결과:**
- 앱이 첫눈에는 합리적으로 보였다.
- 레이아웃이 공간을 낭비했다(고정 높이 패널, 대부분 빈 뷰포트).
- 경직된 워크플로우로 필요한 순서에 대한 안내가 없었다(스프라이트 생성 → 엔티티 → 레벨 채우기).
- **게임이 근본적으로 작동하지 않았다:** 엔티티가 화면에 나타났지만 입력에 아무것도 반응하지 않았다. 엔티티 정의와 게임 런타임 간의 연결이 끊어져 있었다.

**전체 하네스 결과:**
- 플래너가 프롬프트를 10개 스프린트에 걸친 16개 기능 명세로 확장했다. 추가된 기능: 스프라이트 애니메이션, 행동 템플릿, 사운드 효과/음악, AI 지원 스프라이트 생성기 및 레벨 디자이너, 공유 링크가 포함된 게임 내보내기.
- 앱이 전체 뷰포트를 사용했고, 일관된 시각적 정체성과 합리적인 패널 크기를 가졌다.
- **게임이 실제로 작동했다** — 캐릭터를 움직이고 레벨을 플레이할 수 있었다.
- 물리 엔진에 거친 부분(캐릭터가 플랫폼과 겹침)과 AI 생성 레벨 디자인 문제(넘을 수 없는 벽)가 있었지만, 핵심 기능은 정상 작동했다.
- 내장된 Claude 통합으로 프롬프팅을 통해 게임 컴포넌트를 생성할 수 있었다.

**평가자가 잡아낸 버그 예시:**

| 계약 기준 | 발견 |
|---|---|
| 사각형 채우기 도구가 클릭-드래그로 영역을 채워야 함 | 실패 — 드래그 시작/끝 지점에만 타일 배치. `fillRectangle` 함수가 존재하지만 mouseUp에서 트리거되지 않음. |
| 사용자가 엔티티 스폰 지점을 선택하고 삭제할 수 있어야 함 | 실패 — Delete 핸들러가 `selection`과 `selectedEntityId` 모두 필요하지만, 클릭 시 `selectedEntityId`만 설정됨. |
| 사용자가 API를 통해 애니메이션 프레임 순서를 변경할 수 있어야 함 | 실패 — `PUT /frames/reorder`가 `/{frame_id}` 라우트 뒤에 정의됨. FastAPI가 "reorder"를 정수 frame_id로 매칭 → 422 에러. |

**평가자 튜닝의 어려움:**
- 기본 상태에서 Claude는 QA 에이전트로서 성능이 낮았다.
- 실제 문제를 발견하고도 스스로 별 것 아니라고 판단하여 승인해 버렸다.
- 피상적으로만 테스트하여 엣지 케이스를 놓쳤다.
- 여러 차례의 튜닝 루프가 필요했다: 평가자 로그 읽기 → 인간 평가와의 판단 차이 찾기 → 프롬프트 업데이트 → 반복.

## 파트 3: 하네스 단순화 (V2)

### 동기

V1 하네스는 효과적이었지만 무겁고, 느리고, 비쌌다. 하네스의 모든 컴포넌트는 모델이 스스로 할 수 없는 것에 대한 가정을 인코딩하며, 이 가정들은 틀릴 수 있고 모델이 개선됨에 따라 빠르게 낡을 수 있기 때문에 스트레스 테스트할 필요가 있다.

핵심 원칙: "가능한 가장 단순한 솔루션을 찾고, 필요할 때만 복잡성을 높여라."

### 단순화를 가능하게 한 Opus 4.6의 역량

이 작업 중에 출시된 Opus 4.6은 다음을 제공했다: 더 신중한 계획, 더 긴 에이전틱 태스크 유지, 더 큰 코드베이스에서의 안정적 작동, 더 나은 코드 리뷰 및 디버깅, 크게 개선된 긴 컨텍스트 검색. 이 모두는 하네스가 보완하도록 설계된 역량들이었다.

### 핵심 변경: 스프린트 구조 제거

- Opus 4.6은 스프린트 분해 없이 2시간 이상 일관되게 작업할 수 있었다.
- 에이전트들이 하나의 연속 세션으로 실행되었고, 자동 컴팩션이 컨텍스트 증가를 처리했다.
- 평가자가 스프린트별 대신 실행 끝에 단일 패스로 이동했다.
- 플래너와 평가자는 여전히 명확한 가치를 제공하므로 유지했다.

### 평가자가 중요한 시점

평가자의 가치는 모델 능력에 상대적이다:
- Sonnet 4.5에서: 빌드가 생성자가 단독으로 잘 해낼 수 있는 한계에 있었다 → 평가자가 일관되게 의미 있는 문제를 잡았다.
- Opus 4.6에서: 모델의 기본 능력이 증가하여 경계가 바깥으로 이동했다 → 이전에 평가자가 필요했던 태스크가 이제 단독으로 잘 처리되었다. 하지만 여전히 생성자 능력의 한계에 있는 태스크에서는 평가자가 실질적인 향상을 제공했다.

**핵심 교훈:** 평가자는 고정된 예/아니오 결정이 아니다. 태스크가 현재 모델이 단독으로 안정적으로 해내는 범위를 초과할 때 비용 대비 가치가 있다.

### V2 테스트: 브라우저 기반 DAW (디지털 오디오 워크스테이션)

**프롬프트:** "Web Audio API를 사용하여 브라우저에서 완전한 기능을 갖춘 DAW를 만들어라."

| 빌드 (라운드 1) | 2시간 7분 | $71.08 |

**QA 발견 (라운드 1):** 디자인과 AI 통합이 우수한 강력한 앱이지만, 여러 핵심 DAW 기능이 인터랙티브 깊이 없이 디스플레이만 되고 있었다 — 클립을 타임라인에서 드래그할 수 없고, 악기 UI 패널(신스 노브, 드럼 패드)이 없고, 시각적 이펙트 에디터(EQ 커브, 컴프레서 미터)가 없었다.

**QA 발견 (라운드 2):** 오디오 녹음이 여전히 스텁 상태, 클립 크기 조정/분할 미구현, 이펙트 시각화가 그래픽이 아닌 숫자 슬라이더.

**최종 결과:**
- 브라우저에서 작동하는 어레인지먼트 뷰, 믹서, 트랜스포트.
- 저자가 프롬프팅만으로 짧은 곡 스니펫을 작곡했다 — AI 에이전트가 템포/키 설정, 멜로디 배치, 드럼 트랙 구축, 믹서 레벨 조정, 리버브 추가를 수행했다.
- 전문 DAW에는 미치지 못하고, Claude가 들을 수 없어 음악적 취향에 대한 QA 효과가 제한적이지만, 작곡을 위한 모든 핵심 프리미티브가 존재했고 에이전트가 이를 자율적으로 구동할 수 있었다.

## 교훈 및 핵심 요약

1. **생성과 평가를 분리하라.** 독립 평가자를 회의적으로 튜닝하는 것이 생성자를 자기 비판적으로 만드는 것보다 훨씬 쉽다.

2. **주관적 품질도 채점 가능하게 만들 수 있다.** "이것이 아름다운가?"에는 일관되게 답하기 어렵지만, "구체적인 디자인 원칙을 따르는가?"는 모델에게 실행 가능한 채점 기준을 제공한다.

3. **하네스의 모든 컴포넌트는 모델 한계에 대한 가정이다.** 이 가정들은 모델이 개선됨에 따라 낡을 수 있으므로 정기적으로 스트레스 테스트하라.

4. **새 모델이 출시되면 하네스를 재검토하라.** 더 이상 성능에 기여하지 않는 부분을 걷어내고, 이전에는 불가능했던 새로운 능력을 달성하기 위한 부분을 추가하라.

5. **모델이 개선됨에 따라 흥미로운 하네스 설계의 공간은 줄어들지 않고 이동한다.** AI 엔지니어에게 흥미로운 작업은 다음 새로운 조합을 계속 찾아가는 것이다.

6. **에이전트 트레이스를 읽는 데 투자하라.** 평가자 튜닝 루프(로그 읽기 → 판단 차이 찾기 → 프롬프트 업데이트)는 필수적이며 지름길이 없다.

7. **플래너 에이전트는 뛰어난 모델에서도 가치를 더한다.** 플래너 없이는 생성자가 범위를 축소하여 덜 기능이 풍부한 애플리케이션을 만들었다.
