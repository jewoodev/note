## 1. 시작하기 — 해상도와 스케일링

클릭 정확도는 컴퓨터 사용 기능 통합의 가장 기본이 되는 요소다. 가장 효과가 큰 최적화는 **스크린샷을 API로 보내기 전에 다운스케일하는 것**.

### 모델별 입력 이미지 제한
- **Claude 4.6 계열**: 긴 변 최대 1568px, 총 1.15MP
- **Claude Opus 4.7**: 긴 변 최대 2576px, 총 3.75MP

### 권장 해상도
- **4.6 계열**: 1280×720 을 안전한 기본값으로
- **Opus 4.7**: 1080p 권장
- 종횡비를 유지하면서 모델 한계에 맞게 줄여주는 "max API fit" 헬퍼 사용을 권장

### 피해야 할 패턴
- 원본 해상도를 축소 없이 전송
- 960×540 이하의 지나치게 낮은 해상도
- 4.6 계열에 1920×1080 이상의 이미지 전송
- macOS에서 디바이스 픽셀 비율 2배가 적용된 그대로의 스크린샷

## 2. 좌표 스케일링

API는 모델이 본 이미지 기준으로 좌표를 돌려주므로, 실제 화면 좌표로 다시 환산해야 한다.

```python
scale_x = screen_w / display_w
scale_y = screen_h / display_h
screen_x = int(api_returned_x * scale_x)
screen_y = int(api_returned_y * scale_y)
```

다운스케일된 이미지를 보낼 때는 이 변환을 빼먹지 않는 것이 중요하다.

---

## 3. 메시지 내 콘텐츠 순서

같은 메시지 안에서 **텍스트 지시 → 이미지** 순서로 배치하면 클릭 정확도가 올라간다. 모델이 스크린샷을 처리하기 전에 "무엇을 찾아야 하는지"를 먼저 알게 되기 때문이다.

## 4. 클릭 문제 진단 가이드

| 증상 | 원인 추정 | 해결 방향 |
|---|---|---|
| 한 방향으로 일정하게 오프셋 | 해상도 불일치 / API 한계 초과 | 1280×720 등으로 사전 다운스케일 |
| 근처는 가는데 자꾸 빗나감 | 타깃 UI 요소가 너무 작음 | zoom 활성화 또는 UI 확대 |
| 완전히 엉뚱한 요소 클릭 | 지시문이 모호함 | 더 구체적인 프롬프트 |
| 전반적으로 부정확 | 스크린샷 해상도 과다 | 다운스케일 또는 Opus 4.7 사용 |

## 5. 모델 선택

- **Sonnet 4.6** — 정확도/비용 균형이 좋은 기본 선택지
- **Opus 4.7** — 강한 추론이 필요하면서 정밀 클릭도 요구되는 작업
- **Haiku 4.5** — 지연 시간이 최우선인 경우

---

## 6. 작은 타깃 다루기

- `enable_zoom: True` 로 zoom 기능 활성화
- 가능하다면 UI 요소 크기를 키움
- 작은 요소는 키보드 단축키 등 대안 경로로 우회
- 4K 디스플레이 환경이라면 Opus 4.7 권장

## 7. Thinking / Effort 레벨 권장값

### Opus 4.7
- 기본: `high`
- 비용 민감: `low`
- 복잡한 일회성 작업: `max`

### 4.6 계열
- 기본: `medium` (정확도/비용비 최적)
- 고처리량 환경: `low`
- 단순 워크플로우: thinking 비활성화

## 8. 프롬프트 인젝션 방어

Claude는 다층 방어 구조를 갖춤.
- 학습 시점의 견고성 강화 (RL 포함)
- 실시간 분류기 (자동·무료·저지연)
- 지속적인 레드팀 평가

공식 도구(`computer_20251124`)를 사용하면 이러한 보호가 자동 적용된다.

### 운영 시 권장 사항
- 민감한 작업에는 사람의 승인을 끼워 넣기
- 에이전트 권한 최소화
- 모든 액션 로깅
- 웹에서 들어오는 콘텐츠는 **신뢰할 수 없는 입력**으로 취급

---

## 9. 컨텍스트 관리 (장기 실행 에이전트)

### 캐시 컨트롤 배치 권장
- 시스템 프롬프트 1개
- 최근 도구 결과에 최대 3개

### 접근 1 — 롤링 버퍼
- 초기값 예시: `keep_n = 3`, `interval = 25`
- 오래된 스크린샷은 **배치로** 제거해 캐시 일관성을 유지
- 단점: 버퍼 범위를 벗어난 맥락이 사라짐

### 접근 2 — LLM 기반 압축
- 요약 프롬프트로 대화를 압축
- 사용자 원본 지시문은 그대로 보존
- 작업 템플릿, 제약 조건, 진행 상태를 요약에 포함

### 서버 측 압축 (베타)
```python
context_management = {
    "edits": [
        {
            "type": "compact_20260112",
            "trigger": {"type": "input_tokens", "value": 150_000},
            "instructions": COMPACT_PROMPT,
        }
    ]
}
```

## 10. 실험적 설정 (프롬프트 인젝션 분류기를 끄는 경우)

### 배치 도구
- 여러 액션을 한 번의 도구 호출로 묶음
- 적합: 폼 자동 입력, 정해진 키보드 시퀀스
- 부적합: 매 단계 시각 상태 확인이 필요한 탐색

### 어드바이저 도구 (베타)
- Opus 4.7이 실행자 모델에 조언을 제공
- 장시간 작업 중 추론이 중요한 순간에만 비용 발생
- `max_uses` 로 호출 횟수 제한 가능

## 11. Teach Mode — 시연으로 신뢰성 끌어올리기

사용자가 직접 작업을 수행하는 과정을 기록해 Claude의 가이드로 활용한다.

### 워크플로우 스텝 데이터 모델
- 액션 종류 (클릭, 입력, 이동, 스크롤, 선택)
- 셀렉터와 좌표
- 스크린샷 (클릭 위치에 파란 원으로 주석)
- (옵션) 음성 트랜스크립트

### 기록 단계
- 각 액션마다 시연 컨텍스트, 설명, 주석 스크린샷을 저장

### 재생 단계
- 모델이 시연 스크린샷을 참고하며 같은 순서를 따라감
- 필요 시 현재 UI에 맞게 조정

### 재생 모드
- **엄격(strict)**: 시연을 그대로 따름
- **적응형(adaptive)**: UI 변경에 유연하게 대응 — 기본값
- **목표 지향(goal-oriented)**: 결과만 같으면 OK

## 핵심 정리

1. **해상도가 1순위** — 사전 다운스케일이 가장 큰 효과
2. **좌표 스케일링 필수** — API 좌표를 실제 화면 좌표로 환산
3. **텍스트 먼저, 이미지 나중** — 메시지 콘텐츠 순서 주의
4. **프롬프트 캐싱** — 반복 호출 비용 절감
5. **장기 작업은 컨텍스트 압축** — 롤링 버퍼 + LLM 요약 조합
6. **보안은 다층 방어** — 공식 도구 + 휴먼 게이트 + 권한 제한 + 로깅
