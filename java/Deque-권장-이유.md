Java에서 `Stack`과 `Queue` 대신 `Deque`(보통 `ArrayDeque`)를 쓰는 게 성능상 유리한 이유는 내부 구현 차이에서 비롯됩니다.

# Stack의 문제: Vector 기반의 동기화 오버헤드
`Stack`은 `Vector`를 상속합니다. `Vector`는 모든 메서드에 `synchronized`가 걸려 있어서, 단일 스레드 환경에서도 매번 모니터 락을 획득/해제하는 비용이 발생합니다. 실제로 멀티스레드에서 스택이 필요한 경우는 드물고, 필요하더라도 `ConcurrentLinkedDeque` 같은 전용 자료구조를 쓰는 게 맞기 때문에, `Stack`의 동기화는 사실상 불필요한 오버헤드입니다.

또한 `Vector`는 내부적으로 배열 기반이긴 하지만, 용량 초과 시 **2배**로 증가하는 전략을 씁니다. `ArrayDeque`도 2배 증가이긴 한데, `Stack`은 여기에 동기화 비용이 더해지니 전체적으로 느립니다.

# Queue(LinkedList)의 문제: 노드 할당과 캐시 미스
`Queue` 인터페이스의 대표적 구현체인 `LinkedList`는 이중 연결 리스트입니다. 매 `offer`/`poll`마다 `Node` 객체를 힙에 할당하고, GC 대상으로 남깁니다. 이로 인해 두 가지 문제가 생깁니다.

첫째, **객체 할당/GC 비용**입니다. 원소 하나 넣을 때마다 `Node` 객체가 생성되고, 뺄 때마다 GC 대상이 됩니다. 대량 연산 시 GC 압력이 커집니다.

둘째, **CPU 캐시 지역성(locality) 저하**입니다. 연결 리스트의 노드들은 힙 곳곳에 흩어져 있어서, 순차 접근 시 캐시 라인을 제대로 활용하지 못합니다. 반면 배열 기반 구조는 연속된 메모리에 원소가 모여 있어 캐시 히트율이 훨씬 높습니다.

# ArrayDeque가 빠른 이유
`ArrayDeque`는 **원형 배열(circular array)** 기반입니다. `head`와 `tail` 인덱스만 이동시키면서 양쪽 끝에서 O(1)으로 삽입/삭제합니다. 핵심 이점을 정리하면:

- **동기화 없음** → 락 오버헤드 제로
- **노드 객체 할당 없음** → GC 압력 최소화
- **연속 메모리 배치** → CPU 캐시 지역성 우수
- **resize도 효율적** → 용량 초과 시 배열 복사 한 번으로 끝

실제로 Java 공식 문서에서도 스택으로 쓸 때 `Stack`보다, 큐로 쓸 때 `LinkedList`보다 `ArrayDeque`를 권장하고 있습니다.

**한 가지 주의점**은, `ArrayDeque`는 `null` 원소를 허용하지 않는다는 점입니다. `null`을 넣어야 하는 특수한 경우라면 `LinkedList`를 써야 하지만, 일반적인 스택/큐 용도에서는 `ArrayDeque`가 거의 항상 더 나은 선택입니다.
