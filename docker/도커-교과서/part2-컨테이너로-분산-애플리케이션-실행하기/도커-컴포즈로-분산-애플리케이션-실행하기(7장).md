대부분의 애플리케이션은 여러 개의 요소로 구성된다. 그리고 도커는 n-티어 모놀리식 설계부터 현대적 마이크로서비스 설계까지, 분산된 컴포넌트를 실행하기에 이상적인 환경이다. 각 컴포넌트는 자신만의 경량 컨테이너에서 실행되며 도커가 표준 네트워크 프로토콜을 통해 컨테이너들을 엮어낸다. 도커 컴포즈를 사용하면 여러 컨테이너에 걸쳐 실행되는 애플리케이션을 정의하고 관리할 수 있다.

# 도커 컴포즈 파일의 구조
Dockerfile 스크립트는 애플리케이션을 패키징하기 위한 스크립트이다. 해당 스크립트는 그 역할에만 충실하기때문에 여러 컴포넌트에 걸쳐 실행될 애플리케니션을 패키징하려면 각 컴포넌트에 하나씩 세 개의 Dockerfile 스크립트가 필요하다. 그럼 이 컨테이너들은 누가 실행할까?

직접 순서대로 도커 명령행을 통해 일일이 지정해가며 실행할 수도 있지만 그런 수동적인 프로세스는 온갖 실수와 오류의 원인이 된다. 대신 도커 컴포즈 파일에 애플리케이션의 구조를 정의하면 된다.

도커 컴포즈 파일은 애플리케이션의 **원하는 상태**, 모든 컴포넌트가 실행 중일 때 어떤 상태여야 하는지를 기술하는 파일이다. 그리고, `docker container run` 명령을 사용할 때 지정하는 모든 옵션들을 모아 놓은 단순한 형식의 파일이다. 도커 컴포즈 도구를 사용해 애플리케이션을 실행하면 도커 컴포즈가 컨테이너, 네트워크, 볼륨 등 필요한 모든 도커 객체를 만들도록 도커 API에 명령을 내린다.

이 파일에 애플리케이션의 모든 실행 옵션이 기술되기 때문에 README 파일에 애플리케이션 이미지 이름이나 공개해야 할 포트 번호를 문서화할 필요가 아예 없다.

# 도커 컴포즈를 사용해 여러 컨테이너로 구성된 애플리케이션 실행하기
도커 컴포즈에 기술되는 서비스의 옵션으로 `depends_on` 을 설정하면 해당 서비스가 의존하는 다른 서비스들을 지정할 수 있다. 그럼 도커 컴포즈는 의존성을 만족시키기 위해 `depends_on`에 나열된 서비스들을 먼저 실행하려 시도하게 된다. 

도커 컴포즈를 사용하면 특정 컨테이너(API 서비스, stateless)를 스케일 아웃하는 것도 가능하다. 스케일링하기만 하면 도커가 여러 개의 API 컨테이너에 요청을 고르게 분배해준다. 

```
docker compose up -d --scale api=3

docker compose logs --tail=1 api
```

도커 컴포즈 명령은 모든 컨테이너 혹은 원하는 컨테이너의 로그만 골라 출력하게 만들 수 있다. `--tail 1` 파라미터는 각 API 컨테이너의 마지막 로그를 출력하라는 의미이다. 

도커 컴포즈가 우리를 대신해 컨테이너를 관리해준다. 우리가 도커 컴포즈에 의존할 수 밖에 없는 것은 아니다. 컴퓨팅 자원을 절약하기 위해 모든 컨테이너를 중지시킬 수도 있고 다시 컨테이너를 시작해 애플리케이션을 재가동할 수도 있다. 도커 컴포즈는 컨테이너를 관리하는 별도의 명령이지만 내부적으로는 마찬가지로 도커 API를 사용한다. 그렇기에 도커 컴포즈로 실행한 컨테이너라도 똑같이 도커 명령행으로 관리할 수 있다.

도커 컴포즈는 다양한 기능이 있지만 그런 기능을 알기 전에 미리 알아야 할 것이 있다. 바로 도커 컴포즈가 클라이언트 측에서 동작하는 도구라는 것이다. 도커 컴포즈 명령을 실행하면 도커가 컴포즈 파일의 내용에 따라 도커 API로 지시를 보낸다. 도커 엔진 자체는 컨테이너를 실행할 뿐, 여러 컨테이너가 하나의 애플리케이션으로 동작하는지는 모른다. 그걸 아는 건 컴포즈 파일을 읽어 애플리케이션 구조를 이해한 컴포즈 뿐이다. 그러니까 컴포즈를 사용해 애플리케이션을 관리하려면 컴포즈 파일을 작성하고 해당 파일을 읽을 수 있게 해야 한다. 이 말이 무슨 말이냐면, 컴포즈 파일의 수정 없이 애플리케이션을 관리하지 말라는 말이다. 

컴포즈 파일에 `external` 플래그가 붙지 않으면 네트워크와 볼륨도 제거 대상이 된다. 

# 도커 컨테이너 간의 통신
컴포즈를 통해 분산 애플리케이션을 구성하는 컨테이너들은 서로 어떻게 통신할까? 컨테이너는 별도의 네트워크 공간을 가진 가상환경이다. 컨테이너는 도커 엔진으로부터 부여받은 자신만의 가상 IP 주소를 가지며 모두 같은 도커 네트워크로 연결돼 이 IP주소를 통해 서로 통신할 수 있다. 그런데 애플리케이션 생애주기 동안에 컨테이너가 교체되면 IP주소도 변경된다. 도커는 DNS를 이용한 서비스 디스커버리 기능을 제공해 IP주소가 변경돼도 문제가 없는 환경을 제공한다.

도커 컴포즈는 컨테이너를 실행할 때 지정한 모든 옵션값을 기억하고, 그 옵션값들을 통해 컨테이너 간의 통신을 처리한다.

# 도커 컴포즈로 애플리케이션 설정값 지정하기

```
services:
  example:
    image:jewoo/example
    environment:
        - Database:Provider=Postgres
    secrets:
        - source: postgres-connection
        target: /app/config/secrets.json
        
secrets:
  postgres-connection:
    file: ./config/secrets.json
```


- `environment` 플래그를 사용해 컨테이너 안에서 사용할 환경 변수 값을 정의할 수 있다.
- `secret` 플래그를 사용해 컨테이너 내부의 파일에 기록될 비밀값을 정의할 수 있다. 이 애플리케이션이 실행되면 컨테이너에 `/app/config/secrets.json` 파일이 생기고 해당 파일에는 `postgres-connection` 이라는 이름의 비밀값의 값이 기록된다.


패키징된 애플리케이션과 설정값을 분리할 수 있다는 것도 도커의 핵심적인 장점 중 하나다. 애플리케이션 이미지는 빌드 파이프라인을 통해 만들어지고 다시 테스트 환경을 거치며 운영 환경에 적합한지 검증된다. 그런 과정 속에서 컴포즈 파일에서 쉽게 설정할 수 있는 환경 변수나 바인드 마운트 설정, 비밀값 등이 사용되어 설정값이 적용될 수 있다. 

# 도커 컴포즈도 만능은 아니다
도커 컨테이너를 본격적으로 사용하려면 도커 컴포즈가 매우 중요한 도구가 될 것이다. 하지만 그 전에 도커 컴포즈의 목적은 무엇이고 어떤 기능과 어떤 제한 사항이 있는지를 잘 이해해야 한다. 

컴포즈를 사용하면 애플리케이션을 정의하고 이 정의를 도커 엔진을 실행 중인 단일 컴퓨터에 적용할 수 있다. 이 컴퓨터에 실제 활성 상태인 도커 리소스를 파악하고 컴포즈 파일에 기재된 리소스와 비교해 추가로 필요한 리소스를 도커 API를 통해 요청해서 수정하거나 생성한다.

`docker compose up` 명령을 실행하기만 하면 우리가 정의한 상태대로 애플리케이션을 실행할 수 있다. 하지만 도커 컴포즈가 할 수 있는 일은 여기까지다. 도커 컴포즈는 도커 스웜이나 쿠버네티스같은 완전한 컨테이너 플랫폼이 아니다. 도커 컴포즈에는 그러한 것들과 달리 애플리케이션을 지속적으로 정의된 상태를 유지하도록 하는 기능이 없다. 일부 컨테이너가 오류를 일으키거나 강제로 종료되더라도 `docker compose up` 명령을 다시 실행하지 않는 한 애플리케이션의 상태를 원래대로 되돌릴 수 없다.

운영 환경에 도커 컴포즈가 부적합하다는 말은 아니다. 비록 고가용성, 로드 밸런싱, 이중화 같은 기능은 갖추지 못하겠지만, 그런 기능은 애초에 가상 머신에서도 없던 기능이다. 하지만 애플리케이션 전체를 일관적인 요스들의 조합으로 만들 수 있다. 애플리케이션의 모든 구성 요소가 각자의 Dockerfile 스크립트와 컴포즈 파일을 갖게 되고, 항상 동일한 도구로 애플리케이션을 배포하고 관리할 수 있다. 컨테이너 클러스터를 운영할 계획이 없는 한 도커 컴포즈로도 충분하다.

https://docs.docker.com/compose/compose-file 에서 도커 컴포즈 파일의 상세 규격을 볼 수 있다.
