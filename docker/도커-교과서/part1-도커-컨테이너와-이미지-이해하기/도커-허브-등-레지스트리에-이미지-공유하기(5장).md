이번엔 도커의 빌드와 실행이 아닌 공유에 대해 배워보자. 도커의 여러 장점 중에서도 바로 이 공유가 핵심에 해당한다고 생각한다. 우리가 개발한 소프트웨어를 모든 의존 모듈과 함께 패키징한다면 누가 어떤 환경에서 시도하더라도 쉽게 실행할 수 있을 것이다. 환경의 차이를 없애버리기 때문에 환경 설정이나 배포로 인해 생겨나는 문제를 해결하기 위한 시간도 줄일 수 있다.

# 레지스트리, 리포지토리, 이미지 태그
도커 이미지에는 이름이 부여되는데, 이 이름에 해당 이미지를 내려받기 위해 필요한 모든 정보가 들어 있다. 이미지의 전체 이름은 네 개의 요소로 구성된다. 

```
                   이미지 작성자의 계정 이름.
                   개인 혹은 단체의 이름에 해당한다.   이미지 태그. 애플리케이션의 버전 혹은
                                  |               변종을 나타낸다. 기본값은 latest다.
                                __|_          ___|_ 
                               |    |        |     |
                    docker.io/library/ubuntu:latest
                    |_______|         |____|  
                 _______|                |
                 |                     이미지 리포지토리의 이름. 일반적으로 
                 |                     애플리케이션의 이름에 해당한다.
이미지에 저장된 레지스트리의               하나의 리포지터리는 여러 버전의 이미지를
도메인. 기본값은 도커 허브다.              담을 수 있다.
```

직접 개발한 애플리케이션의 이미지를 관리하려면 이미지 참조의 모든 구성 요소를 다 사용해야 한다. 로컬 컴퓨터에서만 사용한다면 이미지 이름을 맘대로 지어도 문제없지만, 레지스트리를 통해 다른 사람이 이미지를 사용하게 하려면 좀 더 상세한 정보를 이미지 이름에 포함시켜야 한다. 그 이유는 이미지 참조가 레지스트리에서 특정한 이미지를 식별하는 식별자 역할을 하기 때문이다.

레지스트리와 태그 등의 정보는 따로 지정하지 않아도 도커가 이미 정해둔 기본값이 사용된다. 

# 도커 허브에 직접 빌드한 이미지 푸시하기
이미지의 참조는 변경되기 위해 재빌드가 필요하진 않다. 그리고 이미지는 여러 개의 참조를 가질 수 있다. 

```
docker image tag ubuntu $dockerId/ubuntu:v1
```

위 명령어로 태그와 작성자 정보가 없는 이미지의 새로운 참조를 생성할 수 있다. 이제 ubuntu 이미지는 계정 이름과 버전이 지정된 참조를 갖는다. 이미지는 유일 식별자도 갖는다. 유일 식별자를 이용하면 여러 개의 이미지 참조가 같은 이미지를 가리키고 있는지도 알 수 있다.

```
docker image ls --filter reference=ubuntu --filter reference='*/ubuntu'
```

위 명령어를 통해 보여지는 ubuntu 이미지들의 ID 값이 유일 식별자이다. 

여기까지 수행했다면 도커 허브 계정 이름이 포함된 이미지 참조가 만들어졌을 것이다. 이제 `pull` 명령과 반대되는 기능을 가진 `docker image push` 명령을 사용해 로컬 컴퓨터에 저장된 이미지 레이어를 레지스트리로 푸시해보자. 아래의 명령어를 통해 이미지를 푸시할 수 있다.

```
docker image push $dockerId/ubuntu:v1
```

도커 레지스트리도 로컬 컴퓨터에서 동작하는 도커 엔진과 같은 방식으로 이미지 레이어를 다룬다. 즉, 이미지 푸시에서 실제로 업로드 대상은 이미지 레이어다. 

레지스트리 역시 도커 엔진과 같은 방식으로 이미지 레이어를 다루기 때문에 그만큼 Dockerfile 스크립트의 최적화가 더욱 중요해진다. 하지만 완전하게 동일하진 않은데, 레지스트리 상의 전체 이미지를 대상으로 한다는 점이 다르다. 레이어의 90%가 기존 캐시를 재사용할 수 있도록 이미지를 최적화했다면, 이 이미지를 푸시할 때 90%의 레이어는 레지스트리의 캐시를 재사용할 수 있다. 이런 이유들때문에 최적화된 Dockerfile 스크립트는 빌드 시간, 디스크 용량을 넘어 네트워크 대역폭까지 영향을 끼치는 중요한 요소다.

도커 허브는 레지스트리 중에서도 가장 접근성이 좋고 거의 제로에 가까운 비용으로 다양한 기능을 제공한다. 

레지스트리는 개방형 API 명세다. 또 코어 레지스트리 서버도 도커에서 오픈 소스 형태로 제공한다. 모든 클라우드 서비스는 자체 레지스트리 서비스를 제공하므로, 데이터 센터에서 도커 레지스티드 레지스트리 등의 제품을 이용해 직접 레지스트리를 운영할 수 있다. 아니면 단군 컨테이너 형태의 레지스트리를 운영하는 것도 가능하다.

# 공식 이미지에서 골든 이미지로 전환하기
골든 이미지는 공식 이미지를 기반 이미지로 삼아 인증서나 환경 설정값 등 별도의 설정을 추가한 것이다. 골든 이미지는 도커 허브의 기업 리포지토리나 자체 리포지토리에서 관리된다. 그리고 해당 기업의 모든 애플리케이션 이미지는 바로 이 골든 이미지를 기반으로 한다. 이런 방식은 프로젝트 개발 팀과 도커가 함께 최적화한 공식 이미지의 이점을 그대로 누리면서 필요한 설정을 추가할 수 있다는 장점이 있다.
