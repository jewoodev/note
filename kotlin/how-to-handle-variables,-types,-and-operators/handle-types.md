# 기본 타입? 래퍼 타입?
- 코틀린은 자바와 다르게 기본 타입과 래퍼 타입을 구분짓지 않는다.
- Int, Long 등등 모두 int, long 같은 별도의 타입 없이 기본 타입으로 제공된다.
- 이는 자바 언어를 바라보던 시선으로 바라봤을 때 래퍼 타입 같고, 따라서 연산 성능에 불이익이 있을 것 같지만 코틀린이 내부적으로 효율적으로 연산을 처리하기 때문에 그런 문제는 고려하지 않아도 된다.

# 명시적인 타입 변환
- 코틀린에선 암시적 타임 변환이 지원되지 않는다. 
- 매번 명시적인 타입 변환을 하는 것을 자바 코드를 떠올려보면 굉장히 불편할 것 같지만 다양한 `to~()` 메서드가 제공되어 그런 불편함은 거의 없다.
- 일반 타입을 명시적으로 변환할 때는 `as` 연산자를 사용한다.
  - `as` 연산자를 사용할 때 null이 오게되면 NPE가 발생하므로, null이 올 수 있으면 `as!` 를 사용해야 한다.
- `as?`
  - 타입이 맞으면 타입 캐스팅
  - 값이 null이면 연산 자체를 null로 대체
  - 타입이 아니면 null로 대체

## 스마트 캐스트
- 코틀린 컴파일러가 컨텍스트를 분석해 if 문 내에서 타입 체크를 한 것이 확인되면, "이 타입으로 간주될 수 있겠구나" 라고 인지하고 있다.
  - 따라서 `as` 연산자를 생략해도 컴파일러가 타입을 안전하게 추론할 수 있다.

# 타입 비교
- 코틀린에서는 타입 비교를 위해 `is`와 `!is` 연산자를 사용한다.
  - `is` 연산자는 객체가 특정 타입의 인스턴스인지 확인한다.   
  - `!is` 연산자는 객체가 특정 타입의 인스턴스가 아닌지 확인한다.
  - 이는 `instance of`를 대신하는 연산자다.

# Kotlin에만 존재하는 타입
## Any
- Java의 Object 역할, 즉 모든 클래스의 최상위 클래스
  - 모든 기본 타입의 최상위 타입도 Any
- Any 자체로는 null을 포함할 수 없음. null을 포함하고 싶다면 Any?로 표현
- Any에 `equals()`, `hashCode()`, `toString()`이 존재함

## Unit
- Java의 void와 동일한 역할.
- 타입 추론이 가능하기 때문에 생략 가능
- void와는 다르게 그 자체로 타입 인지로써 사용될 수 있다.
- 함수형 프로그래밍에서 Unit은 단 하나의 인스턴스만 갖는 타입을 의미.
  - 코틀린의 Unit은 실제 존재하는 타입이라는 것을 표현.

## Nothing
- 함수가 정상적으로 끝나지 않았다는 사실을 표현하는 역할
- 무조건 예외를 반환하는 함수 / 무한 루프 함수 등

# String interpolation / String indexing
자바에선 `String.format()` 을 사용해 문자열 안의 내용을 대체해야 했지만 코틀린에서는 interpolation 기능이 제공되어 `"나는 ${user.name} 입니다."` 처럼 대체할 수 있다. 중괄호를 생략하고 `$변수` 를 사용할 수도 있다.

자바에서는 `charAt()` 메소드를 사용해야 했지만 코틀린에서는 `str[1]` 로 특정 문자를 가져올 수 있다.
