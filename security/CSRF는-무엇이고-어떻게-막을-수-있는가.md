- CSRF는 Cross-Site Request Forgery의 약자로, 사용자가 인증된 상태(세션/쿠키)를 악용해서 사용자의 의도와 무관한 요청을 서버에 보내는 공격
  - 예를 들어, 사용자가 은행 사이트에 로그인한 상태에서 공격자가 만든 페이지를 방문하면, 그 페이지에서 은행 서버로 송금 요청을 보낼 수 있음 
    - 브라우저가 쿠키를 자동으로 포함시키기 때문에 서버 입장에서는 정상 요청과 구분이 안 됨
- 대표적인 방어 방법은 세 가지
  - 첫째, CSRF 토큰 방식
    - 서버가 폼 렌더링 시 랜덤 토큰을 발급하고, 요청 시 이 토큰을 함께 보내도록 해서 토큰이 없거나 불일치하면 요청을 거부
    - Spring Security에서는 `CsrfFilter`가 기본적으로 활성화되어 있고, `CsrfTokenRepository`를 통해 토큰을 관리합니다.
  - 둘째, SameSite 쿠키 속성
    - 쿠키에 `SameSite=Strict` 또는 `Lax`를 설정하면, 외부 사이트에서 발생한 요청에 쿠키가 포함되지 않아 근본적으로 차단됨
  - 셋째, Referer/Origin 헤더 검증
    - 요청의 출처를 확인해서 자사 도메인이 아닌 곳에서 온 요청을 차단
- 다만 REST API 서버에서 인증을 JWT 같은 토큰 기반으로 처리하고 쿠키를 사용하지 않는다면, 
  - 브라우저가 자동으로 인증 정보를 포함시키는 구조가 아니기 때문에 CSRF 공격 자체가 성립하지 않음
  - 그래서 Spring Security에서도 이런 경우 csrf.disable()을 하는 것이 일반적
  - 반대로 세션 기반 인증이나 쿠키에 JWT를 담는 구조라면 CSRF 방어가 반드시 필요

### 꼬리질문 1 - CSRF 토큰은 어디에 저장되나요?
- Spring Security 기본값은 HttpSessionCsrfTokenRepository(세션 저장)
- SPA 구조에서는 CookieCsrfTokenRepository를 써서 쿠키로 내려주고 JS에서 읽어 헤더에 실어 보내는 방식을 씀

### 꼬리질문 2 - XSS와 CSRF의 차이는?
- XSS는 악성 스크립트를 사이트 내에 삽입해서 사용자의 브라우저에서 실행시키는 공격
- CSRF는 사용자의 인증 상태를 이용해 위조 요청을 보내는 공격
- XSS는 "사이트를 신뢰하는 사용자"를 공격
- CSRF는 "사용자를 신뢰하는 서버"를 공격한다고 구분할 수 있음

### 꼬리지문 3 - CORS와 CSRF의 관계는?
- CORS는 브라우저의 Same-Origin Policy를 완화하는 메커니즘이고, CSRF 방어와는 별개
- CORS를 엄격하게 설정해도 <form> 태그를 이용한 POST 요청은 CORS 정책을 타지 않기 때문에 CSRF를 완전히 막지 못함
