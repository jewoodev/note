# 최소 신장 트리(MST) — Kruskal과 Prim으로 같은 문제를 두 번 풀어보기

LeetCode [1584. Min Cost to Connect All Points](https://leetcode.com/problems/min-cost-to-connect-all-points/) 문제를 Kruskal과 Prim 두 가지 방법으로 풀어보면서, MST의 핵심 개념과 각 알고리즘의 차이를 정리했다.

---

## 문제 요약

2D 평면 위의 점들이 주어진다. 두 점을 연결하는 비용은 맨해튼 거리 `|xi - xj| + |yi - yj|`이고, **모든 점을 연결하는 최소 비용**을 구해야 한다. 단, 임의의 두 점 사이에 경로가 정확히 하나만 존재해야 한다.

## 이 문제가 MST인 이유

조건을 하나씩 뜯어보면 자연스럽게 MST로 귀결된다.

- "모든 점을 연결" → 그래프의 모든 노드를 포함
- "비용의 합을 최소화" → 간선 가중치 합 최소
- "경로가 정확히 하나" → 사이클이 없는 연결 그래프 → **트리**

이 세 가지를 합치면 **최소 신장 트리(Minimum Spanning Tree)**를 구하는 문제다.

> **신장 트리(Spanning Tree)**: N개 노드를 N-1개 간선으로 사이클 없이 연결한 그래프.
> **최소 신장 트리**: 가능한 신장 트리 중 간선 가중치 합이 가장 작은 것.

## 풀이 1: Kruskal 알고리즘

### 핵심 아이디어

**Greedy**다. 가장 싼 간선부터 하나씩 골라가되, 사이클이 생기는 간선은 버린다.

1. 모든 간선을 가중치 오름차순으로 정렬
2. 가장 작은 간선부터 꺼내서, 사이클이 아니면 채택
3. N-1개 간선을 채택하면 완성

사이클 판별에는 **Union-Find**를 사용한다.

### Union-Find란?

"두 원소가 같은 그룹인가?"를 빠르게 판별하는 자료구조다. 연산은 두 가지뿐이다.

- **Find(x)**: x가 속한 그룹의 루트를 찾는다
- **Union(x, y)**: x의 그룹과 y의 그룹을 합친다

Kruskal에서 간선 `(u, v)`를 꺼냈을 때:
- `find(u) == find(v)` → 이미 같은 그룹 → 스킵 (사이클 방지)
- `find(u) != find(v)` → 다른 그룹 → union하고 채택

**경로 압축(Path Compression)**을 적용하면 find 연산이 거의 O(1)에 수렴한다. 루트를 찾아 올라가면서 경로 위의 모든 노드가 루트를 직접 가리키게 만드는 기법이다.

### 구현

```java
class Solution {
    public int minCostConnectPoints(int[][] points) {
        int[] group = new int[points.length];
        for (int i = 0; i < points.length; i++) {
            group[i] = i;
        }

        var pq = new PriorityQueue<Edge>((e1, e2) -> e1.cost - e2.cost);
        for (int i = 0; i < points.length; i++) {
            for (int j = i + 1; j < points.length; j++) {
                pq.offer(new Edge(
                    i, j,
                    Math.abs(points[i][0] - points[j][0]) + Math.abs(points[i][1] - points[j][1])
                ));
            }
        }

        int answer = 0;
        while (!pq.isEmpty()) {
            var edge = pq.poll();
            int rootFrom = find(group, edge.from);
            int rootTo = find(group, edge.to);
            if (rootFrom != rootTo) {
                union(group, rootFrom, rootTo);
                answer += edge.cost;
            }
        }

        return answer;
    }

    private int find(int[] group, int x) {
        return group[x] == x ? x : (group[x] = find(group, group[x]));
    }

    private void union(int[] group, int x, int y) {
        group[y] = group[x];
    }

    private class Edge {
        int from, to, cost;
        Edge(int from, int to, int cost) {
            this.from = from;
            this.to = to;
            this.cost = cost;
        }
    }
}
```

### 동작 예시

점 4개 A, B, C, D가 있고 간선이 아래와 같다고 하자.

```
간선 (정렬 후): A-B:1, B-C:2, A-C:3, C-D:4, A-D:5, B-D:6

A-B(1) → A,B 다른 그룹 → 채택  cost=1
B-C(2) → B,C 다른 그룹 → 채택  cost=3
A-C(3) → A,C 같은 그룹 → 스킵 (사이클!)
C-D(4) → C,D 다른 그룹 → 채택  cost=7 → 완성
```

### 시간복잡도

- 간선 생성: O(N²)
- PriorityQueue 정렬: O(N² log N)
- Union-Find (경로 압축): 거의 O(1) per 연산

**전체: O(N² log N)**

## 풀이 2: Prim 알고리즘

### 핵심 아이디어

Kruskal이 "간선 중심"이라면 Prim은 **"정점 중심"**이다. 시작점에서 출발해서, 현재 트리에서 가장 가까운 점을 하나씩 추가한다.

1. 아무 점에서 시작 (예: 0번)
2. 트리에 포함된 점에서 나가는 간선 중 가장 비용이 작은 것을 선택
3. 그 간선이 연결하는 새 점을 트리에 추가
4. 모든 점이 트리에 포함되면 종료

구조가 **다익스트라와 거의 같다**. 차이는 딱 하나다.
- 다익스트라: "시작점에서 해당 노드까지의 **총 거리**" 기준
- Prim: "현재 트리에서 해당 노드까지의 **간선 하나의 거리**" 기준

### Kruskal과의 차이

Kruskal은 시작 전에 **모든 간선**을 미리 만들어 PQ에 넣는다. 반면 Prim은 트리에 점을 추가할 때마다 **그 점에서 나가는 간선만** PQ에 넣는다. 간선을 미리 다 만들 필요가 없다는 게 메모리 측면에서 유리할 수 있다.

### 구현

```java
class Solution {
    public int minCostConnectPoints(int[][] points) {
        var pq = new PriorityQueue<Edge>((e1, e2) -> e1.cost - e2.cost);
        boolean[] visited = new boolean[points.length];
        visited[0] = true;

        for (int i = 1; i < points.length; i++) {
            pq.offer(new Edge(
                i,
                Math.abs(points[0][0] - points[i][0]) + Math.abs(points[0][1] - points[i][1])
            ));
        }

        int answer = 0;
        int cnt = 0;
        int target = points.length - 1;

        while (cnt != target) {
            var edge = pq.poll();
            if (visited[edge.to]) continue;

            answer += edge.cost;
            cnt++;
            visited[edge.to] = true;

            for (int i = 0; i < points.length; i++) {
                if (!visited[i]) {
                    pq.offer(new Edge(
                        i,
                        Math.abs(points[edge.to][0] - points[i][0])
                            + Math.abs(points[edge.to][1] - points[i][1])
                    ));
                }
            }
        }

        return answer;
    }

    private class Edge {
        int to, cost;
        Edge(int to, int cost) {
            this.to = to;
            this.cost = cost;
        }
    }
}
```

### 동작 예시

점 4개(0, 1, 2, 3)가 있다고 하자.

```
시작: 0번을 트리에 추가
→ PQ에 0-1, 0-2, 0-3 간선 추가

PQ에서 꺼냄 → 0-1(비용 2) → 1번을 트리에 추가
→ PQ에 1-2, 1-3 추가

PQ에서 꺼냄 → 0-3(비용 3) → 3번을 트리에 추가
→ PQ에 3-2 추가

PQ에서 꺼냄 → 1-2(비용 4) → 2번을 트리에 추가
→ 모든 점 연결 완료!
```

### 시간복잡도

- 각 점을 트리에 추가할 때 최대 N개 간선 추가: O(N)
- N개 점에 대해 반복: O(N²)
- PQ 연산: O(N² log N)

**전체: O(N² log N)**

## 두 알고리즘 비교

|              | Kruskal               | Prim                       |
|--------------|-----------------------|----------------------------|
| **관점**       | 간선 중심                 | 정점 중심                      |
| **전략**       | 전체 간선을 정렬 후 사이클 없이 선택 | 시작점에서 확장하며 가장 가까운 점 추가     |
| **핵심 자료구조**  | Union-Find            | PriorityQueue + visited 배열 |
| **간선 생성 시점** | 사전에 모든 간선 생성          | 트리 확장 시 필요한 간선만 생성         |
| **유사 알고리즘**  | -                     | 다익스트라                      |
| **시간복잡도**    | O(E log E)            | O(E log V)                 |

이 문제에서는 E = N²이므로 두 알고리즘의 실질적인 성능 차이는 크지 않다. 다만 간선이 매우 많은 완전 그래프에서는 Prim이 메모리 면에서 유리하고, 간선이 적은 희소 그래프에서는 Kruskal이 간결하다.

## 핵심 정리

- MST 문제인지 판별하는 법: "모든 노드 연결 + 최소 비용 + 사이클 없음(트리)"
- Kruskal: 간선 정렬 + Union-Find → Greedy로 가장 싼 간선부터 채택
- Prim: 다익스트라 구조 + visited → 가장 가까운 점부터 트리에 흡수
- Union-Find의 경로 압축으로 find 연산을 O(1)에 가깝게 최적화 가능
