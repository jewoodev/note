## 1. 언급하신 WAS의 정의는 무엇일까요?

저도 이 부분을 어떻게 정의할지 고민이 많았는데, 웹 프레임워크라고 엄밀하게 정의하는게 더 좋을까 싶다가도 다음의 이유들로 WAS라는 명칭을 사용했습니다.

1. 'WAS' 라는 단어보다 '웹 프레임워크' 라는 단어가 더 길다.
2. 아키텍처 관점에서 'WebFlux'와 'WebMVC'을 각각 선택했을 때 달라지는 것이 'WAS가 달라지는 것'이기 때문에 WAS 선택이라고 하는게 맞지 않을까?
3. Tomcat도 비동기 방식으로 동작하는 게 가능하지만, 태생적으로 비동기 방식으로 동작하기 위해 설계된 Netty를 선택했다는 것도 WebFlux를 선택한 이유 중 하나이다.

이러한 이유들이 있는데 'WebFlux를 선택하게 되는 이유가 다 비슷하지 않을까?' 하는 생각에 중요도를 낮게 생각했습니다.  
이런 논리도 녹여내야 될 거 같네요.


## 2. MongoDB만 Fully Async/Non-blocking Driver를 지원하는 것은 아닙니다.

기존 JDBC로 비동기 API를 제공하는게 있나요? R2DBC를 말씀하시는 거라면 저도 알고 있습니다. 근데 R2DBC는 JDBC 기반이 아니라서, 다른 걸 이야기하시는 것 같은데 무엇인지 궁금합니다.

다시 MongoDB를 선택했던 이유에 대한 이야기로 돌아가보면, R2DBC를 사용해서 RDBMS를 사용하는 것도 고려했었습니다. 그럼에도 MongoDB를 선택했던 건 'DB가 가진 근본적인 설계 철학이 무엇이느냐' 에 중점을 두었기 때문입니다.

지원 여부를 생각해보면 데이터베이스 자체가 동기/비동기를 강제하는 것은 아닙니다. 클라이언트 드라이버와 애플리케이션에서 그것은 선택하는 건데,
RDBMS의 설계 의도는 "데이터의 정확성과 일관성을 보장하면서 안정적으로 처리하자" / MongoDB는 "대용량 데이터를 빠르고 유연하게 처리하자" 에 있습니다.

'스토리지 엔진의 동작 방식이 어떠한가'에 대해 생각해봐도 생성/변경 작업이 많은 채팅 서버의 사용 케이스에서는 MongoDB가 적합합니다.

이런 내용까지 녹여내야 될까요? 사실 1번의 내용도 2번의 내용도 뺐던 건 내용이 장황해지고 가독성이 떨어질까봐 뺐던 건데... 고민이네요.


## 3. Reactive Stack을 선택하는 것은 성능 그 이상의 리스크가 있습니다.

말씀하신대로 채팅 애플리케이션을 별도의 서버로 개발한 것이 맞습니다.

인스턴스 자원이 제약된 상황에서 왜 마이크로 서비스를 지향했는지에 대해서는 부끄럽지만 명확한 논리가 없습니다.

부족하지만 이유들을 이야기해보면, 그 이유들은 "채팅 애플리케이션에는 '웹소켓' 프로토콜이 쓰이는 게 적합하다" 에서 부터 시작됩니다.

웹소켓 세션이 생성되어서 앱 가동 기간동안 유지가 되고, 동시 접속자들의 I/O 요청들이 오가는 상황에서 '비용효율적이며 확장성이 있는' 서버가 되기 위해선 Reactive Stack이 필요했습니다.

또 웹소켓 프로토콜을 통한 채팅 애플리케이션의 사용 양상과, 구매자/판매자/관리자 애플리케이션의 사용 양상이 너무나 다르기 때문에 이를 분리하는 것이 자연스러웠습니다.

인스턴스 자원이 굉장히 제약된 상황에서 마이크로 서비스를 지향했다기 보단, 멀티 모듈을 채택했다는 쪽에 가깝다고 생각합니다.

Java 21 부터 등장한 Virtual Thread 덕분에 Blocking I/O에 대한 비용이 매우 싸진 것에 대해서는 잘 몰랐습니다... Reactive Stack이 필요 없을 정도인가요?
