Boolean 타입이 많은 엔티티 설계에 대해서 공부해보자.

yes or no 속성을 갖는 엔티티를 설계할 때 단순하게 컬럼을 계속해서 추가하는 처리는 피하는게 좋다.  
근본적인 아키텍처 관점에서의 문제와 속성이 늘어날 때마다 ALTER 로 테이블 스키마를 변경해야 하는 점과 불리언 타입의 데이터 표현 한계가 있는 점이 좋지 못한 설계의 원인이다.  
그리고 늘어나는 컬럼은 I/O 비용 증가를 야기하고 수십 개 수백 개의 불리언 컬럼을 관리하는 것이 어려워지는 결과로 이어진다. 이러한 문제는 한 마디로 "확장성이 나빠진다." 는 현상으로 표현할 수 있다.

# 물리 모델과 논리 모델
좋은 설계를 하기 위해서는 물리 모델과 논리 모델의 차이를 짚고 넘어가자. 논리 모델은 비즈니스 도메인의 개념을 표현한 것, 물리 모델은 논리 모델을 실제 구현에서의 성능이나 확장성을 고려한 것이다. 

## Anchor
Anchor는 데이터 모델링에서 핵심적인 역할을 하는, 비즈니스 도메인의 중심이 되는 엔티티다.  
이 엔티티는 비즈니스 개념의 본질적인 속성만을, 시간이 자나도 변하지 않는 기본적인 정보만을 갖는다.

Anchor에 속성을 추가할 땐 논리 모델과 물리 모델을 분리하는 전략을 사용하는 것이 데이터베이스 설계의 기본적인 원칙이다.
