# 정렬 조건
SELECT에 ORDER BY를 사용하는데 모든 데이터에 대해 정렬 연산이 필요한 경우엔 `filesort`가 된다. 이는 메모리나 임시 파일을 사용하기 때문에 인덱스를 사용하는 것에 비해 매우 느리다. 동작 흐름은 다음과 같다.

1. WHERE 조건으로 레코드 필터링
2. `sort_buffer_size`의 크기의 버퍼를 활용해 데이터를 정렬한다.
   1. `sort_buffer_size`는 MySQL Config의 설정값
      1. 너무 크게 설정하면 사용되지 않는 메모리 크기가 커져 낭비
      2. 너무 적게 설정하면 디스크 기반 `filesort`로 전환됨
      3. 그냥 손대지 말것. 디폴트 설정값을 변경하면 사이드 이펙트로 문제 발생이 많아질 것임.
   2. 정렬할 데이터를 메모리로 로드
   3. 정렬 수행 (퀵 소트, 머지 소트)

- 디스크 기반 `filesort`는 external sort를 수행
  - external sort: 데이터를 여러 chunk로 나눠서 각각 정렬한 후 병합하는 방식
    - 매우 성능적으로 나쁨

# 집계 함수
## COUNT
null이 아닌 컬럼의 갯수를 가져오고 싶다면 와일드 카드가 아닌 특정 컬럼을 넣어줘야 한다.

## GROUP BY 동작 원리
1. WHERE 조건절로 레코드를 필터링
2. GROUP BY 컬럼으로 그룹핑
3. 각 그룹에 대한 집계 함수 계산
4. HAVING 조건으로 그룹 필터링
5. ORDER BY 주어진 대로 정렬

동작 순서를 알고 있어야 문제가 발생했을 때 최적화하고 트러블 슈팅하는 것이 원활해짐.

# DISTINCT
## 처리 방법
1. 정렬 기반 중복 제거
   1. 결과를 정렬한 후 인접한 중복 값을 제거
2. 해시 테이블 기반의 중복 제거
   1. 각 고유한 값을 해시 테이블로 저장
   2. 이미 저장한 값은 건너뛰기
   3. 더 빠르지만 메모리를 더 많이 사용

## 주의할 점
```sql
SELECT DISTINCT address, age FROM member;
```
address의 중복만 제거하고 싶지만 (address, age) 조합의 중복이 제거됨

## vs GROUP BY
그룹 연산으로도 중복된 값을 제거할 수 있다. 둘 중에 그룹 바이를 더 권장한다.

DISTINCT는 내부적으로 정렬하기와 해시테이블을 사용하기의 두 방법 중 하나를 선택하거나 둘 모두를 사용하며 중복을 제거해 데이터가 많아질수록 효율이 나오지 않는다.
