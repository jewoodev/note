# CASE 활용
```sql
UPDATE member
SET 
    grade = CASE 
    WHEN age < 20 THEN 'Junior'
    WHEN age BETWEEN 20 AND 30 THEN 'Senior'
    WHEN age > 30 THEN 'Export'
    ELSE 'Unknown'
    END;
```
위와 같이 CASE 문을 활용해 UPDATE 하는 게 가능하다. 

CASE 문은 위에서 아래 순서로, 순차적으로 조건을 평가하면서 첫 번째로 참인 조건의 결과를 반환한다.  
이를 Short-Circuit Evaluation 이라고도 표현한다.  
이런 특성 때문에 가장 매칭이 잘 되는 조건을 가장 위에 위치시키는 것이 성능을 향상시킨다.

# 서브쿼리 활용
서브쿼리는 임시 테이블을 생성하므로 메모리 사용량을 크게 만든다. 서브쿼리 결과를 매칭하는 연산자로 EXISTS, IN이 있는데 전자를 사용하길 권장한다. 
전자는 Short Circuit을 지원하기 때문이다. null값이 나오더라도 안전하다는 장점도 있다.  
후자는 일단 모든 결과를 반환하고 비교하게 된다.

## 상관 서브쿼리
`=` 연산자에 사용되는 서브쿼리를 말한다. 외부 쿼리의 각 레코드마다 서브 쿼리가 실행되기에 특히 사용할 때 성능에 끼치는 영향을 생각해야 한다.

# JOIN 활용
JOIN을 활용하면 여러 테이블의 관계를 활용할 수 있다. 이 방법은 복잡한 관계를 가진 테이블 간의 데이터를 효율적이고 안정적으로 동기화한다.  
이를 애플리케이션 레벨에서 검증하는 것보다 훨씬 효율적이고 안전할 것이다.

JOIN 연산은 옵티마이저가 복잡한 실행 계획을 세우게 만든다.  

1. 쿼리 분석
2. 어떤 테이블을 먼저 읽어야 효율적인지 판단해 선택
   1. = 크기가 더 작거나 WHERE 조건으로 더 많이 필터링 되는 것
3. 알맞은 조인 알고리즘 결정
   1. Nested Loop Join, Hash Join, Sorted-Merge Join
4. 잠금을 어떻게 잡을건지 결정

JOIN은 서브쿼리와 동일한 결과를 얻기 위한 방법으로, 서브쿼리가 외부 테이블의 레코드마다 실행되는 것에 반해 한 번의 조인 연산만 수행해 효율적이다.

# LIMIT 활용
대량의 데이터를 처리할 때 시스템이 멈추지 않도록 청크 단위로 데이터를 나눠 처리할 수 있는 기법이다.  

이 기법없이 처리하면 메모리 사용량이 높아지고 수십 만개의 레코드에 X Lock이 걸려 다수의 '로우 데이터를 참조하는 쿼리'들이 block된다.  
언두 로그도 증가하게 된다. MySQL은 트랜잭션이 실패했을 때 ACID를 보장하기 위해 이전 상태를 로그성 데이터로 기록해두는데, 그게 언두 로그다.  
레플리케이션 관점에서도 문제가 생긴다. 대용량 업데이트가 발생하면 복제 지연이 길어지며 고가용성이 유지되지 않을 수 있다.

이 배치 처리에 대해서 ORDER BY로 정렬하여 처리할 수도 있다.

# IF 함수 활용
다음과 같은 IF 함수의 활용으로 애플리케이션 로직을 대신하면서 동시성 문제까지 해결할 수 있다.
```sql
UPDATE member
SET address = IF(age > 30, 'Seoul', 'Busan'),
    status = IF(address = 'Seoul', 'BUSY', 'FREE')
```

하지만 CASE 문에 비해서 가독성이 떨어지는 단점이 있다. 그래서 조건이 2~3 개 일 때는 사용하지만 그보다 많아지면 CASE 문이 더 나은 편이다.
