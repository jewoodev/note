MySQL은 단순하게 통짜로 존재하는 엔진이 아니다. 아파트처럼 여러 레이어로 이루어져 있고 각 층이 다른 역할을 수행한다. 

첫 번째 층은 '연결 계층'이다. 클라이언트가 MySQL에 연결한다면 가장 먼저 우선적으로 접속하게되는 곳이다.

두 번째 층은 "쿼리를 해석하고 계획을 세우는" 계층이다. 요청받은 쿼리의 문법이 정상적인지, 쿼리를 실행하기 위한(테이블에 대한 혹은 작업 종류에 대한) 권한이 있는지 확인하고 나서 최적의 계획을 수립한다. 어떤 인덱스를 사용할 것인지, 어떤 순서로 조인할 것인지 전반적인 "쿼리를 실행하기 위한" 전략을 세우는 단계라고 이해하면 된다.

마지막으로 "실제 쿼리를 실행하는" 스토리지 엔진 계층이다. 세운 계획을 바탕으로 실제 데이터를 스토리지 엔진이 읽고 쓰는 작업을 수행한다. 그럼 데이터를 읽을 때 어떤 작업을 수행하게 될까?

가장 먼저 "데이터의 위치를 찾는" 작업을 수행한다. 인덱스를 통해 데이터가 어느 페이지에 있는지 찾고 B tree 구조를 따라 리프 노드까지 이동하게 된다. 그 다음 페이지를 읽기 시작한다. 필요한 데이터 페이지를 디스크에서 읽어오는 작업을 수행하는데 먼저 버퍼 풀, 즉 메모리에 해당 페이지가 있는지 확인한다. 만약 버퍼 풀에 데이터가 있다면 '**캐시 히트**'가 발생하며 데이터를 바로 제공한다. 하지만 없다면 디스크까지 가서, 읽어오게 된다.

이후에는 데이터 **필터링** 과정을 거치게 된다. 페이지에서 조건에 맞는 레코드를 선별하는 작업을 수행한다. 이때 우리가 작성한 `WHERE` 절이 동작하게 된다. 그리고 최종적으로 필터링된 데이터를 SQL 계층으로 전달하면서 SQL 계층에서 GROUP BY나 ORDER BY 같은 추가적인 쿼리를 적용한 후 반환하게 된다.

대부분의 테이블은 스토리지 엔진으로 InnoDB를 사용하게 될 것이다. MyISAM은 효율적이지 않은데다가 트랜잭션을 지원하지 않기 때문에 거의 사용하지 않는다.

# InnoDB 내부 구조
## 버퍼 풀
메모리 구조로 빠른 처리를 위한 임시 저장소이다. 페이지 단위로 관리하며 다음의 쿼리로 관리 상태를 확인할 수 있다.

```sql
-- 버퍼 풀 기본 정보 확인하기
SELECT
    POOL_SIZE as 전체페이지수,
    ROUND(POOL_SIZE * 16384 / 1024 / 1024, 0) as 전체크기_MB,
    FREE_BUFFERS as 빈페이지수,
    DATABASE_PAGES as 데이터페이지수,
    MODIFIED_DATABASE_PAGES as 수정된페이지수,
    ROUND(HIT_RATE, 2) as 히트율_퍼센트
FROM information_schema.INNODB_BUFFER_POOL_STATS;

-- 어떤 종류의 페이지들이 메모리에 있는지 확인
SELECT
    PAGE_TYPE as 페이지종류,
    COUNT(*) as 개수,
    ROUND(COUNT(*) * 16 / 1024, 1) as 크기_MB,
    ROUND(COUNT(*) / (SELECT COUNT(*) FROM information_schema.INNODB_BUFFER_PAGE) * 100, 1) as 비율_퍼센트
FROM information_schema.INNODB_BUFFER_PAGE
GROUP BY PAGE_TYPE
ORDER BY 개수 DESC
    LIMIT 10;
```

페이지는 데이터를 저장하고 관리하는 고정된 크기의 메모리 혹은 디스크 블록이다. 기본값은 16kb이다. 데이터를 청크 단위로 쪼개는 단위 기준값이라고 이해하면 된다.

> _참고!_
> 
> 인덱스 페이지나 큰 데이터를 담는 파일 페이지도 트랜잭션 롤백용 데이터 등이 포함되어 있다.

## 로그 버퍼
로그 버퍼는 트랜잭션 로그를 저장하는 메모리 버퍼로 변경 결과를 디스크에 쓰기 전에 기록함으로써 성능을 높인다. 버퍼 풀과 함께 메모리를 사용하는 주요 구성 요소이다.

**WAL**(Write-Ahead Logging) 원칙을 따라 데이터 변경이 디스크에 저장되기 전에 로그 버퍼의 로그성 메세지로 기록하는 역할을 수행하는데 이는 캐시와 다를게 없지 않나? 싶다. 하지만 캐시와는 다르게 디스크에 저장하는 과정이 있다. 로그성 데이터를 결국 저장해야만 한다. 그 과정을 '플러시'라고 한다. 트랜잭션을 커밋하거나 버퍼가 가득차거나, 설정해둔 주기적인 인터벌 기준으로 플러시가 되면서 로그 버퍼에 있는 데이터가 디스크에 저장된다.
