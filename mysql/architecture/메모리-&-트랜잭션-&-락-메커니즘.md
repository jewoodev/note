MySQL의 메모리 영역은 전역과 세션별로 나누어진다. 두 영역의 차이를 이해하면 엔진으로 인한 이슈를 해결하고 최적화하는 데에 큰 도움이 된다.

# 전역
모든 연결에 공유되는 영역

- 버퍼 풀
- 로그 버퍼
- 테이블 캐시

# 세션 
- 정렬 버퍼
    - ORDER BY나 GROUP BY 같은 쿼리를 실행할 때 사용.
    - 데이터 크기가 버퍼의 크기를 넘어서면 '디스크 임시 파일'을 생성하여 처리.
- 조인 버퍼
    - 인덱스가 없는 조인 작업을 처리할 때 사용.
- 읽기 버퍼
    - 테이블을 스캔할 때 사용.
- 읽기 전용 버퍼
    - 인덱스 기반의 스캔을 할 때 사용.

# 버퍼 풀
여러 버퍼들을 앞서 살펴보았지만 모든 걸 알 필요는 없다. 가장 중요한 건 버퍼 풀이다. 

# LRU
버퍼 풀은 LRU라는 전통적인 알고리즘을 사용한다. 이는 가장 최근에 사용되지 않은 데이터를 우선적으로 제거해 메모리 공간 확보하는 알고리즘이다.  
이 알고리즘은 다음과 같은 단점이 있다. 

- 대량의 데이터를 읽으면 유용한 데이터가 밀려남.
  - 풀 테이블 스캔 혹은 예측 읽기(순차적 읽기 시 불필요한 페이지들이 캐시를 건드는 현상).

그래서 MySQL은 LRU를 있는 그대로 사용하지 않고 일부 개선해서 사용한다. 개선하는 방법에는 다음과 같은 것들이 있다.

- Minpoint
  - 새 페이지를 LRU 리스트의 중간 지점에 삽입.
  - 위 과정을 통해 Old 영역과 New(= Young) 영역을 분리.
  - Old 영역은 새로 읽은 페이지가 처음 들어가는 곳, 1초 후에 New 영역으로 이동.
  - New 영역은 자주 사용되는 중요한 페이지들만 위치.
    - 실제 LRU 알고리즘이 이 영역에서 실행.
- 어댑티브 해시 인덱스
  - 자주 사용되는 데이터를 빠르게 접근할 수 있게 하기 위해 설계됨.
  - 특정 패턴 or 쿼리가 4번 이상 반복되면 자동으로 해시 인덱스를 만듬.
    - 일반적인 B+ Tree 탐색 대신 O(1) 시간에 데이터 탐색.
  - 메모리 사용량 ↑.
  - 특정 시점에 만드는 것이기에 락 경합이 발생할 수 있음.
- Change Buffer
  - Secondary Index에 대한 쓰기 작업을 지연처리해 성능을 높이려 설계됨.
    - `innodb_change_buffer_max_size`와 `innodb_change_buffering` 설정으로 어떤 연산(inserts, deletes, purges, changes, all, none)을 버퍼링할지 제어 가능.
  - 변경하려는 페이지가 버퍼 풀에 없으면 디스크에서 즉시 읽어오지 않고,  
    변경 정보를 Change Buffer에 저장해두었다가 해당 페이지가 버퍼 풀에 로드될 때 merge 처리함.
  - 이를 통해 불필요한 랜덤 디스크 I/O 횟수를 줄여 성능을 높임.
    - 대량 INSERT 등 배치 작업에서 성능 크게 ↑.
- 예측 읽기
  - 데이터를 미리 읽어오는 기술.
  - 순차적인 데이터 읽기가 감지되면 다음 페이지들을 미리 로드해서 대용량 테이블 스캔 성능 ↑.
  - Linear Read-Ahead라고도 함.
  - Random Read-Ahead 방식도 있는데,  
    랜덤 접근이 많더라도 데이터 블록 내에서 여러 페이지가 자주 읽히면 그 전체 블록을 미리 로드한다.

# 락
## 테이블 락
- 테이블 전체에 락을 걸어 모든 행에 대한 접근을 제어하는 행위.
- `WHERE` 조건 없이 업데이트를 한다면 발생.

## 갭락
- 인덱스의 특정 범위에 락을 걸어 그 범위에 새 행이 추가되는 것을 막는 행위.

## Exclusive Lock
- 데이터를 수정하고 있을 때 다른 트랜잭션이 같은 레코드를 수정하는 것을 막는 행위.

## Shared Lock
- 데이터를 읽고 있을 때 다른 트랜잭션이 읽는 것은 허용하지만 쓰는 건 막는 행위.

# MVCC
락이 아닌 동시성 제어 행위로, 락은 작업이 block되는 일을 만들어 성능을 나쁘게 만들기에 더 좋은 성능의 동시성 제어 방법을 고민하다 만들어졌다.  
락 없이 동시성 제어를 하여 읽기 트랜잭션이 쓰기 트랜잭션을 기다리지 않는다.

MVCC는 스냅샷을 찍어 동시성 제어 기능을 제공한다. 이는 트랜잭션이 시작하는 시점을 기준으로 일관성을 유지한다. 만약 트랜잭션 A가 데이터 수정 중일 때 트랜잭션 B가 읽기를 하려고 하면 B가 시작된 시점의 스냅샷(수정되기 전)을 제공한다. 두 트랜잭션 모두 쓰기를 한다면 락을 걸어 무결성을 지킨다.
