# Nested Loop Join (NLJ)
가장 기본적인 조인 방식으로, 외부 테이블(driving table)의 각 행마다 내부 테이블(driven table)을 반복 탐색한다.  

동작 방식은 이중 for문과 동일하다. 외부 테이블에서 한 행을 꺼내고, 내부 테이블 전체를 스캔하면서 조인 조건에 맞는 행을 찾는다.  
MySQL에서는 이를 개선한 **Block Nested Loop Join** (BNL)이 있었는데, 그것은 join buffer에 외부 테이블의 행을 여러 개 모아두고 내부 테이블을 한 번 스캔할 때 버퍼 내 모든 행과 비교하는 방식이다.

## 특성
- 인덱스가 있을 때 매우 효율적 (내부 테이블 탐색이 O(log N)이 되므로)
- 인덱스가 없으면 O(N × M)으로 최악 
- MySQL이 전통적으로 가장 많이 사용해온 방식

# Hash Join
MySQL 8.0.18부터 도입되었고, 8.0.20부터 BNL을 완전히 대체했다.  
Build 단계에서 작은 쪽 테이블(build input)의 조인 키로 해시 테이블을 메모리에 구축한다. 그 다음 Probe 단계에서 큰 쪽 테이블(probe input)을 스캔하면서 각 행의 조인 키를 해시 테이블에서 O(1)로 조회한다.

## 특성
- 등가 조인(equi-join)에서만 사용 가능 (`=` 조건)
- 인덱스가 없는 대용량 테이블 간 조인에서 NLJ보다 훨씬 빠름 
- 해시 테이블이 `join_buffer_size`를 초과하면 디스크에 spill하므로 메모리 설정이 중요 
- 시간 복잡도: O(N + M) (build O(N) + probe O(M))

# Sort-Merge Join
두 테이블을 각각 조인 키 기준으로 정렬한 뒤, 양쪽을 동시에 순차 탐색하면서 머지하는 방식이다.

## 특성
- 두 테이블이 이미 정렬되어 있으면 (인덱스가 있거나 ORDER BY 결과) 정렬 비용이 없어 매우 효율적 
- 범위 조인(`>=`, `BETWEEN` 등)에도 사용 가능 (Hash Join과의 차별점)
- 시간 복잡도: 정렬 O(N log N + M log M) + 머지 O(N + M)

## ⚠️ 중요한 점
MySQL은 Sort-Merge Join을 명시적으로 지원하지 않는다. 

Oracle, PostgreSQL, SQL Server에서는 옵티마이저가 선택할 수 있지만, MySQL의 옵티마이저는 NLJ와 Hash Join만 사용한다.  
MySQL에서 비슷한 효과를 내는 경우가 있는데, 양쪽 테이블에 조인 키 인덱스가 걸려 있을 때 NLJ가 인덱스 순서대로 머지하듯 동작하는 것이 가장 유사하다.

## 요약 비교

| 구분       | Nested Loop                | Hash Join        | Sort-Merge           |
|----------|----------------------------|------------------|----------------------|
| 시간복잡도    | O(N × M), 인덱스 시 O(N log M) | O(N + M)         | O(N log N + M log M) |
| 인덱스 의존   | 높음                         | 낮음               | 중간(정렬 여부)            |
| 조인 조건    | 모든 조건                      | 등가 조인만           | 등가 + 범위              |
| 메모리 사용   | 낮음                         | 높음 (해시 테이블)      | 중간 (정렬 버퍼)           |
| MySQL 지원 | ✅ 핵심                       | ✅ 8.0.18+        | ❌ 미지원                |
| 유리한 상황   | 소량 데이터, 인덱스 있을 때           | 대량 데이터, 인덱스 없을 때 | 이미 정렬된 대량 데이터        |
