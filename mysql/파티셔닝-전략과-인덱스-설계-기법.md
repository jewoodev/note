파티셔닝은 Mysql에서 큰 테이블을 더 작고 관리하기 쉬운 조각과 같은 형태로 나누는 기술이다. 테이블을 더 작게 쪼개는 것으로 쪼개지는 단위를 '파티션'이라고 부르며, 이 쪼개진 데이터는 여전히 단일 객체로 취급되는 특징이 있다.

파티셔닝은 대표적으로 큰 데이터 테이블을 다룰 때 효과적이다. 예를 들어 날짜로 파티션을 나눈다면 특정 날짜 범위를 필터링하는 쿼리는 특정 파티션에만 접근하게 된다. 큰 사이즈의 테이블에 대한 조회는 디스크 I/O 작업이 굉장히 부담스럽다. 그런 테이블을 여러 파티션으로 쪼개면 더 작은 데이터에 대한 조회가 되어 부담이 적어진다.

파티셔닝이 적용될 땐 '파티셔닝 키'를 기준값으로 분배가 된다. 이 키를 기준으로 데이터 분배가 적절하게 이루어지고 고르게 분포되어야 한다. 키를 잘못 설정하면 특정 파티션에만 과도하게 가중치가 부여되어 크기가 불균형해질 수 있다.

# 해시 파티셔닝
DDL 마지막에 `PARTITION BY HASH (column_name) PARTITIONS 파티셔닝갯수`를 붙여주면 해시 파티셔닝을 적용할 수 있다.

이 파티셔닝 기법은 시계열 데이터가 아닌 경우에도 도입할 수 있다는 장점이 있다. 데이터를 분배할 때 해시함수로 할당하기 때문에 시계열 데이터가 아니라도 균일하게 분산된다.

대신 해시함수 대상이 되는 컬럼이 아닌 컬럼에 '파티션 프루닝' 적용이 어려울 수 있다는 단점이 있다. 파티션 프루닝은 쿼리 실행 중 불필요한 파티션은 제외하고 필요한 파티션만 스캔하면서 최적화하는 방법이다. 즉, 해시 파티셔닝을 한 경우엔 다음과 같은 경우에만 파티션 프루닝이 적용된다.

1. 파티션 키에 대한 동등 조건 `=`, `IN`
2. `IS NULL` 조건: NULL이 특정 파티션의 VALUES IN 목록에 포함되어 있을 때

프루닝이 불가한 경우는 다음과 같다. 

1. 범위 조건 (`<`, `>`, `BETWEEN`, `!=`)
    범위 조건은 전부 프루닝이 불가하다. 해시 결과가 원래 값의 순서를 보장하지 않기 때문이다.

    ```sql
    -- 프루닝 X: user_id 1~1000이 어떤 파티션에 분포되는지 알 수 없음
    SELECT * FROM orders WHERE user_id > 100;
    SELECT * FROM orders WHERE user_id BETWEEN 1 AND 1000;
    SELECT * FROM orders WHERE user_id != 100;
    ```
2. 파티션 키에 함수/연산을 적용한 경우
    ```sql
    SELECT * FROM orders WHERE region_id + 0 = 1;
    SELECT * FROM orders WHERE ABS(region_id) = 1;
    ```
3. 파티션 키가 아닌 컬럼에 대한 조건
    ```sql
    -- 프루닝 X: 당연히 다른 컬럼 조건으로는 파티션 제거 불가
    SELECT * FROM orders WHERE user_id = 100;
    ```

## LINEAR HASH의 경우
LINEAR HASH도 프루닝 규칙은 동일하다(=, IN, IS NULL만 가능). 다만 LINEAR HASH는 power-of-two 알고리즘을 사용해서 파티션 추가/삭제 시 데이터 재분배 범위가 작다는 운영상의 차이만 있다.

## KEY 파티셔닝의 경우
PARTITION BY KEY도 내부적으로 해시 기반이여서 프루닝 규칙은 해시 파티셔닝과 동일하다.

# 리스트 파티셔닝
```sql
PARTITION BY LIST COLUMNS(market_code) (
        PARTITION p_kospi VALUES IN ('KOSPI'),
        PARTITION p_kosdaq VALUES IN ('KOSDAQ'),
        PARTITION p_nasdaq VALUES IN ('NASDAQ'),
        PARTITION p_nyse VALUES IN ('NYSE'),
        PARTITION p_other VALUES IN ('OTHER', 'CRYPTO', 'FOREX')
        );
```
리스트 파티셔닝 파티션 프루닝 적용 조건은 해시 파티셔닝과 동일하다.

다른 점은 다음과 같다.

- `market_code` 의 값이 각각 독립적이기에 특정 파티션만을 백업하거나 삭제하는 것이 가능.
- 논리적으로 파티션이 분리됨.
- 파티션 키의 종류가 제한됨.
- 특정 고정 값에 대해서만 지원함.
  - 아직 파티션이 생성되기 전인데 새로운 `market_code`가 쿼리에 포함될 경우 문제 발생.
    - 이를 대비하기 위해 기본 파티션 키를 설정할 수 있음.
- 특정 `market_code`로만 데이터가 몰리면 파티션 크기 불균형 발생.

# 인덱스 설계, Cardinality 기반 원칙
카디널리티는 "얼마나 고유한지"에 대한 기준이다. 이 기준값이 높을 수록, 그 컬럼에 인덱스를 생성해야 한다.  
그리고 복합 인덱스를 생성할 때 더 높은 카디널리티를 가진 컬럼을 먼저 인덱스로 생성하는 것이 좋다. 더 많이 고유한 컬럼을 기준으로 먼저 필터링하는 것이 성능을 더 높이기 때문이다.

# 문자열 기반 인덱스
긴 문자열 컬럼에 대해서는 적절한 접두사 인덱스를 생성해주는 게 좋다.

```sql
-- 접두사 인덱스 생성(10자가 적절하다면)
CREATE INDEX idx_post_title_prefix ON posts(title(10));
```
접두사 인덱스는 전체 문자열에 대한 인덱스가 아닌 (설정된 길이 만큼의) 앞 부분만을 대상으로 인덱스를 설정한다.  
더 적은 크기를 대상으로 하기에 저장공간을 절약하고, 성능을 최적화할 수 있다.

# Key
1. 슈퍼키
   1. 튜플을 유일하게 식별할 수 있는 하나 이상의 컬럼.
   2. 유일성을 만족해야함.
   3. 최소성은 만족x 해도 됨.
2. 후보키
   1. 슈퍼키 중에서 최소성을 만족하는 키.
   2. 유일성을 만족하는 가장 작은 속성의 집합.
3. 기본키
   1. 후보키 중에서 개발자가 선택한 단 하나의 메인 키.
   2. NULL 불가.
   3. 중복 불가.
4. 대체키
   1. 후보키 중에 기본키로 선택되지 않은 키.
5. 외래키
   1. 다른 테이블의 기본키를 참조하여 테이블 간의 관계를 연결하는 키.
6. 복합키
   1. 두 개 이상의 컬럼으로 만들어지는 기본키.

# 정규화
1. 1NF
   1. 원자성 확보.
2. 2NF
   1. 부분함수에 대한 종속 제거.
   2. 복합키를 가질 때 키의 일부에만 종속되는 컬럼이 없어야 함.
3. 3NF
   1. 이행함수에 대한 종속 제거.
   2. PK가 아닌 다른 컬럼에 의해 결정되는 컬럼이 없어야 함.
