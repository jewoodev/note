MySQL의 기본 레플리케이션(Async/Semi-sync)은 자동 페일오버를 지원하지 않는다. 마스터가 죽으면 슬레이브가 알아서 마스터가 되는 게 아니라, 별도의 메커니즘이 필요하다.  
자동 페일오버가 되는 경우는 **MySQL InnoDB Cluster**(Group Replication) 또는 **MHA**, **Orchestrator** 같은 외부 도구를 쓸 때이다.

# 전통적 레플리케이션 (Async/Semi-sync) + 외부 도구
대표적으로 MHA(Master High Availability) 기준으로 설명하면  
장애 감지 단계에서 MHA Manager가 주기적으로 마스터에 헬스체크(ping)를 보내다가, 연속 실패하면 마스터 다운으로 판단한다.

승격 대상 선정 시에는 각 슬레이브의 relay log 적용 상태를 확인해서, 마스터의 binlog를 가장 많이 따라잡은(가장 최신 상태인) 슬레이브를 후보로 선택한다.  
수동으로 우선순위를 지정할 수도 있다.

데이터 정합성 보정이 핵심인데, 마스터가 죽기 직전에 보낸 binlog 이벤트를 일부 슬레이브만 받았을 수 있기 때문이다.  
MHA는 가장 최신 슬레이브의 relay log에서 다른 슬레이브에 없는 차분(diff) 이벤트를 추출해서, 나머지 슬레이브에 적용시킨다.  
이걸 통해 모든 슬레이브의 데이터를 최대한 맞춘다.

승격 실행 단계에서는 선택된 슬레이브에 `STOP SLAVE` → `RESET SLAVE ALL` → `SET GLOBAL read_only=OFF` 처리를 하고, 나머지 슬레이브들은 CHANGE MASTER TO로 새 마스터를 바라보도록 재설정한다.  
그리고 **VIP 전환** 또는 **DNS 변경**을 통해 애플리케이션이 새 마스터로 연결되도록 한다.

## MySQL Group Replication (InnoDB Cluster)
이건 MySQL 자체적으로 자동 페일오버를 지원하는 방식이다.  

노드들이 **Paxos 기반의 그룹 통신 프로토콜**로 서로 상태를 주고받는다.  
Single-Primary 모드에서 프라이머리가 죽으면, 남은 노드들이 **합의**(consensus)를 통해 새 프라이머리를 자동 선출한다.  
선출 기준은 `group_replication_member_weight`와 `server_uuid` 순서이다.

이 조치는 MySQL Router가 앞단에서 프라이머리 변경을 감지하고, 애플리케이션 연결을 자동으로 새 프라이머리로 라우팅해줘서 애플리케이션 코드 변경 없이 처리된다.

# 페일오버 시 주의할 점
**데이터 유실 가능성**이 있다. 비동기 레플리케이션에서는 마스터에만 커밋되고 슬레이브에 전파 안 된 트랜잭션이 유실될 수 있다.  
Semi-sync를 쓰면 줄일 수 있고, Group Replication은 과반수 노드에 전파된 후 커밋하기 때문에 유실이 거의 없다.

**GTID**(Global Transaction ID) 를 사용하면 각 트랜잭션에 고유 ID가 붙기 때문에, 페일오버 시 어디까지 적용됐는지 정확히 파악할 수 있어서 정합성 보정이 훨씬 수월해진다.
