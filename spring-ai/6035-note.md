## 1. 배경 — JSON Schema, `$defs`, `$ref`

### 1-1. JSON Schema는 "데이터의 모양"을 적은 문서

도구(tool)를 LLM에게 줄 때, "이 도구는 이런 모양의 입력을 받는다"를 JSON Schema로 적어 보낸다. 평범한 타입은 인라인으로 그냥 펼쳐 쓰면 된다.

```java
record Person(String name, int age) {}
```

```json
{
  "type": "object",
  "properties": {
    "name": { "type": "string" },
    "age":  { "type": "integer" }
  }
}
```

### 1-2. 재귀 타입은 인라인으로 표현 불가능

자기 자신을 다시 품는 타입은 인라인으로 펼치면 무한 중첩이 되어 유한한 문서로 적을 수 없다.

```java
record TreeNode(String name, List<TreeNode> children) {}
```

`TreeNode` 안에 `TreeNode`가 또 있고, 그 안에 또… 끝나지 않는다.

### 1-3. 해법 — `$defs`(이름 붙은 스키마 사전) + `$ref`(그 이름을 가리키는 포인터)

재귀 함수가 본문에서 자기 *이름*을 호출하듯, JSON Schema도 타입에 이름을 붙이고 그 이름을 참조한다.

- **`$defs`** — 재사용할 스키마 조각에 이름을 붙여 모아두는 사전(dictionary). 키=이름, 값=스키마. (옛 드래프트에서는 `definitions`라는 이름이었고 역할은 동일.)
- **`$ref`** — "여기에 스키마를 인라인하는 대신 저기 정의된 걸 가져다 써"라는 포인터. 값은 경로 문자열.

```json
{
  "type": "object",
  "properties": {
    "name": { "type": "string" },
    "children": {
      "type": "array",
      "items": { "$ref": "#/$defs/TreeNode" }
    }
  },
  "$defs": {
    "TreeNode": {
      "type": "object",
      "properties": {
        "name": { "type": "string" },
        "children": {
          "type": "array",
          "items": { "$ref": "#/$defs/TreeNode" }
        }
      }
    }
  }
}
```

`TreeNode` 정의 안에서 다시 `#/$defs/TreeNode`를 가리킨다 — **자기 자신을 참조**. 이것이 재귀 타입을 유한한 문서로 표현하는 유일한 방법이다. (`$defs`/`$ref`는 중복 제거에도 쓰이지만, #5888에서 *반드시* 필요한 건 재귀 쪽이다.)

### 1-4. `$ref` 경로 문자열 `#/$defs/TreeNode` 뜯어보기 (JSON Pointer)

| 조각 | 의미 |
|---|---|
| `#` | **문서의 루트(최상단)** |
| `/$defs` | 루트에서 `$defs`라는 키로 들어감 |
| `/TreeNode` | 그 안에서 다시 `TreeNode`라는 키로 들어감 |

파일 시스템의 절대경로와 같다. `#`이 절대경로의 루트 `/` 역할.

> **핵심 규칙: `#`은 언제나 "문서 루트"를 가리킨다.** 따라서 `#/$defs/...` 참조가 동작하려면 `$defs` 블록이 반드시 **문서 루트 바로 아래**에 있어야 한다.

## 2. #5888 버그의 정체

spring-ai는 자바 타입 → JSON Schema 변환을 외부 라이브러리 **victools**(`com.github.victools:jsonschema-generator`)에 맡긴다. 흐름:

```
파라미터 1개의 자바 타입  ──victools──▶  자기완결 서브스키마
                                        { ..., $defs: { Node: ... }, $ref: "#/$defs/Node" }
                                          └─ 이때 # 은 "이 서브스키마"의 루트를 의미
```

도구는 보통 파라미터가 여러 개이므로, spring-ai 생성기는 이 서브스키마들을 모아 **하나의 래퍼 스키마**로 조립한다.

```json
{ "type": "object", "properties": { "param1": <서브스키마1>, "param2": <서브스키마2> } }
```

서브스키마1이 `properties.param1` 아래로 들어가는 순간, 그 안의 `#/$defs/Node`는 의미가 바뀐다 — 원래 "서브스키마1의 `$defs`"였는데, 이제 `#`은 **래퍼의 루트**를 가리킨다. 래퍼 루트에는 `$defs`가 없으므로 참조가 깨진다. 이것을 **재부모화(re-parenting) 버그**라 부른다.

버그가 든 출력 예시:

```json
{
  "type": "object",
  "properties": {
    "node": {
      "type": "object",
      "$defs": {                                  ← 루트가 아니라 properties/node 안쪽에 박힘
        "TreeNode": { ... "$ref": "#/$defs/TreeNode" ... }
      },
      "properties": {
        "items": { "$ref": "#/$defs/TreeNode" }   ← 이 ref는 "루트의 $defs"를 찾는다
      }
    }
  }
}
```

- `$ref`는 `#/$defs/TreeNode` = "루트의 `$defs`"를 찾으라 한다.
- 실제 `$defs`는 `#/properties/node/$defs/TreeNode`에 있다 — 루트에는 `$defs`가 없다.
- 결과: **해석 불가능한 참조(unresolvable `$ref`)** → 도구 입력 검증이 깨진다.

이슈 제목 그대로: "nested `$defs`, root-relative `$ref`".

**수정 방향**: `$ref`가 기대하는 위치, 즉 **래퍼 루트로 `$defs`를 끌어올린다(hoist)**. 이것이 `hoistDefsToRoot`라는 이름의 뜻.

## 4. `JsonSchemaUtils.java` 상세

### 4-1. 파일 구조 — "우리 코드"가 어디인지

함수가 4개 있는데 그중 **3개만** #6035 작업이다.

| 함수 | 줄 | #6035 작업? |
|---|---|---|
| `hoistDefsToRoot` | 60–100 | ✅ 핵심 |
| `uniqueDefsKey` | 102–110 | ✅ private 헬퍼 |
| `rewriteDefsRefs` | 112–137 | ✅ private 헬퍼 |
| `ensureValidInputSchema` | 147–173 | ❌ 기존 코드(#4855) — 무관 |

- `public final class` + `private` 생성자 → 인스턴스를 못 만드는 순수 정적 유틸 모음.
- import가 `tools.jackson.databind` → Jackson 3.x의 새 패키지 네임스페이스(옛 `com.fasterxml.jackson`). 동작 이해엔 영향 없음.

### 4-2. doc 주석 (44–59줄) — "왜"가 여기 다 있다

핵심 문장: victools는 `$defs`와 `$ref`가 **서브스키마 기준**인 자기완결 스키마를 만든다. 이 서브스키마를 `properties.<paramName>` 아래로 인라인하면 기존 `#/$defs/<Name>` ref들이 바깥 루트로 재부모화되어, `$defs`를 먼저 끌어올리지 않으면 해석 불가능해진다. (2장의 설명과 동일.)

### 4-3. `hoistDefsToRoot(rootSchema, subSchema)` — 60~100줄

인자: `rootSchema`(래퍼, 정의를 받을 곳), `subSchema`(파라미터 서브스키마, `$defs`를 내줄 곳).

**① `$defs`를 서브스키마에서 떼어낸다 (61–64줄)**

```java
JsonNode nestedDefs = subSchema.remove("$defs");
if (nestedDefs == null || !nestedDefs.isObject()) return;
```

`remove`는 떼어낸 노드를 반환하면서 서브스키마에서는 삭제한다. 재귀 타입이 아니면 `$defs`가 없으니 그냥 종료. (62줄 가드 상세는 5장 참고.)

**② 루트의 `$defs`를 확보 (65–66줄)**

```java
ObjectNode rootDefs = rootSchema.has("$defs") ? (ObjectNode) rootSchema.get("$defs")
        : rootSchema.putObject("$defs");
```

래퍼에 `$defs`가 있으면 쓰고, 없으면 새로 만든다.

**③ 이번 배치의 키를 전부 미리 수집 (67–70줄)** — 머지 회고 #2 버그픽스의 일부

```java
Set<String> batchKeys = new HashSet<>();
((ObjectNode) nestedDefs).properties().forEach(e -> batchKeys.add(e.getKey()));
```

이번에 들어올 `$defs` 항목 이름을 전부 미리 모은다. 이름 충돌 해결 시 같은 배치의 형제 항목이 차지할 이름을 빼앗지 않기 위해서(④-c, 4-4 참고).

**④ 항목별로 루트에 합치기 (73–88줄)** — 각 항목마다 세 갈래

```java
if (!rootDefs.has(key)) {                 // (a) 루트에 같은 이름 없음
    rootDefs.set(key, value);             //  → 그대로 넣는다
    insertedKeys.add(key);
    return;
}
if (rootDefs.get(key).equals(value)) {    // (b) 이름·내용 모두 동일
    return;                                //  → 중복이므로 버린다 (dedup)
}
String renamed = uniqueDefsKey(rootDefs, key, batchKeys);  // (c) 이름은 같은데 내용 다름
rootDefs.set(renamed, value);                              //  → 이름 바꿔 넣는다
renames.put(key, renamed);                                 //  → 옛이름→새이름 기록
insertedKeys.add(renamed);
```

- **(a)** 새 이름이면 그대로 추가.
- **(b)** 이름·내용이 똑같으면(두 파라미터가 같은 타입 사용) 하나로 합침.
- **(c)** 진짜 충돌(이름 같고 타입 다름). 이름을 바꿔 넣고 `renames` 맵에 기록.
- `insertedKeys`는 "이번 호출이 실제로 루트에 넣은 항목" 목록 — ⑦에서 사용.

**⑤ 이름 변경이 없었으면 종료 (89–91줄)**

```java
if (renames.isEmpty()) return;
```

이름을 바꾼 게 없으면 모든 `#/$defs/<key>` ref가 여전히 유효 → ref 재작성 불필요.

**⑥ 인라인되는 서브스키마의 ref 재작성 (92줄)**

```java
rewriteDefsRefs(subSchema, renames);
```

`Node`→`Node_2`로 바꿨다면 서브스키마 안의 `#/$defs/Node` ref들을 `#/$defs/Node_2`로 갱신.

**⑦ 이번에 넣은 정의 *내부*의 ref만 재작성 (93–99줄)** — 미묘한 부분

```java
for (String insertedKey : insertedKeys) {
    rewriteDefsRefs(rootDefs.get(insertedKey), renames);
}
```

재귀 타입의 정의는 자기 자신을 `$ref`로 가리키므로, 루트로 옮긴 정의 안의 ref도 새 이름으로 고쳐야 한다. 단 **이번 호출이 넣은 것만** 고친다 — 이전 `hoistDefsToRoot` 호출이 넣어둔 항목은 이미 자기 시점 rename이 적용 완료라, 지금 배치의 rename 맵으로 또 건드리면 원래 키를 정상적으로 가리키던 ref가 망가진다.

### 4-4. `uniqueDefsKey(rootDefs, base, batchKeys)` — 102~110줄

충돌난 이름에 안 겹치는 새 이름을 찾는 헬퍼.

```java
int suffix = 2;
do {
    candidate = base + "_" + suffix++;     // Node_2, Node_3, Node_4 ...
}
while (rootDefs.has(candidate) || batchKeys.contains(candidate));
```

종료 조건 두 개를 모두 통과해야 한다.

- `rootDefs.has(candidate)` — 루트에 이미 들어간 이름과 겹치면 안 됨.
- `batchKeys.contains(candidate)` — 같은 배치에서 곧 들어올 형제 이름과도 겹치면 안 됨.

`batchKeys` 검사가 **머지 회고 #2의 "진짜 버그 수정"**이다. 원래 우리 구현은 `rootDefs`만 봤다. 그러면 한 배치에 `Node`와 `Node_2`가 둘 다 원래 이름으로 있고 `Node`가 충돌나면 `uniqueDefsKey`가 `Node_2`를 후보로 골라버려, 그 배치 형제 항목과 조용히(silent) 덮어쓰기 충돌이 난다. 메인테이너가 `batchKeys` 검사를 더해 막았다.
