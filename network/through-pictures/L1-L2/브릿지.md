허브와 허브를 계속 연결해 네트워크를 형성하면 콜리전 도메인이 너무 커지게 된다. 이 문제를 해결할 방법으로 "메모리와 프로세서를 허브에 장착하면 모든 포트로 브로드캐스팅할 필요가 없잖아?" 라는 아이디어에서 브릿지가 만들어진다. 

브릿지는 **MAC 주소 테이블**을 메모리에 만들어 브로드캐스팅하지 않고 유니캐스팅한다. PC1 ~ PC4가 브릿지로 연결되어 있는 상황에서 PC1 -> PC4로 데이터를 전송하는 과정을 살펴보자.

# 브릿지를 통한 통신 예시
브릿지는 MAC 주소 테이블이 비어있는 상태에서 시작한다.

## 전송 과정
1. **1단계: PC1이 프레임 전송**
   - PC1은 목적지 MAC 주소를 PC4의 MAC 주소로 설정한 이더넷 프레임을 생성
   - 출발지 MAC 주소는 PC1의 주소로 설정
2. **2단계: 브릿지의 학습(Learning)**
   - 브릿지는 PC1에서 온 프레임을 받으면 출발지 MAC 주소(PC1)를 확인
   - "PC1은 이 포트에 연결되어 있구나"라고 MAC 주소 테이블에 기록
   - 테이블: {PC1 MAC → 포트1}
3. **3단계: 브릿지의 플러딩(Flooding)**
   - 브릿지는 목적지 MAC 주소(PC4)가 어느 포트에 있는지 아직 모름
   - 따라서 수신한 포트를 제외한 모든 포트로 프레임을 복사해서 전송(플러딩)
   - PC2, PC3, PC4 모두 이 프레임을 받게 됨
4. **4단계: 수신 확인**
   - PC2, PC3는 목적지 MAC 주소가 자신의 것이 아니므로 프레임을 버림
   - PC4는 목적지 MAC 주소가 자신의 것이므로 프레임을 수신하고 처리
5. **5단계: 응답 과정(PC4 → PC1)**
   - PC4가 응답을 보낼 때, 브릿지는 PC4의 MAC 주소를 학습
   - 이번엔 브릿지가 PC1의 위치를 이미 알고 있으므로 해당 포트로만 전송(플러딩 불필요)
     - 이 작업을 **Forwarding**이라고 함
   - 브릿지 테이블: {PC1 MAC → 포트1, PC4 MAC → 포트4}

이후 PC1과 PC4 간의 통신은 브릿지가 양쪽의 위치를 모두 알고 있으므로, 해당 포트로만 정확하게 전달되어 네트워크 트래픽이 줄어든다. 이것이 브릿지의 핵심 기능인 "**필터링**"이다.
