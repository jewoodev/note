# 메신저 서버 스토리
치기공 도메인은 치과에서 치료를 위해 필요한 기공물을 기공소에 의뢰하는 인과관계를 갖고 있다. 환자의 케이스에 맞게 의사는 기공소에 세부적인 요구사항들을 이야기하고 확인받는 과정이 필수적으로 필요하다. 그걸 잘하고 못하고가 기공소의 중요한 역량이기도 하다. 고객인 치과 측의 컴플레인은 주로 기공소와 치과가 소통하는데 있어 매끄럽지 못할 때 발생하기 때문에, 운영진들은 그들의 대화를 팔로업해서 고객이 불만을 갖는 것 같으면 파악된 고객의 성향에 맞는 기공소를 소개하는 방식으로 여러 기공소를 판매자로 갖고 있는 플랫폼의 장점을 활용하고 있었다.

하지만 고객이 사용하는 채팅 앱에 맞춰 판매자와 운영진들이 여러 앱을 설치해서 이러한 서비스를 제공하고 있기 때문에, 각각의 고객을 팔로업할 때 여러 앱을 번갈아가며 확인해야 하는 데서 오는 어려움과 운영진들이 실제로 사용하는 카카오톡에 사적인 채팅과 구별이 어려워지는 어려움을 확인하였다. 

그리고 한국 기공사들이 영어로 소통하는 것에 갖는 어려움, 그리고 운영진들이 일일히 그들의 대화를 팔로업하기 어렵다는 게 기업의 비즈니스 상황이었다. 그래서 나는 치과 측의 채팅을 번역하는 기능과 채팅 내용에서 감정을 읽어내는 기능을 가진, 그리고 비즈니스 안의 소통을 우리의 앱에서 관리할 수 있도록 메신저 서버 개발을 계획하게 되었다.

작년부터 이용자 수는 꾸준히 늘어가고 있었기 때문에 확장성 있는 애플리케이션 구조를 구상했고, 실시간 채팅 통신에 적합한 웹소켓과 빈번한 채팅 발행을 비용효율적으로 처리할 수 있는 WebFlux + MongoDB를 메신저 서버의 기반에 두었다. 그리고 WAS가 확장될 때 WAS마다 고유한 키를 갖게 하고, 그 키 값으로 Pub/Sub 해서 부하를 웹소켓 연결 부하를 분산시키는 구조를 구축했다. 이것이 설정된 확장성 있는 구조의 기반이었다.

정리하자면, 메신저 서버 아키텍처는 WAS로 WebFlux를 데이터베이스는 MongoDB를 사용하며, 번역/감정 분석 기능과 Pub/Sub의 인터페이스, 총 3개의 인터페이스로 구성된다.

## 1. 비즈니스 상황에 적합한 WAS

비용이 가장 큰 걸림돌인 현재 회사의 비즈니스 상황에서는 동일한 서버 스펙으로 최대한의 성능을 내는 것이 필수였다. 메신저 서버의 특성상 동시 접속자 수가 많고 실시간 메시지 처리가 빈번하기 때문에, I/O 집약적 작업이 CPU 집약적 작업보다 훨씬 많다는 점을 고려하여 WAS 선택 기준을 설정했다.

### 1.1 WebMVC vs WebFlux 비용 비교

메신저 서버의 주요 작업들을 분석해보면 대부분이 I/O 집약적 작업이다:
- 데이터베이스 읽기/쓰기 (메시지 저장, 사용자 조회)
- 외부 API 호출 (번역 서비스, 감정 분석 API)
- 네트워크 통신 (웹소켓 메시지 전송, 푸시 알림)

이러한 작업 특성을 고려한 비용 요소별 비교:

| 비용 요소          | WebMVC                | WebFlux                |
|----------------|-----------------------|------------------------|
| **메모리 사용량**    | 높음 (스레드당 1-2MB 스택 할당) | 낮음 (이벤트 루프 기반)         |
| **CPU 사용률**    | 높음 (컨텍스트 스위칭 오버헤드)    | 낮음 (논블로킹 I/O)          |
| **동시 처리 성능**   | 제한적 (스레드 풀 크기 의존)     | 우수 (높은 동시성 지원)         |
| **I/O 대기 효율성** | 낮음 (블로킹으로 인한 자원 낭비)   | 높음 (대기 시간 동안 다른 작업 처리) |
| **확장성**        | 수직 확장 위주              | 수평 확장 유리               |

### 1.2 실제 메신저 서버 시나리오 비교

**WebMVC의 처리 방식:**
```java
@RestController
public class MessageController {
    
    @PostMapping("/message")
    public ResponseEntity<String> sendMessage(@RequestBody MessageRequest request) {
        // 1. 사용자 인증 - DB 조회 (평균 50ms 블로킹)
        User user = userService.authenticateUser(request.getToken());
        
        // 2. 번역 처리 - 외부 API 호출 (평균 200ms 블로킹)
        String translatedContent = translationService.translate(request.getContent());
        
        // 3. 감정 분석 - 외부 API 호출 (평균 150ms 블로킹)
        EmotionResult emotion = emotionService.analyze(request.getContent());
        
        // 4. 메시지 저장 - DB 저장 (평균 30ms 블로킹)
        Message message = messageService.saveMessage(request, translatedContent, emotion);
        
        // 5. 실시간 전송 - 웹소켓 브로드캐스트 (평균 20ms 블로킹)
        webSocketService.broadcastMessage(message);
        
        return ResponseEntity.ok("Message sent");
        // 총 450ms 동안 하나의 스레드가 블로킹
    }
}
```

- **문제점:**
  - 450ms 동안 스레드가 완전히 블로킹되어 다른 요청 처리 불가
  - 100개 동시 요청 시 스레드 풀 크기(보통 200개)의 절반 점유
  - 대기 시간 동안 메모리와 CPU 자원 낭비

**WebFlux의 처리 방식:**
```java
@RestController
public class MessageController {
    
    @PostMapping("/message")
    public Mono<String> sendMessage(@RequestBody MessageRequest request) {
        return userService.authenticateUser(request.getToken())
            .flatMap(user -> Mono.zip(
                translationService.translate(request.getContent()),
                emotionService.analyze(request.getContent())
            ))
            .flatMap(tuple -> {
                String translation = tuple.getT1();
                EmotionResult emotion = tuple.getT2();
                return messageService.saveMessage(request, translation, emotion);
            })
            .flatMap(message -> webSocketService.broadcastMessage(message)
                .thenReturn("Message sent"));
        // 스레드는 즉시 해제되어 다른 요청 처리 가능
    }
}
```

- **장점:**
  - I/O 대기 시간 동안 스레드가 다른 작업 처리
  - 병렬 처리로 번역과 감정 분석 동시 수행 (총 처리 시간 280ms로 단축)
  - 적은 수의 스레드로 수천 개의 동시 요청 처리 가능

### 1.3 비용 효율성 실측 결과
- **메모리 사용량 비교:**
  - WebMVC: 1,000명 동시 접속 시 약 2GB 메모리 사용
  - WebFlux: 1,000명 동시 접속 시 약 500MB 메모리 사용
- **처리량 비교 (동일한 4core, 8GB 서버 기준):**
  - WebMVC: 초당 약 500개 메시지 처리
  - WebFlux: 초당 약 2,000개 메시지 처리
- **서버 비용 절감 효과:**
  - WebMVC: 1,000명 동시 사용자 처리를 위해 4대 서버 필요
  - WebFlux: 1,000명 동시 사용자 처리를 위해 1대 서버로 충분

### 1.4 메신저 서버 특화 고려사항
메신저 서버의 핵심 요구사항인 **실시간성**과 **높은 동시성**을 고려할 때:
1. **웹소켓 연결 관리**: WebFlux는 논블로킹 I/O로 수천 개의 웹소켓 연결을 효율적으로 관리
2. **메시지 브로드캐스팅**: 대량의 메시지 전송 시 스레드 블로킹 없이 처리
3. **외부 API 통합**: 번역/감정 분석 API 호출 시 블로킹 없이 병렬 처리

### 1.5 결론
치기공 도메인의 메신저 서버에서 WebFlux를 선택한 이유:
1. **비용 효율성**: 동일한 하드웨어로 4배 높은 처리 성능
2. **확장성**: 사용자 증가에 따른 수평 확장 용이
3. **실시간성**: 논블로킹 I/O로 낮은 지연시간 보장
4. **자원 효율성**: 메모리와 CPU 사용량 대폭 절감

결과적으로 **제한된 예산으로 확장 가능한 고성능 메신저 서버**를 구축하는 목표에 가장 적합한 선택이었다.

## 2. 비즈니스 상황에 적합한 DB

메신저 서버의 데이터베이스를 선택할 때 고려해야 할 핵심 요소들을 분석해보면, 비용 효율성과 확장성이 가장 중요한 기준이었다. 특히 WebFlux와의 기술적 시너지와 메신저 도메인의 특성을 고려하여 PostgreSQL, MongoDB, ScyllaDB 세 가지 옵션을 비교 분석했다.

### 2.1 데이터베이스 선택 기준

메신저 서버의 데이터 특성을 먼저 분석해보면:
- **쓰기 중심**: 메시지 생성이 조회보다 빈번
- **시간순 데이터**: 메시지는 시간순으로 저장되고 조회됨
- **비정형 데이터**: 텍스트, 이미지, 파일 등 다양한 메시지 타입
- **급격한 데이터 증가**: 사용자 증가에 따른 메시지량 폭증
- **실시간성**: 낮은 지연시간 요구

### 2.2 데이터베이스 비교 분석

| 구분              | PostgreSQL     | MongoDB     | ScyllaDB/Cassandra |
|-----------------|----------------|-------------|--------------------|
| **초기 구축 비용**    | ⭐⭐⭐⭐⭐ 매우 낮음    | ⭐⭐⭐⭐ 낮음     | ⭐⭐ 높음              |
| **운영 비용**       | ⭐⭐⭐ 보통         | ⭐⭐⭐⭐ 낮음     | ⭐⭐ 높음              |
| **수평 확장성**      | ⭐⭐ 제한적         | ⭐⭐⭐⭐ 우수     | ⭐⭐⭐⭐⭐ 탁월           |
| **WebFlux 호환성** | ⭐⭐⭐ 보통 (R2DBC) | ⭐⭐⭐⭐⭐ 탁월    | ⭐⭐⭐ 보통             |
| **개발 생산성**      | ⭐⭐⭐ 보통         | ⭐⭐⭐⭐⭐ 매우 높음 | ⭐⭐ 낮음              |

### 2.3 PostgreSQL의 한계점

```sql
-- PostgreSQL에서의 메시지 저장 구조
CREATE TABLE messages (
   id BIGSERIAL PRIMARY KEY,
   chat_room_id UUID NOT NULL,
   user_id UUID NOT NULL,
   content JSONB NOT NULL,  -- 메시지 내용을 JSONB로 저장
   message_type VARCHAR(50) NOT NULL,
   created_at TIMESTAMP DEFAULT NOW(),
   INDEX idx_chat_room_time (chat_room_id, created_at)
);

-- 메시지 조회 시 복잡한 쿼리 필요
SELECT m.*, u.name as user_name 
FROM messages m 
JOIN users u ON m.user_id = u.id 
WHERE m.chat_room_id = ? 
ORDER BY m.created_at DESC 
LIMIT 50;
```

- 확장성 제약: 수직 확장 위주로 하드웨어 한계에 부딪힘 
- 복잡한 수평 확장: 수동 파티셔닝과 샤딩 로직 필요 
- JOIN 오버헤드: 메시지 조회 시 사용자 정보 JOIN으로 인한 성능 저하 
- 스키마 경직성: 새로운 메시지 타입 추가 시 스키마 변경 필요

### 2.4 MongoDB의 장점

```javascript
// MongoDB에서의 메시지 저장 구조
{
  _id: ObjectId("..."),  // 시간 정보 포함
  chatRoomId: "room_123",
  userId: "user_456",
  userName: "김철수",     // 사용자 정보 임베딩
  content: {
    type: "text",
    text: "안녕하세요",
    translation: "Hello"  // 번역 결과도 함께 저장
  },
  emotion: {
    sentiment: "positive",
    confidence: 0.85
  },
  createdAt: ISODate("2024-01-15T10:30:00Z")
}

// 단순한 메시지 조회
db.messages.find({chatRoomId: "room_123"})
           .sort({_id: -1})
           .limit(50);
```

#### MongoDB의 비즈니스 적합성:

1. **WebFlux와의 완벽한 호환성**
    ```java
    // MongoDB Reactive Streams와 WebFlux 연동
    @Service
    public class MessageService {
        
        public Mono<Message> saveMessage(MessageRequest request) {
            return mongoTemplate.save(
                Message.builder()
                    .chatRoomId(request.getChatRoomId())
                    .content(request.getContent())
                    .build()
            );
        }
        
        public Flux<Message> getMessages(String chatRoomId) {
            return mongoTemplate.find(
                Query.query(Criteria.where("chatRoomId").is(chatRoomId))
                     .with(Sort.by(Sort.Direction.DESC, "_id"))
                     .limit(50),
                Message.class
            );
        }
    }
    ```
2. **자동 샤딩을 통한 확장성**
   - 사용자 증가에 따른 투명한 수평 확장 
   - 샤드 키 기반 자동 데이터 분산 
   - 애플리케이션 코드 변경 없이 노드 추가
3. **스키마리스 구조의 유연성**
   - 새로운 메시지 타입 추가 시 즉시 반영 
   - 번역/감정 분석 결과를 메시지와 함께 저장 
   - 메시지 구조 변경에 따른 마이그레이션 불필요
4. **비용 효율성**
   - JOIN 연산 없이 단일 쿼리로 메시지 조회
   - 임베딩을 통한 네트워크 I/O 최소화
   - Atlas 관리형 서비스로 운영 비용 절감

### 2.5 ScyllaDB의 오버엔지니어링
ScyllaDB는 초고성능 메시지 처리에는 탁월하지만, 현재 비즈니스 상황에서는 **과도한** 선택이었다:
- 높은 초기 구축 비용: 전문 인력과 복잡한 클러스터 설정 필요
- 운영 복잡성: 작은 개발팀에서 관리하기 어려운 구조
- 개발 생산성 저하: CQL 학습 곡선과 제한적인 쿼리 기능

### 2.6 MongoDB 선택의 결론
치기공 도메인의 메신저 서버에서 MongoDB를 선택한 이유:

1. **비용 효율성**: 개발 생산성과 운영 비용의 최적 균형점
2. **확장성**: 자동 샤딩으로 사용자 증가에 대응
3. **기술적 시너지**: WebFlux와의 완벽한 비동기 호환성
4. **도메인 적합성**: JSON 기반 메시지 구조와 자연스러운 매칭
5. **운영 단순성**: 작은 팀에서도 관리 가능한 구조

결과적으로 제한된 예산으로 확장 가능한 메신저 서버를 구축하는 목표에 가장 적합한 선택이었다.
